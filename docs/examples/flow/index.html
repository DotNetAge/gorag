    完整工作流程示例¶

    本文件提供GoRAG从数据准备到生产部署的端到端工作流程示例，覆盖问答系统、文档检索、知识图谱应用等实际业务场景。内容包括完整的项目结构、代码组织与部署流程，以及数据预处理、模型训练、系统集成与运维监控的全流程实践。读者可据此将GoRAG无缝集成到现有系统中，并在生产环境中稳定运行。

    项目结构¶

    GoRAG采用模块化分层设计，核心目录与职责如下： - pkg/core：定义统一接口与数据类型（检索器、生成器、向量存储、指标等） - pkg/indexer：索引器工厂与管道构建，支持本地与企业级向量/文档/图存储 - pkg/retriever：多种检索策略（原生、高级、智能路由、图检索） - pkg/steps：可复用的流水线步骤（索引、向量搜索、生成、融合、重排序等） - pkg/generation/evaluator：基于LLM的评估（RAGAS风格） - pkg/di：依赖注入容器，便于服务注册与解析 - cmd/gorag：命令行入口（CLI框架占位）

    核心组件¶

    • 应用接口与工厂：统一的RAG应用接口与多种预设（原生、高级、智能路由、图检索），自动装配索引器与检索器
    • 索引器：支持并发文件扫描、多格式解析、语义切分、批量嵌入、实体抽取与多存储写入
    • 检索器：原生向量检索与答案生成；高级融合检索；智能路由按意图分发；图检索支持三元组关系推理
    • 评估器：基于LLM的RAGAS风格评估（事实性、相关性、上下文精度）
    • 依赖注入：全局容器支持单例/瞬态注册与解析，便于测试与扩展

    详细组件分析¶

    应用接口与工厂（RAG）¶

    • 提供统一的RAG接口（索引文件/目录、检索）与四种预设模式（原生、高级、智能路由、图检索）
    • 工厂根据配置选择向量存储、文档存储与检索器组合，并注入日志与指标
    classDiagram class RAG { +IndexFile(ctx, filePath) error +IndexDirectory(ctx, dirPath, recursive) error +Search(ctx, query, topK) RetrievalResult } class defaultRAG { -indexer -retriever +IndexFile(...) +IndexDirectory(...) +Search(...) } class RAGConfig { +WorkDir +VectorDBType +Dimension +EmbedderType +LLMType +ModelPath +ModelName +APIKeyEnv +BaseURL +TopK } RAG <|.. defaultRAG defaultRAG --> RAGConfig : "使用"

    索引器与数据管线¶

    • 支持并发文件遍历与索引，内置步骤包括：文件发现、多解析器流式解析、语义切分、批量嵌入、实体抽取、多存储写入
    • 默认工厂提供原生、高级、图三种预设，自动装配本地SQLite与GoVector或企业级向量数据库
    flowchart TD Start(["开始索引"]) --> Discover["文件发现"] Discover --> Parse["多解析器流式解析"] Parse --> Chunk["语义切分"] Chunk --> Embed["批量嵌入"] Embed --> Entities["实体抽取(可选)"] Entities --> Stores["多存储写入<br/>向量/文档/图"] Stores --> End(["结束"])

    检索器与答案生成¶

    • 原生检索器：向量搜索 + 答案生成，支持分布式追踪与日志
    • 答案生成器：基于模板构建上下文提示，调用LLM生成回答，并记录可观测性指标
    sequenceDiagram participant Q as "查询" participant RET as "原生检索器" participant VS as "向量存储" participant GEN as "生成器" Q->>RET : "Retrieve(queries, topK)" RET->>VS : "Search 向量" VS-->>RET : "匹配块" RET->>GEN : "Generate(query, chunks)" GEN-->>RET : "Result{Answer}" RET-->>Q : "RetrievalResult"

    评估与质量控制¶

    • LLMJudge实现RAGAS风格评估，分别计算事实性、答案相关性与上下文精度
    • 通过标准提示词与结果解析，输出评分与理由
    flowchart TD A["输入: 查询/上下文/答案"] --> B["事实性评估"] A --> C["答案相关性评估"] A --> D["上下文精度评估"] B --> E["评分与理由"] C --> E D --> E

    依赖注入与服务管理¶

    • 全局容器支持注册单例/瞬态服务，提供类型安全解析与默认容器便捷方法
    • 适用于测试环境重置与生产环境服务替换
    classDiagram class Container { -services +Register(iface, factory, isSingleton) +RegisterInstance(iface, instance) +Resolve(iface) +ResolveTyped[T]() +Clear() +IsRegistered(iface) } class DefaultContainer { +Default() +Register/Resolve/... } Container <|-- DefaultContainer

    依赖关系分析¶

    • 外部依赖：gochat（聊天与嵌入）、向量库（govector、milvus、pinecone、qdrant、weaviate）、图数据库（neo4j）、日志（zap）、观测（OpenTelemetry、Prometheus）
    • 内部模块：core接口层、indexer索引层、retriever检索层、steps流水线步骤、generation评估层、di依赖注入
    graph LR CORE["core 接口/类型"] --> INDEXER["indexer 索引器"] CORE --> RETRIEVER["retriever 检索器"] CORE --> STEPS["steps 流水线步骤"] CORE --> EVAL["generation/evaluator 评估"] INDEXER --> STORES["Doc/Vector/Graph 存储"] RETRIEVER --> LLM["LLM 客户端"] EVAL --> JUDGE["LLMJudge"]

    性能考虑¶

    • 并发索引：默认启用并发与可配置工作线程数，适合TB级知识库增量索引
    • 流式解析：多解析器支持io.Reader流式处理，降低内存占用
    • 向量存储：本地GoVector零CGO跨平台，企业级Milvus/Qdrant/Weaviate支持高吞吐
    • 观测性：内置OpenTelemetry与Prometheus指标，便于生产监控与性能优化

    故障排除指南¶

    • 索引失败但存储初始化成功：检查嵌入模型配置与API密钥；测试用例验证了即使缺少嵌入器，向量数据库文件仍会被初始化
    • 日志与追踪：通过索引器选项启用zap日志与OpenTelemetry追踪，定位耗时步骤
    • 评估异常：确认LLM可用与提示词解析正确，评估器会返回评分与原因

    结论¶

    GoRAG提供了从数据准备到生产部署的一体化解决方案，具备高性能、可观测性与模块化扩展能力。通过统一的应用接口与多种检索策略预设，可在问答系统、文档检索与知识图谱应用中快速落地，并平滑集成到现有系统中。

    集成到现有系统的建议¶

    • 将RAG应用接口封装为微服务或库模块，对外暴露HTTP/SDK接口
    • 使用依赖注入容器集中管理LLM/嵌入器/存储实例
    • 通过可观测性指标与分布式追踪建立生产监控体系
    • 利用评估器持续校验回答质量，形成闭环优化
