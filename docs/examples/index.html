    示例和教程¶

    本教程面向希望在Go生态中构建生产级RAG应用的工程师，提供从基础使用到高级配置的完整示例与实操指南。内容覆盖： - 快速开始：本地Agent/本地知识库、企业级高级RAG、图RAG等预设模式 - 应用场景：问答系统、文档检索、知识图谱推理 - 自定义开发：自定义解析器、管道步骤、检索策略 - 第三方集成：企业级向量数据库、图数据库 - 性能优化与最佳实践 - 完整项目模板与脚手架思路 - 运行方法与预期结果说明

    项目结构¶

    GoRAG采用模块化分层设计，核心接口位于pkg/core，具体实现分布在indexer、retriever、steps、generation、retrieval等包中；通过依赖注入容器统一装配。

    graph TB subgraph "应用入口" A["gorag.go<br/>统一RAG应用接口"] end subgraph "核心接口层(pkg/core)" C1["interfaces.go<br/>Retriever/Generator/VectorStore等"] C2["types.go<br/>Query/RetrievalResult等数据模型"] C3["docstore.go<br/>DocStore接口"] end subgraph "检索层" R1["native/retriever.go<br/>原生检索器"] R2["steps/vector/search.go<br/>向量搜索步骤"] R3["steps/generate/generate.go<br/>答案生成步骤"] end subgraph "索引层" I1["steps/indexing/chunk.go<br/>语义分块步骤"] I2["indexing/parser/base/parser.go<br/>流式解析器接口"] end subgraph "DI容器" D1["di/container.go<br/>依赖注入容器"] end subgraph "评估与可观测" E1["generation/evaluator/ragas.go<br/>RAGAS评测"] end A --> C1 A --> C2 A --> R1 R1 --> R2 R1 --> R3 A --> I1 I1 --> I2 A --> D1 A --> E1

    核心组件¶

    • RAG应用接口：统一的索引与检索入口，内置多种预设模式（本地、高级、智能路由、图RAG）
    • 检索器：基于管道的可插拔检索器，支持向量搜索与答案生成
    • 步骤：向量搜索、答案生成、语义分块等可复用步骤
    • 存储：向量存储与文档存储分离，支持多后端
    • 解析器：流式解析器接口，支持大规模文件处理
    • 评测：RAGAS风格的LLM评测器
    • DI容器：全局依赖注入容器，便于装配与测试

    详细组件分析¶

    基础使用与预设模式¶

    • 本地Agent/本地知识库：默认本地向量与文档存储，适合轻量部署
    • 企业级高级RAG：融合多策略（如RAG-Fusion）与高召回
    • 智能路由：按意图自动分发到原生/高级/图RAG
    • 图RAG：结合Neo4j等图数据库进行关系推理

    运行方法与预期结果 - 使用统一RAG接口创建应用实例 - 调用索引目录或文件，等待索引完成 - 执行Search，返回包含答案与上下文的结构化结果

    检索器与管道执行¶

    原生检索器通过管道串联"向量搜索"和"答案生成"两个步骤，支持分布式追踪与日志记录。

    flowchart TD S["开始"] --> Q["读取查询(Query)"] Q --> VEC["向量搜索(VectorSearch)"] VEC --> EMB["生成查询嵌入"] EMB --> VS["向量存储检索"] VS --> CH["转换为Chunk组"] CH --> FUSE["合并/融合(可选)"] FUSE --> GEN["答案生成(Generator)"] GEN --> OUT["输出RetrievalResult"]

    自定义解析器与索引步骤¶

    • 流式解析器接口：支持Reader流式解析，内存复杂度O(1)，适用于超大文件
    • 语义分块步骤：将文档切分为语义片段，供后续嵌入与检索
    • 可扩展性：新增解析器类型只需实现Parser接口并注册
    classDiagram class Parser { +ParseStream(ctx, reader, metadata) chan Document +GetSupportedTypes() []string } class SemanticChunker { +Chunk(ctx, document) []*Chunk } class ChunkStep { +Execute(ctx, IndexingContext) error } Parser <|.. MyParser SemanticChunker <|.. MyChunker ChunkStep --> SemanticChunker : "使用"

    文档存储与向量存储¶

    • 文档存储：保存完整文档与块，支持父子关系与父文档检索
    • 向量存储：仅存嵌入与元数据，用于高效相似检索
    • 可替换性：通过接口抽象，可接入不同后端
    erDiagram DOCUMENT { string id PK string title text raw_text map metadata } CHUNK { string id PK string doc_id FK text content map metadata } VECTOR { string id PK float_vector vector map metadata } DOCUMENT ||--o{ CHUNK : "包含" CHUNK ||--o{ VECTOR : "嵌入对应"

    评测与可观测性¶

    • RAGAS风格评测：对一致性、相关性、上下文精度进行评分
    • 分布式追踪与指标：贯穿索引与检索全流程，便于定位瓶颈
    sequenceDiagram participant APP as "应用" participant JUDGE as "RagasLLMJudge" participant LLM as "评测LLM" APP->>JUDGE : "EvaluateFaithfulness/AnswerRelevance/ContextPrecision" JUDGE->>LLM : "构造提示词并请求" LLM-->>JUDGE : "评分与理由" JUDGE-->>APP : "RAGEvaluation"

    性能考虑¶

    • 并发与流式：解析器与索引步骤采用流式与并发，内存复杂度O(1)
    • 向量检索：合理设置TopK与过滤条件，避免无谓的全量扫描
    • 管道复用：将常用步骤（如HyDE、重排序、融合）按需插入，平衡准确率与延迟
    • 观测性：开启Prometheus与OpenTelemetry，定位耗时环节（索引、嵌入、检索、生成）

    故障排查指南¶

    • 空查询错误：向量搜索步骤要求非空查询，检查上下文是否正确填充
    • 嵌入失败：确认嵌入Provider可用且维度匹配
    • 管道执行异常：查看每步日志与Span，定位具体失败步骤
    • 评测结果异常：检查提示词格式与LLM响应解析

    结论¶

    通过模块化的接口设计与可插拔的管道机制，GoRAG为不同规模与场景提供了即开即用的RAG解决方案。配合DI容器、流式解析与企业级评测能力，开发者可以快速搭建从原型到生产的工业级应用。
