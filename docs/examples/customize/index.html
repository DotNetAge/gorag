    自定义开发示例¶

    本文件面向希望在 GoRAG 基础上进行“自定义扩展”的开发者，系统讲解如何实现自定义解析器、管道步骤、检索策略与存储后端，并提供接口实现、注册机制与集成方法的具体路径指引。内容覆盖从接口契约到工厂构建、从 DI 容器到管道执行的全链路实践，帮助你快速扩展现有能力或创建全新的 RAG 应用变体。

    项目结构¶

    GoRAG 采用分层模块化设计： - 核心接口与数据模型位于 core 层，统一抽象检索、生成、向量化、指标等能力 - DI 容器提供服务注册与解析能力 - 索引阶段通过多解析器与多步管道完成文档解析、切片、嵌入与入库 - 检索阶段通过向量搜索、重排序、融合与增强等步骤组合形成可插拔策略 - 存储后端提供向量与文档的持久化能力

    核心组件¶

    • RAG 接口与工厂：统一对外能力，按模式装配索引器与检索器
    • 核心接口：Retriever、Parser、VectorStore、Reranker、IntentClassifier 等
    • DI 容器：注册单例/瞬态服务，支持类型安全解析
    • 索引管道：多解析器路由、流式解析、语义切片
    • 检索管道：向量搜索、重排、融合、增强
    • 存储后端：向量存储与文档存储

    自定义解析器开发¶

    目标：实现一个新的文档解析器并接入现有索引流水线。

    要点 - 实现基础解析器接口，支持流式解析与类型声明 - 在解析器注册表中注册新类型 - 在索引步骤中由文件扩展名自动路由到该解析器

    自定义管道步骤开发¶

    目标：在索引或检索阶段插入自定义处理步骤（如过滤、去重、增强）。

    通用模式 - 实现 Step 接口（名称与执行） - 在上下文中读取/写入数据（如 Documents、Chunks、RetrievedChunks） - 与核心接口解耦，通过上下文传递数据

    参考步骤 - 索引阶段：解析与切片 - 多解析器路由：parse.go:36-89 - 语义切片：chunk.go:33-72 - 检索阶段：向量搜索、重排、融合、增强 - 向量搜索：search.go:37-111 - 文档增强：docstore.go:32-62 - 交叉编码重排：cross_encoder.go:41-73 - 融合说明：doc.go:6-11

    自定义检索策略开发¶

    目标：实现新的检索策略（如意图分类路由、融合、HyDE/Step-back 等）。

    要点 - 实现 Retriever 接口或基于现有检索器组合 - 可结合意图分类、子查询生成、后退一步查询等策略 - 与向量搜索、重排、融合等步骤协作

    参考实现 - RAG 工厂与模式装配：gorag.go:103-152 - 意图类型与结果：types.go:25-41 - 向量搜索步骤（含 HyDE/Fusion/Step-back 分支）：search.go:37-111 - 融合与重排步骤：doc.go:6-11、cross_encoder.go:41-73

    自定义存储后端开发¶

    目标：实现 VectorStore 或 DocStore 接口，替换或扩展底层存储。

    要点 - 实现 VectorStore 接口（Upsert/Search/Delete/Close） - 实现 DocStore 接口（GetDocument 等） - 在工厂中注入你的实现

    参考接口与实现 - VectorStore 接口：interfaces.go:40-46 - 内存向量存储实现：store.go:18-202 - DocStore 接口（来自 core/store 包）：gorag.go:8

    性能考虑¶

    • 流式解析与 O(1) 内存：解析器接口明确要求 ParseStream，避免一次性加载超大文件
    • 参考：parser.go:9-17
    • 向量搜索优化：内存向量存储使用预计算范数与 Top-K 快速选择算法
    • 参考：store.go:54-93
    • 管道并行化：索引阶段的解析与切片可并行处理多个文档通道
    • 参考：parse.go:79-89
    • 指标观测：通过 Metrics 接口记录搜索耗时、结果数量、嵌入计数等
    • 参考：interfaces.go:48-57

    故障排查指南¶

    • 解析器未找到
    • 现象：按扩展名无法路由到解析器
    • 排查：确认注册表是否包含对应类型；检查扩展名大小写与映射
    • 参考：parse.go:36-50、registry.go:74-96
    • 向量搜索失败
    • 现象：查询嵌入生成或向量检索报错
    • 排查：检查 embedder 是否配置；确认维度与向量存储一致
    • 参考：search.go:68-89
    • 文档增强失败
    • 现象：无法从 DocStore 获取父文档
    • 排查：确认 DocumentID 是否正确；检查 DocStore 实现
    • 参考：docstore.go:32-62
    • DI 解析失败
    • 现象：Resolve 返回未注册或类型不匹配
    • 排查：确认 Register 与 Resolve 的接口类型一致；单例初始化状态
    • 参考：container.go:104-171

    结论¶

    通过统一的接口契约、灵活的 DI 容器与可插拔的管道步骤，GoRAG 提供了强大的自定义扩展能力。开发者可以按需实现解析器、管道步骤、检索策略与存储后端，并通过工厂与注册表无缝集成，快速构建定制化的 RAG 应用变体。

    附录：项目模板与开发工作流¶

    • 项目模板建议
    • 新增解析器：实现基础解析器接口 → 在注册表中注册 → 在索引步骤中自动路由
    • 新增管道步骤：实现 Step 接口 → 在索引/检索流水线中添加
    • 新增检索策略：实现 Retriever 或组合现有策略 → 在工厂中注入
    • 新增存储后端：实现 VectorStore/DocStore 接口 → 在工厂中替换
    • 开发工作流
    • 设计阶段：确定接口契约与上下文数据结构
    • 实现阶段：实现具体组件并注册到 DI 容器
    • 集成阶段：在工厂中装配组件，编写最小可用流水线
    • 测试阶段：单元测试 + 端到端测试；使用 ResetForTesting 清理全局状态
    • 性能阶段：关注流式处理、向量搜索与指标观测
