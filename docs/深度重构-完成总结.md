## 💡 架构决策总结

### 1. 为什么 AgenticMetadata 定义在 entity 包？

**原因**: 避免循环依赖

```go
// ✅ 正确方案
package entity  // AgenticMetadata 定义在这里
// retrieval 包可以使用 entity.AgenticMetadata，不会循环
```

### 2. 为什么 Step 层不注入 Metrics Collector？

**原因**: 保持职责单一和超薄特性

- **Service 层**: 厚业务逻辑 → 需要 metrics 监控性能
- **Step 层**: 薄适配器（<30 行核心逻辑）→ 只负责状态转换和日志记录
- **Metrics 收集**已经在 Service 层完成，Step 层不需要重复

### 3. 为什么使用指针类型 `CacheHit *bool`？

**原因**: 区分三种状态

- `nil` = 未检查缓存
- `false` = 缓存未命中
- `true` = 缓存命中

## 🎓 学习要点

### 给开发者的建议

1. ✅ **先定义接口**: 在 entity 包中定义 AgenticMetadata
2. ✅ **再实现业务**: Service 层处理厚业务逻辑
3. ✅ **最后创建适配器**: Step 层保持超薄（<30 行核心逻辑）
4. ✅ **注入依赖**: Logger 和 Metrics 通过构造函数注入

### 常见陷阱

- ❌ **跳过接口定义**: 直接创建 Step 和 Service
- ❌ **Service 太薄**: 只包含简单的委托调用
- ❌ **Step 太厚**: 包含 Prompt 工程和 JSON 解析
- ❌ **命名不规范**: 使用 XXXXImpl 这种 Java 风格
