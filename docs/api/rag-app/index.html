    RAG应用API¶

    本文件为 GoRAG 应用API的完整技术文档，聚焦于统一的RAG应用接口及其配套的索引与检索能力。内容涵盖： - RAG接口方法：IndexFile、IndexDirectory、Search 的参数、返回值与使用场景 - RAGConfig 配置结构与 RAGOption 选项函数 - 默认RAG应用工厂：DefaultNativeRAG、DefaultAdvancedRAG、DefaultAgenticRAG、DefaultGraphRAG - 数据模型与类型：查询、意图、检索结果等 - 架构与流程图：索引流水线、检索流水线、智能路由 - 版本兼容与迁移建议

    项目结构¶

    GoRAG采用模块化分层设计，核心接口位于 pkg/core，索引与检索分别在 pkg/indexing 与 pkg/retrieval 下实现，并通过 pkg/di 提供依赖注入容器。

    核心组件¶

    • RAG 接口：统一的RAG应用入口，提供 IndexFile、IndexDirectory、Search 三个方法
    • RAGConfig：集中式配置，覆盖持久化、模型“接地”（本地或API）、检索调优
    • RAGOption：选项函数模式，用于构建RAGConfig
    • 默认RAG工厂：按模式创建具备预设索引器与检索器的RAG实例

    • 核心接口与类型：Retriever、VectorStore、RetrievalResult、IntentType 等

    详细组件分析¶

    RAG 接口与默认实现¶

    • 方法
    • IndexFile(ctx, filePath) error：处理单个文件，返回错误
    • IndexDirectory(ctx, dirPath, recursive) error：并发遍历目录并处理文件
    • Search(ctx, query, topK) (*core.RetrievalResult, error)：执行检索并返回结果

    • 默认实现 defaultRAG 将上述方法委托给内部 indexer 与 retriever

    • gorag.go:56-74

    RAGConfig 与 RAGOption¶

    • RAGConfig 字段
    • 持久化：WorkDir、VectorDBType、Dimension
    • 模型“接地”：EmbedderType、LLMType、ModelPath、APIKey、BaseURL
    • 检索调优：TopK
    • RAGOption
    • WithWorkDir、WithModelPath、WithAPIKey、WithOllama 等
    • 默认RAG工厂
    • DefaultNativeRAG、DefaultAdvancedRAG、DefaultAgenticRAG、DefaultGraphRAG

    • 工厂内部通过 buildRAG 根据模式选择索引器与检索器，并初始化向量库与文档库

    • gorag.go:77-100

    • gorag.go:103-152

    索引器 Indexer¶

    • 接口
    • IndexFile(ctx, filePath) (*core.IndexingContext, error)
    • IndexDirectory(ctx, dirPath, recursive) error
    • 默认工厂
    • DefaultNativeIndexer、DefaultAdvancedIndexer、DefaultGraphIndexer
    • 支持并发工作线程、日志与指标集成
    • 关键步骤（由流水线组装）

    • 发现文件、解析、分块、嵌入、实体抽取、多存储写入

    检索器 Retriever¶

    • 接口
    • Retrieve(ctx, queries []string, topK int) ([]*core.RetrievalResult, error)
    • 基础实现
    • New(...)：基于向量库与嵌入器进行检索，支持并发查询
    • 原生检索器
    • DefaultNativeRetriever：内置向量搜索与答案生成
    • 高级检索器（融合）

    • DefaultAdvancedRetriever：查询分解 + 向量搜索 + RRF 融合 + 答案生成

    • RetrievalResult：包含查询、块集合、分数、答案与元数据
    • 其他DTO

    性能与可扩展性¶

    • 并发与流水线
    • 索引器支持并发工作线程与错误组控制
    • 检索器对多查询并行执行，提升吞吐
    • 存储与嵌入
    • 默认使用本地向量库与SQLite文档库；可通过选项切换至企业级向量库
    • 支持本地模型与API提供商（OpenAI、Ollama等）
    • 可观测性
    • 日志、指标、分布式追踪贯穿索引与检索阶段
    • 扩展点
    • 自定义 Parser、Chunker、Embedder、VectorStore、Retriever

    • 通过 DI 容器管理生命周期

    故障排查指南¶

    • 常见问题
    • 缺少嵌入器导致索引失败：在默认RAG中需提供嵌入器或使用企业级向量库
    • 向量维度不匹配：确保 RAGConfig.Dimension 与嵌入器维度一致
    • 权限与路径：确认 WorkDir 可写且向量/文档数据库文件存在
    • 建议
    • 使用 WithWorkDir、WithModelPath、WithAPIKey 等选项正确配置
    • 开启日志与指标，定位瓶颈

    • 对复杂查询启用高级检索器（融合）以提升召回

    • gorag.go:103-152

    结论¶

    GoRAG 提供了统一、可扩展、可观测的RAG应用API。通过 RAG 接口与默认工厂，开发者可以快速搭建从本地到企业级的RAG系统；通过清晰的接口与模块化设计，易于替换与扩展底层组件。

    附录：API参考与示例¶

    RAG 接口方法¶

    • IndexFile(ctx, filePath) error
    • 参数：上下文、文件路径
    • 返回：错误
    • 场景：离线准备单个文件的向量/图索引
    • IndexDirectory(ctx, dirPath, recursive) error
    • 参数：上下文、目录路径、是否递归
    • 返回：错误
    • 场景：批量离线准备知识库
    • Search(ctx, query, topK) (*core.RetrievalResult, error)
    • 参数：上下文、查询文本、返回条数
    • 返回：检索结果对象、错误
    • 场景：在线问答与检索

    RAGConfig 与 RAGOption¶

    • RAGConfig 字段
    • WorkDir：工作目录
    • VectorDBType：向量库类型（如 govector）
    • Dimension：向量维度
    • EmbedderType、LLMType：嵌入器与LLM类型
    • ModelPath、APIKey、BaseURL：模型路径与API密钥/端点
    • TopK：默认检索返回数量
    • RAGOption

    • WithWorkDir、WithModelPath、WithAPIKey、WithOllama 等

    • gorag.go:36-47

    默认RAG应用工厂¶

    • DefaultNativeRAG(opts...) (RAG, error)
    • 适用：标准语义检索、本地知识库
    • 预设：govector、TopK=5
    • DefaultAdvancedRAG(opts...) (RAG, error)
    • 适用：高召回企业级场景
    • 预设：govector、TopK=10
    • DefaultAgenticRAG(opts...) (RAG, error)
    • 适用：意图路由与多策略协作
    • 预设：组合原生、高级、图检索器
    • DefaultGraphRAG(opts...) (RAG, error)
    • 适用：关系推理与图谱增强

    • 预设：图索引器与图检索器

    • gorag.go:103-152

    数据模型与类型¶

    • Query、IntentType、RetrievalResult、SearchRequest/Response、AgenticSearchResponse、ChatRequest/Response、IndexRequest/Response

    依赖注入容器¶

    • Container、Register、RegisterInstance、RegisterSingleton、Resolve、MustResolve、Default 等

    示例与最佳实践¶

    • 快速开始（本地）
    • 创建默认原生RAG，索引目录，执行搜索
    • 企业级（高级）
    • 连接 Milvus，启用 OpenAI 嵌入器，使用高级RAG
    • 图谱增强（GraphRAG）
    • 连接 Neo4j，启用图索引与检索

    版本兼容与迁移指南¶

    • Go 版本要求：1.25+
    • 依赖管理：go.mod 中列出所有外部依赖
    • 迁移建议
    • 从本地 govector 切换到 Milvus/Qdrant 等时，仅需调整 RAGOption 与存储初始化
    • 若需自定义检索器，实现 core.Retriever 接口并通过工厂注入

    • 如需替换嵌入器，确保 Dimension 与 RAGConfig 一致
