    检索器API¶

    本文件为 GoRAG 检索器API的完整技术文档，覆盖多种检索器类型（Native、Advanced、Graph、Agentic、CRAG、Self-RAG）的接口规范、配置参数、查询处理流程、结果格式、性能优化与并发使用建议，并提供不同检索策略的API使用示例路径与最佳实践。

    项目结构¶

    检索器相关代码主要位于以下模块： - 核心接口与数据模型：pkg/core - 基础检索服务：pkg/retrieval/service - 具体检索器实现：pkg/retriever/* - 向量化与生成步骤：pkg/steps/vector、pkg/steps/generate - 高级融合与查询处理：pkg/retriever/advanced、pkg/retrieval/fusion、pkg/retrieval/query

    核心组件¶

    • 统一检索接口
    • 接口定义：Retriever.Retrieve(ctx, queries []string, topK int) 返回 []*RetrievalResult
    • 作用：对多个查询执行检索与可选的答案生成，返回每个查询的结果集合
    • 数据模型
    • Query：包含查询文本、元数据与时间戳
    • RetrievalResult：包含查询、块集合、分数、最终答案及可选元数据
    • IntentType/IntentResult：用于意图分类
    • CRAGLabel/CRAGEvaluation、RAGEvaluation：用于评估与反馈
    • 通用检索服务

    • 提供并发检索能力，支持批量查询并行化，内置指标采集与日志记录

    架构总览¶

    检索器采用“管道+步骤”的可插拔架构。每个检索器通过组合不同的步骤（如向量搜索、答案生成、查询改写、意图分类、图谱搜索、融合等）形成完整的检索流程。统一的 Retriever 接口保证了不同实现的一致调用方式。

    详细组件分析¶

    Native 检索器¶

    • 角色定位：开箱即用的原生检索器，基于向量相似度检索与LLM生成答案
    • 关键特性
    • 默认向量存储：若未提供，自动创建本地向量数据库
    • 可配置项：向量存储、文档存储、嵌入模型、LLM、TopK、工作目录、日志、追踪
    • 流程：向量搜索 -> 答案生成
    • 典型用途：快速原型、小规模知识库、基础问答场景
    classDiagram class NativeRetriever { +Retrieve(ctx, queries, topK) []*RetrievalResult -pipeline -tracer } class Options { +Logger +Tracer +Embedder +LLM +TopK +WorkDir +VectorStore +DocStore } NativeRetriever --> Options : "构造时注入"

    Advanced 检索器族¶

    • RAG-Fusion（多视角融合）
    • 特性：查询分解 + 多子查询检索 + RRF 融合 + 答案生成
    • 关键点：默认 TopK、可注入向量存储、嵌入模型、LLM、日志
    • 适用：高召回需求、复杂问题拆解
    • 查询改写（Rewrite）
    • 特性：先改写/澄清查询，再检索与生成
    • 适用：模糊或歧义查询
    • 抽象回溯（Stepback）
    • 特性：生成更抽象的回溯查询，扩大检索范围
    • 适用：细节不足或需要全局视角的问题

    Graph 检索器¶

    • 角色定位：结合知识图谱与向量检索的混合检索器
    • 关键特性
    • 实体抽取（可选）
    • 图谱邻居遍历（深度、数量限制）
    • 可选文档存储增强（PDR）
    • 自定义步骤扩展（如Cypher推理）
    • 可配置提示模板
    • 可注入向量/图/文档存储、嵌入模型、LLM、日志、追踪
    • 流程：实体抽取 -> 图搜索 -> 向量搜索 -> 文档增强 -> 生成（含图谱上下文）

    Agentic 检索器¶

    • 角色定位：由智能体驱动的检索器，支持工具调用与推理
    • 关键特性
    • 将 Agent 包装为管道步骤
    • 支持会话ID记忆（通过查询元数据）
    • 记录 agent 步骤到结果元数据
    • 可注入日志
    • 流程：Agent 执行 -> 生成最终答案（可携带中间步骤信息）
    classDiagram class AgenticRetriever { +Retrieve(ctx, queries, topK) []*RetrievalResult -agent -pipeline -logger } class agentStep { +Name() string +Execute(ctx, context) error } AgenticRetriever --> agentStep : "封装为步骤"

    CRAG 检索器¶

    • 角色定位：基于评估的纠错RAG，必要时触发网络搜索作为回退
    • 关键特性
    • 初始向量检索
    • 评估（CRAGLabel：相关/不相关/模糊）
    • 文档存储增强（PDR）
    • 回退步骤：根据评估标签决定替换或追加检索结果
    • 可注入 WebSearcher、文档存储、日志、追踪
    • 流程：检索 -> 评估 -> 文档增强 -> 回退（可选）-> 生成
    flowchart TD S(["开始"]) --> V["向量检索"] V --> Eval["CRAG 评估"] Eval --> Label{"标签"} Label --> |relevant| Gen["生成答案"] Label --> |irrelevant| WS["网络搜索回退(替换)"] Label --> |ambiguous| WS2["网络搜索回退(追加)"] WS --> Gen WS2 --> Gen Gen --> E(["结束"])

    Self-RAG 检索器¶

    • 角色定位：具备自反评估与迭代精炼能力的检索器
    • 关键特性
    • 初始检索 -> 初始生成 -> 自反评估循环（阈值/最大重试次数）
    • 根据反馈迭代优化答案
    • 可注入文档存储、日志、追踪
    • 流程：检索 -> 生成 -> 评估 -> 反思与精炼（多次）-> 输出最终答案

    • 单查询与多查询路径

    依赖分析¶

    • 组件耦合
    • 检索器均依赖 core.Retriever 接口，确保统一调用
    • 步骤层（vector/search、generate）为可复用组件，被各类检索器组合使用
    • 高级检索器依赖融合引擎接口（RRF）、意图分类器等
    • 外部依赖
    • 向量存储与嵌入提供者通过接口注入，便于替换实现
    • LLM 客户端用于答案生成与查询改写/抽象等

    性能考虑¶

    • 并发检索
    • retrieval/service.retriever 在多查询场景下使用 goroutine 并行执行，显著降低延迟
    • 建议：合理设置 topK 与并发数，避免向量存储与嵌入服务成为瓶颈
    • 向量搜索优化
    • 选择合适维度与索引实现；在 GraphRAG 中可结合图遍历减少无效节点
    • 使用过滤条件（SearchOptions.Filters）缩小搜索空间
    • 生成与评估
    • 控制生成长度与提示词模板大小；CRAG/Self-RAG 的评估与回退可能增加耗时
    • 指标与观测
    • 使用 Tracer/Metrics 记录关键阶段耗时与错误，便于定位性能瓶颈

    故障排查指南¶

    • 常见错误与定位
    • 向量搜索失败：检查嵌入模型与向量存储初始化；确认查询非空
    • LLM 调用失败：检查客户端配置与网络连通性；查看生成步骤日志
    • 评估失败：确认评估器实现与输入格式；关注 Self-RAG/CRAG 的反馈链路
    • 日志与追踪
    • 各检索器均支持 Logger 与 Tracer 注入，建议开启详细日志以捕获错误事件

    • GraphRAG/CRAG/Self-RAG 在关键步骤记录 span 与事件，便于端到端追踪

    结论¶

    GoRAG 检索器API通过统一接口与可插拔步骤，提供了从基础向量检索到高级图谱、智能体、纠错与自反评估的全栈能力。开发者可根据业务需求选择合适的检索器，并通过丰富的配置项与可观测性工具实现稳定高效的检索系统。

    附录：API使用示例与最佳实践¶

    • 统一调用方式
    • 所有检索器均实现 Retriever 接口，调用 Retrieve(ctx, queries, topK) 获取结果
    • 结果结构：每个查询对应一个 RetrievalResult，包含查询文本、块集合、答案与可选元数据
    • 配置与参数
    • Native：向量存储、嵌入模型、LLM、TopK、工作目录、日志、追踪
    • Advanced：向量存储、嵌入模型、LLM、TopK、日志；可选启用 RAG-Fusion/改写/回溯
    • Graph：图存储、向量存储、嵌入模型、LLM、TopK、深度、限制、文档存储、提示模板、自定义步骤、日志、追踪
    • Agentic：Agent 实例、日志
    • CRAG：向量存储、嵌入模型、评估器、LLM、WebSearcher、文档存储、TopK、日志、追踪
    • Self-RAG：向量存储、嵌入模型、评估器、LLM、文档存储、TopK、阈值、最大重试次数、日志、追踪
    • 并发与性能
    • 使用 retrieval/service.retriever 的并发路径处理批量查询
    • 合理设置 topK 与过滤条件，避免过度检索
    • 在 GraphRAG 中利用图遍历深度与限制控制搜索规模
    • 最佳实践
    • 先用 Native 快速验证，再按需引入 Advanced/Graph/Agentic/CRAG/Self-RAG
    • 为每个检索器注入 Tracer 与 Logger，便于监控与排错

    • 对 CRAG/Self-RAG 的评估与回退逻辑进行灰度与阈值调优
