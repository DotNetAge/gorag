    存储API¶

    本文件为 GoRAG 的存储API提供完整文档，覆盖以下存储接口与实现： - 文档存储（DocStore）：用于持久化文档与块（Chunk/Node），支持父子关系与检索。 - 图存储（GraphStore）：用于实体（节点）与关系（边）的增删改查、邻域查询、社区摘要与语义查询。 - 向量存储（VectorStore）：用于高维向量的插入、相似度搜索、过滤与删除。 - 索引存储（IndexStore）：用于追踪已构建索引的拓扑与配置。 - 内存聊天（ChatMemory）：用于多轮对话历史缓冲。

    同时，文档对比不同后端（内存、SQLite、BoltDB、Milvus、Pinecone、Qdrant、Weaviate）的API差异与特性，给出配置选项、生命周期管理与资源清理、性能优化与并发访问建议，并提供自定义存储实现指南。

    项目结构¶

    GoRAG 将“核心接口”与“具体实现”分层组织： - 核心接口位于 pkg/core/store 与 pkg/core，定义 DocStore、GraphStore、VectorStore、IndexStore、ChatMemory 等。 - 具体实现分布在 pkg/indexing/store（KV/SQL图与文档存储）、pkg/indexing/vectorstore（多种向量数据库）以及 pkg/di（依赖注入容器）。

    核心组件¶

    • DocStore：负责文档与块的持久化，支持按文档或按ID读取、批量写入、删除。
    • GraphStore：负责节点与边的增删改查、邻域遍历、语义查询、社区摘要、关闭。
    • VectorStore：负责向量的插入、相似度搜索（支持过滤）、删除、关闭。
    • IndexStore：负责索引拓扑与配置的注册、查询、删除、列举。

    • ChatMemory：负责会话消息的追加、读取、清空。

    组件详解¶

    DocStore 接口与实现¶

    • 接口能力
    • 文档：设置、获取、删除
    • 块：批量设置、按ID获取、按文档ID获取
    • 典型实现
    • SQLite DocStore：使用 sqlite 驱动，表结构含 documents/chunks，支持事务批量写入、按文档删除级联。
    • Bolt DocStore：使用 bbolt，基于桶（bucket）存储文档与块，键为字符串ID。
    • 生命周期
    • 打开：初始化目录与数据库/桶；SQLite 创建表并建立索引；Bolt 初始化桶。
    • 关闭：SQLite/Bolt 关闭连接。
    • 并发与性能
    • SQLite：使用事务批处理提升写入吞吐；按文档ID的块查询通过索引加速。
    • Bolt：读写均在事务中执行，适合低并发场景；按文档ID遍历块为线性扫描，建议配合应用侧缓存。

    GraphStore 接口与实现¶

    • 接口能力
    • 节点：UpsertNodes
    • 边：UpsertEdges
    • 查询：GetNode、GetNeighbors（支持深度与限制）、Query（原生SQL/受限）
    • 社区：GetCommunitySummaries
    • 关闭：Close
    • 典型实现
    • SQLite GraphStore：使用 nodes/edges 表，支持递归CTE邻域查询、索引加速；Query 支持原生SQL。
    • Bolt GraphStore：使用多桶维护节点、边与邻接列表（出边/入边），GetNeighbors 实现广度优先遍历；Query 当前不支持复杂查询。
    • 生命周期
    • 打开：初始化目录与表/桶；SQLite 创建索引；Bolt 创建桶。
    • 关闭：SQLite/Bolt 关闭连接。
    • 并发与性能
    • SQLite：邻域查询使用递归CTE，注意深度与限制；索引可显著提升查找性能。
    • Bolt：邻接列表去重插入；遍历为顺序扫描，适合小规模图。

    VectorStore 接口与实现¶

    • 接口能力
    • Upsert：批量插入向量
    • Search：向量相似度搜索，支持过滤条件与topK
    • Delete：按ID删除
    • Close：关闭客户端/连接
    • 典型实现
    • 内存向量存储：纯内存映射，预计算范数，快速余弦相似度，TopK 使用快速选择+排序。
    • Milvus：支持集合创建、索引创建、Flush/Load，搜索表达式基于JSON字段包含匹配。
    • Pinecone：通过客户端描述索引、建立连接，支持向量Upsert与按向量查询，过滤为结构化元数据。
    • Qdrant：客户端直连，Upsert/Payload/Filter，支持多种类型匹配。
    • Weaviate：自动确保集合存在，向量与属性分离存储，GraphQL查询近邻并支持过滤。
    • 生命周期
    • 打开：各实现负责连接/客户端初始化、集合/索引准备。
    • 关闭：释放连接或客户端。
    • 并发与性能
    • 内存实现：读写锁保护，适合单机低并发；TopK 优化避免全量排序。
    • 外部向量库：利用各自索引与近似最近邻算法，支持大规模高维向量检索。

    性能与并发¶

    • DocStore
    • SQLite：批处理事务写入；按文档ID查询块使用索引；删除文档时级联删除块。
    • Bolt：键值读写在事务内执行；按文档ID遍历块为线性扫描，建议应用侧缓存。
    • GraphStore
    • SQLite：递归CTE邻域查询，合理设置 depth 与 limit；为边表建立源/目标索引。
    • Bolt：邻接列表去重插入；遍历采用队列与访问标记，注意深度与限制。
    • VectorStore
    • 内存：预计算范数与快速选择+排序，适合小规模高并发；注意内存占用。
    • 外部向量库：Milvus/Pinecone/Qdrant/Weaviate 依赖各自索引策略与近似检索，建议根据数据规模与延迟要求选择合适索引类型与参数。
    • 并发
    • 接口方法接收 context，便于超时与取消控制。
    • 内存实现使用读写锁；外部实现通常由SDK内部处理并发，需遵循各自客户端并发模型。

    故障排查¶

    • DocStore
    • SQLite：检查表是否创建成功、索引是否存在；确认事务提交路径。
    • Bolt：确认桶初始化成功、文件权限与路径存在。
    • GraphStore
    • SQLite：递归CTE深度过大可能导致性能问题；检查索引命中情况。
    • Bolt：邻接列表重复插入与去重逻辑；查询不到节点时返回空而非错误。
    • VectorStore
    • 外部向量库：确认客户端连接、集合/索引存在且可用；检查维度与索引类型匹配；过滤表达式语法是否符合SDK要求。
    • 生命周期

    结论¶

    GoRAG 的存储API通过清晰的接口抽象与多样化的实现，满足从本地轻量到云端大规模的多种场景需求。开发者可根据数据规模、查询模式与部署环境选择合适的后端，并结合依赖注入容器进行统一管理与生命周期控制。

      • 默认路径：gorag_graph.bolt
      • 桶：nodes、edges、out_edges、in_edges、communities
