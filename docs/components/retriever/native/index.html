    基础检索器¶

    Native RAG（基础检索增强生成）是 RAG 技术中最简洁、最直接的实现形式。它的工作原理可以类比为在图书馆中查找资料：工作人员根据你的问题在卡片目录中找到最相关的几本书，然后把书中的相关内容念给你听。

    核心特点¶

    特点 说明
    简单直接 没有复杂的查询转换或结果优化，流程清晰
    性能最优 零额外开销，每次查询只需一次向量检索和一次 LLM 调用
    易于调试 流程透明，出现问题时容易定位原因
    零学习曲线 5 分钟即可上手

    一句话理解¶

    Native RAG = 向量检索 + 上下文拼接 + LLM 生成

    为什么需要 Native RAG？¶

    它的价值所在¶

    虽然 Advanced RAG、Agentic RAG 等高级模式功能更强大，但 Native RAG 仍然有其不可替代的价值：

    场景 原因
    快速验证想法 最简单的实现，5 分钟跑通流程
    性能敏感场景 没有额外开销，延迟最低
    简单问答 复杂策略反而可能带来负面影响
    教学入门 理解 RAG 基本原理的最佳起点

    局限性¶

    当然，Native RAG 也有其局限性：

    局限性 说明
    语义鸿沟 口语化问题与文档表述差异大时，检索效果差
    关键词丢失 专有名词等精确匹配容易被忽略
    单一检索 无法处理需要多策略组合的复杂场景

    核心原理¶

    完整工作流程¶

    Native RAG 的处理流程分为四个步骤：

    flowchart LR A[用户问题] --> B[向量化查询] B --> C[向量数据库检索] C --> D[获取 Top-K 文档块] D --> E[拼接上下文] E --> F[LLM 生成答案] F --> G[返回答案] style A fill:#fff3e0 style G fill:#e8f5e8

    每一步详解¶

    第一步：向量化查询

    当用户提出问题时，系统使用 Embedder 将问题文本转换为向量表示。这个向量是问题语义在数学空间中的表达。

    // 向量化示意
queryVector := embedder.Embed("如何学习 Go 语言？")
// 输出示例：[0.123, -0.456, 0.789, ...]  // 1536 维向量

    第二步：向量检索

    将查询向量与向量数据库中存储的文档块向量进行相似度计算，返回最相似的 K 个文档块。

    // 检索示意
chunks := vectorStore.Search(queryVector, topK=5)
// 返回与问题最相关的 5 个文档块

    第三步：拼接上下文

    将检索到的文档块内容与用户问题组合，形成完整的提示词。

    系统提示：你是一个有帮助的助手。请根据以下上下文回答用户问题。

上下文：
[文档块 1] Go 语言是由 Google 开发的一种编译型语言...
[文档块 2] 学习 Go 语言推荐从官方文档开始...
[文档块 3] Go 的核心特性包括并发支持、简洁语法...

用户问题：如何学习 Go 语言？

    第四步：LLM 生成答案

    将拼接好的提示词发送给 LLM，生成最终答案。

    与其他 RAG 模式的对比¶

    功能对比¶

    特性 Native RAG Advanced RAG Agentic RAG Graph RAG
    检索优化 ❌ 无 ✅ HyDE/Rewrite/RRF ✅ 智能路由 ✅ 图增强
    意图识别 ❌ 无 ❌ 无 ✅ 支持 ❌ 无
    图关系 ❌ 无 ❌ 无 ✅ 可选 ✅ 原生
    复杂度 ⭐ 最低 ⭐⭐ 中等 ⭐⭐⭐ 较高 ⭐⭐⭐ 较高
    延迟 最低 中等 较高 较高
    LLM 调用次数 1 次 2-3 次 多次 2-3 次

    性能对比¶

    指标 Native RAG 说明
    延迟 基准 最快
    成本 基准 最低
    召回率 一般 简单场景足够
    准确率 一般 复杂场景有限

    选择建议¶

    问题复杂度低，答案直接 → Native RAG ✓
问题需要改写优化 → Advanced RAG
问题需要多步推理 → Agentic RAG
问题涉及实体关系 → Graph RAG

    适用场景¶

    什么时候选择 Native RAG？¶

    场景 说明 示例
    简单问答 问题明确，答案直接来自文档 FAQ 检索、产品手册查询
    文档搜索 用户需要找到相关文档片段 法律条款检索、技术文档搜索
    快速原型 验证 RAG 概念和想法 POC 项目、技术调研
    性能敏感 对延迟有严格要求 实时客服、高并发场景
    教学入门 学习 RAG 基本原理 培训、教程

    什么时候不用 Native RAG？¶

    场景 原因 推荐替代
    复杂多跳查询 无法处理需要多步推理的问题 Advanced RAG
    模糊意图识别 无法判断用户真实意图 Agentic RAG
    关系推理 无法理解实体间关系 Graph RAG
    高质量生成 无法自我纠错和优化 Self-RAG

    使用示例¶

    基本用法¶

    import (
    "github.com/DotNetAge/gorag/pkg/retriever/native"
)

// 创建 Native Retriever
retriever := native.NewRetriever(
    vectorStore,  // 向量存储
    embedder,     // 向量化模型
)

// 执行检索
chunks, err := retriever.Retrieve(ctx, query, topK=5)

    完整 Pipeline 示例¶

    import (
    "github.com/DotNetAge/gorag/pkg/pipeline"
    "github.com/DotNetAge/gorag/pkg/steps/vector"
    "github.com/DotNetAge/gorag/pkg/steps/generate"
)

// 创建 Pipeline
p := pipeline.New[*core.RetrievalContext]()

// 添加 Step
p.AddStep(vector.NewSearchStep(vecStore, embedder))
p.AddStep(generate.NewGenerationStep(llm))

// 执行
state := &core.RetrievalContext{
    Query: &core.Query{Text: "Go 语言的优势是什么？"},
}
err := p.Execute(ctx, state)

    实现详解¶

    接口定义¶

    Native Retriever 实现了 core.Retriever 接口：

    // pkg/core/interfaces.go
type Retriever interface {
    // Retrieve 执行检索，返回相关的文档块
    Retrieve(ctx context.Context, query *Query, topK int) ([]*Chunk, error)
}

    内部实现¶

    // pkg/retriever/native/native.go
func (r *NativeRetriever) Retrieve(ctx context.Context, query *Query, topK int) ([]*Chunk, error) {
    // 1. 将查询向量化
    queryVector, err := r.embedder.Embed(query.Text)
    if err != nil {
        return nil, err
    }

    // 2. 在向量数据库中检索
    chunks, err := r.vectorStore.Search(queryVector, topK)
    if err != nil {
        return nil, err
    }

    return chunks, nil
}
