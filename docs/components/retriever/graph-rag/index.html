    知识图谱检索器¶

    GraphRAG 将基于向量的检索与知识图谱的结构化信息相结合，能够提供更具上下文准确性和全面性的答案，尤其适用于复杂或涉及多条关系路径的查询。

    核心原理¶

    flowchart TB A[用户查询] --> B[实体提取] B --> C[图谱遍历] B --> D[向量检索] C --> E[上下文融合] D --> E E --> F[答案生成]

    流水线结构¶

    GraphRAG 的工作流程通常包含以下步骤：

    1. 实体提取 (Entity Extraction)：识别用户查询中的关键实体
    2. 向量搜索：搜索相关的文档分块（同传统 RAG）
    3. 图谱遍历 (Graph Traversal)：在 GraphStore 中搜索相关实体、关系以及子图结构
    4. 上下文增强 (DocStore Enrichment)：利用 DocStore 根据检索到的分块 ID 或实体关联，召回完整的 父文档 (Parent Document) 或原文证据，提供更丰富的背景
    5. 生成：利用合并后的信息生成更全面的响应

    知识图谱基础¶

    核心概念¶

    graph LR A[实体 Node] -->|关系| B[实体] A -->|关系| C[实体] B -->|关系| D[实体] style A fill:#e1f5fe style B fill:#e1f5fe style C fill:#e1f5fe style D fill:#e1f5fe

    实体类型¶

    类型 示例
    人物 (PERSON) Alice, Bob, 马斯克
    组织 (ORGANIZATION) Google, MIT, 苹果公司
    概念 (CONCEPT) AI, 机器学习, RAG
    地点 (LOCATION) 北京, 美国, 太平洋
    事件 (EVENT) 收购, 发布, 会议

    关系类型¶

    类型 示例
    WORKS_AT Alice -WORKS_AT→ Google
    KNOWS Alice -KNOWS→ Bob
    ACQUIRED 苹果 -ACQUIRED→ Beats
    DEVELOPS OpenAI -DEVELOPS→ GPT-4
    IS_A GPT-4 -IS_A→ 大语言模型

    核心优化：DocStore 集成¶

    通过集成 DocStore，GraphRAG 支持 父子文档检索 (PDR)：

    • 证据溯源：当图谱找到两个实体的关系时，从 DocStore 召回提及该关系的原始长文本段落，增强回答的可信度
    • 上下文补全：防止由于分块过小（Chunking）导致的语义断层
    • 关系验证：通过原始文档验证图谱中的关系是否准确

    与其他 RAG 模式的区别¶

    特性 Native RAG Advanced RAG Agentic RAG Graph RAG
    检索方式 向量 向量+优化 智能选择 向量+图谱
    关系推理 ❌ 无 ❌ 无 ✅ 可选 ✅ 核心
    多跳推理 ❌ 无 ❌ 无 ✅ ✅ 强
    上下文完整性 低 中 中 高

    适用场景¶

    • 涉及多实体、复杂关联关系的查询
    • 需要全局性概览而非局部性信息的场景
    • 依赖于领域内明确定义的语义关系的应用
    • 需要关系发现和推理的场景
    • 学术研究和知识管理

    不适用场景¶

    • 简单关键词检索（使用 Native RAG 即可）
    • 实时性要求极高的场景（图谱构建有额外开销）
    • 非结构化数据为主的场景

    使用示例¶

    基本用法¶

    import (
    "github.com/DotNetAge/gorag/pkg/retriever/graph"
)

retriever := graph.NewRetriever(
    vectorStore,
    graphStore,
    docStore,
    embedder,
    llm,
)

    通过顶层入口使用¶

    import "github.com/DotNetAge/gorag"

rag, err := gorag.DefaultGraphRAG(
    gorag.WithWorkDir("./data"),
)

    支持的图存储 (GraphStore)¶

    GoRAG 提供多种图存储实现：

    1. SQLite (嵌入式)¶

    推荐用于本地和轻量级应用。支持递归 CTE 遍历：

    graphStore, _ := sqlite.NewGraphStore("./data/graph.db")

    2. BoltDB (嵌入式)¶

    纯 Go 实现，极高性能的本地 K/V 索引：

    graphStore, _ := bolt.NewGraphStore("./data/graph.db")

    3. Neo4j (工业级)¶

    支持大规模知识图谱，具备完整的 Cypher 查询能力：

    import "github.com/DotNetAge/gorag/pkg/indexing/store/neo4j"

graphStore, _ := neo4j.NewGraphStore(
    "bolt://localhost:7687",
    "neo4j",
    "password",
    "neo4j",
)

    工作流程详解¶

    1. 索引阶段¶

    flowchart TB A[文档] --> B[解析] B --> C[分块] C --> D[实体提取] D --> E[关系提取] E --> F[构建知识图谱] subgraph 存储 F --> G[GraphStore] F --> H[VectorStore] F --> I[DocStore] end

    2. 查询阶段¶

    flowchart TB A[用户查询] --> B[实体识别] B --> C[图谱检索] B --> D[向量检索] C --> E[关系路径] D --> F[相关文档块] E --> G[上下文融合] F --> G G --> H[LLM 生成] H --> I[答案]

    使用 Cypher 模板进行深度推理¶

    通过 CypherStep，你可以定义特定的图路径查询模板，帮助 LLM 发现文档中未直接描述的隐藏关系：

    import "github.com/DotNetAge/gorag/pkg/retriever/graph"

// 定义：查找某人入职公司的 CEO
const ceoFinderTemplate = `
    MATCH (p:Entity {id: $id})-[:WORKS_AT]->(c:Entity)<-[:CEO_OF]-(ceo:Entity)
    RETURN ceo.id as name
`

retriever := graph.NewRetriever(
    vectorStore,
    graphStore,
    docStore,
    embedder,
    llm,
    // 注入自定义 Cypher 推理步骤
    graph.WithCustomStep(graph.NewCypherStep(graphStore, ceoFinderTemplate, logger)),
)

    配置选项¶

    选项 类型 默认值 说明
    WithGraphStore GraphStore 必需 图谱存储
    WithDocStore DocStore nil DocStore 增强
    WithMaxHops int 3 最大跳数

    注意事项¶

    1. 图谱构建成本：初始图谱构建需要额外的 LLM 调用
    2. 存储开销：需要同时存储向量、文档和图谱
    3. 查询复杂度：多跳查询可能增加响应时间
    4. Neo4j 配置：生产环境建议使用 Neo4j 获得更好的性能
