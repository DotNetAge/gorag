    自主智能体检索器¶

    AgenticRAG 代表了 RAG 的最高级形式，其中 LLM 扮演自主智能体（Agent）的角色，它根据自身推理决定何时进行搜索、如何搜索，以及何时获取了足够的信息来提供最终答案。

    核心原理¶

    flowchart TB A[用户查询] --> B{Agent 推理} B --> C{选择工具} C --> D1[向量检索] C --> D2[图谱检索] C --> D3[Web 搜索] C --> D4[DocStore 全文搜索] D1 --> E{评估结果} D2 --> E D3 --> E D4 --> E E -->|不充分| B E -->|充分| F[生成答案]

    关键核心能力¶

    1. 自主工具使用 (Autonomous Tool Use)：智能体可以在多种检索工具（如向量搜索、图谱搜索、网页搜索、DocStore 全文搜索等）之间进行选择
    2. 自我修正 (Self-Correction)：如果初始检索结果不满意，智能体可以重新制定计划，尝试不同的检索策略
    3. 多步推理回环 (Multi-hop Reasoning)：参与迭代式循环，不断精炼搜索和最终生成的答案
    4. 动态规划：智能体能够规划检索路径，决定先做什么、后做什么

    核心优化：DocStore 集成¶

    在 Agentic 模式下，DocStore 不仅仅是存储，更是智能体的一项重要技能工具：

    • 父文档深度钻取 (PDR)：智能体可以发出"获取该分块完整上下文"的指令，通过 DocStore 调取原文
    • 元数据检索：智能体可以基于 DocStore 存储的丰富元数据（如作者、发布日期等）进行筛选和过滤
    • 上下文补全：当智能体发现分块信息不完整时，可以主动获取父文档进行补充

    支持的智能体工作流¶

    • CRAG (Corrective RAG)：评估检索质量，在内部数据不足时调用外部网页搜索
    • Self-RAG：生成响应，并基于检索到的信息对其进行自我批判（Critique）和精炼（Refine）
    • 任务导向型智能体：为特定检索任务专门设计的智能体
    • 对话式智能体：能够处理多轮对话，上下文记忆

    与其他 RAG 模式的区别¶

    特性 Native RAG Advanced RAG Agentic RAG
    检索策略 固定 固定（可配置） 动态选择
    工具选择 单一 单一 多种工具
    自我修正 ❌ 无 ❌ 无 ✅ 支持
    迭代循环 ❌ 无 ❌ 无 ✅ 支持
    复杂度 ⭐ 最低 ⭐⭐ 中等 ⭐⭐⭐ 最高
    LLM 调用次数 1 次 2-3 次 多次

    适用场景¶

    • 复杂的多跳推理问答
    • 需要多源信息综合判断的场景
    • 用户意图不明确，需要智能体主动探索
    • 对话式问答系统
    • 需要动态调整检索策略的场景

    Agentic 核心智能模式¶

    GoRAG 的 Agentic 查询管线整合了当前最前沿的几种智能 RAG 论文与范式：

    1. Query Routing (动态路由)¶

    使用轻量级 LLM 判断用户意图。如果是闲聊，直接回答；如果是通用常识，调用 Web Search；如果是域内知识，走向量检索。

    2. Query Decomposition (查询分解)¶

    针对复杂问题（如"A公司和B公司在2023年的营收差距是多少？"），LLM 会将其分解为多个子问题（Sub-queries），然后并发检索。

    3. CRAG (Corrective RAG - 检索纠错)¶

    在召回 Chunk 之后，不立刻生成答案，而是让一个轻量级评估器打分（Relevant / Irrelevant / Ambiguous）。如果相关性极差，触发备用工具（如自动进行 Google 搜索补充知识）。

    4. Self-RAG (自省与幻觉控制)¶

    生成答案后，进行 Grounding 检查。判断生成的答案是否完全由召回的 Chunk 支持，如果不一致，触发重试或拒绝回答（拒答机制）。

    Agentic SOP 环节定义¶

    在 GoRAG 中，这些复杂的逻辑被封装为一个个高级的 Step：

    环节 (Step) 说明 核心输入/输出
    Intent Router Step 分类器。根据分类结果，决定 State 下一步走向哪个子管线或跳过某些步骤。 Query -> IntentType
    Decomposition Step 将长难句拆解为 []SubQuery。 Query -> []Query
    Parallel Retrieval Step 针对多个 SubQuery 开启 Goroutine 并发检索，最终合并去重。 []Query -> []Chunk
    CRAG Evaluator Step (知识纠错) 评估召回的 Chunk 是否足以回答问题。如果不满足，可触发挂载的外部工具。 []Chunk + Query -> Score & Action
    Tool Execution Step (工具执行) 标准接口，允许系统调用预定义的 Go 函数（如 SearchAPI(), Calculator()）。 ToolRequest -> ToolResult
    Self-Reflection Step (自我反思) 使用 LLM-as-a-Judge 审查输出草稿。如果判定为幻觉 (Hallucination)，返回 Error 触发重试逻辑。 Draft + []Chunk -> Boolean (Pass/Fail)

    性能瓶颈与优化策略¶

    痛点解析： 如果严格按照学术界的 Agentic RAG（如 CRAG 或 Self-RAG）执行，一个极端的查询可能需要顺序调用 4-5 次大语言模型（Router -> Decompose -> Eval -> Generate -> Reflect）。这将导致极高的延迟（TTFT 可能达到 10~20 秒）和昂贵的 API 成本。

    1. 大小模型混用路由 (Model Routing)¶

    不是所有节点都需要昂贵的 GPT-4 级别模型：

    • 意图路由 (Router) / CRAG 评估 / Self-RAG 打分：这些本质上是分类或打分任务。建议在这些 Step 中注入极其快速且廉价的小模型（如 GPT-4o-mini, Claude-3-Haiku）。它们的响应通常在百毫秒级。
    • 最终生成 (Generation)：仅在生成最终大段回复时，才调用最强的推理模型。

    2. 步骤融合 (Prompt Merging & Joint Extraction)¶

    通过精妙的 Prompt Engineering，将多个串行的 LLM 调用合并为一次请求：

    • 合并 Router 与 Decompose：让小模型一次性结构化输出意图和拆解的子问题。例如：{"intent": "complex_rag", "sub_queries": ["Q1", "Q2"]}。
    • 合并 Generation 与 Self-RAG：在最终生成的 Prompt 中，严格要求模型在句尾附加置信度标记或文献引用。

    3. 激进的短路机制 (Aggressive Short-circuiting)¶

    5 次调用只是应对"最坏情况（Worst-case Scenario）"的安全网：

    • Router Step 是整个系统性能的守护者。如果 Router 判断用户输入的是一个标准查询，系统会立即短路，跳过所有复杂的评估与分解，直接降级执行只有 2 步的 Standard RAG。
    • 真正的 Agentic 管线应当被视为一种"昂贵但必要的保险"，仅对那 5%~10% 需要复杂多跳推理的长难 Query 生效。

    4. 用户体验补偿 (UX Streaming / Chain of Thought)¶

    对于那 10% 确实需要走完复杂流程的请求：

    • Pipeline 执行过程中，将中间状态（如："🔍 正在将问题拆解为 3 个维度...", "🌐 正在调用 Google 补充背景知识..."）通过流（SSE / WebSocket）推送给前端。
    • 这种类似 OpenAI o1 模型的"思考过程 (Thinking Process)"不仅能掩盖延迟，还能显著提升用户对系统专业度的信任感。

    不适用场景¶

    • 简单的一问一答（使用 Native RAG 即可）
    • 延迟敏感场景（Agent 循环增加额外延迟）
    • 资源受限环境（多次 LLM 调用成本较高）

    使用示例¶

    基本用法¶

    import (
    "github.com/DotNetAge/gorag/pkg/retriever/agentic"
)

// 定义智能体可用的工具
tools := []tool.Tool{
    vectorSearchTool,   // 向量检索工具
    graphSearchTool,    // 图谱检索工具
    webSearchTool,      // Web 搜索工具
    docStoreTool,       // DocStore 全文搜索工具
}

retriever := agentic.NewRetriever(
    llm,
    tools,
    agentic.WithMaxIterations(5),
    agentic.WithTemperature(0.7),
)

response, err := retriever.Retrieve(ctx, "请分析苹果公司过去五年的财务表现")

    通过顶层入口使用¶

    import "github.com/DotNetAge/gorag"

rag, err := gorag.DefaultAgenticRAG(
    gorag.WithWorkDir("./data"),
    gorag.WithMaxIterations(3),
)

    配置 DocStore 工具¶

    // 创建 DocStore 工具
docStoreTool := tools.NewDocStoreTool(
    docStore,
    tools.WithTopK(10),
    tools.WithParentDocumentRetrieval(true), // 启用父文档检索
)

// 将工具注册到智能体
agent := agentic.NewRetriever(
    llm,
    []tool.Tool{vectorSearchTool, docStoreTool},
)

    Agent 工具箱¶

    向量检索工具 (VectorSearchTool)¶

    vectorTool := tools.NewVectorSearchTool(
    vectorStore,
    embedder,
    tools.WithTopK(10),
)

    图谱检索工具 (GraphSearchTool)¶

    graphTool := tools.NewGraphSearchTool(
    graphStore,
    graphTool.WithMaxHops(3), // 最多跳数
)

    Web 搜索工具 (WebSearchTool)¶

    webTool := tools.NewWebSearchTool(
    searchAPI,
    webTool.WithResultLimit(10),
)

    DocStore 全文搜索工具 (DocStoreTool)¶

    docTool := tools.NewDocStoreTool(
    docStore,
    docTool.WithFullTextSearch(true),
    docTool.WithMetadataFilter(true),
)

    工作流程详解¶

    1. 意图理解阶段¶

    flowchart TB A[用户查询] --> B[LLM 分析意图] B --> C{意图类型} C -->|事实查询| D[执行检索] C -->|分析查询| E[制定计划] C -->|比较查询| F[多源检索]

    2. 工具选择阶段¶

    flowchart LR A[查询分析结果] --> B{可用工具} B --> C[向量检索] B --> D[图谱检索] B --> E[Web 搜索] B --> F[DocStore] C -.->|优先级| G[工具排序] D -.->|优先级| G E -.->|优先级| G F -.->|优先级| G G --> H[选择最优工具]

    3. 迭代精炼阶段¶

    flowchart TB A[初始检索] --> B{评估结果} B -->|不充分| C[反思原因] C --> D[调整策略] D --> E[重新检索] E --> B B -->|充分| F[生成答案]

    配置选项¶

    选项 类型 默认值 说明
    WithMaxIterations int 5 最大迭代次数
    WithTemperature float 0.7 LLM 温度参数
    WithTools []Tool 必需 可用工具列表
    WithTimeout time.Duration 60s 超时时间

    注意事项¶

    1. LLM 重要性：Agentic RAG 依赖强大的 LLM 进行推理和决策，建议使用 GPT-4 或同等能力模型
    2. 响应延迟：由于迭代循环和多次 LLM 调用，Agentic RAG 比其他模式延迟高
    3. 成本控制：建议设置最大迭代次数和超时时间，防止无限循环
    4. 工具定义：工具描述要清晰，帮助 LLM 正确选择
