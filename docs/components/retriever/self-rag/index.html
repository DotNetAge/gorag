    自我反思检索器¶

    Self-RAG (Self-Reflective RAG) 是一种具备自我批判和精炼能力的检索架构。它不仅仅直接输出检索结果，还会评估检索分块的相关度、回答的忠实度以及有用性。如果生成的回答未通过质量检查，Self-RAG 会触发迭代式的重写或重新检索。

    核心原理¶

    flowchart TB A[用户查询] --> B[向量检索] B --> C[获取 Top-K 文档块] C --> D[LLM 生成初始答案] D --> E{自我反思评估} E -->|评估通过| F[返回答案] E -->|评估不通过| G[精炼或重新检索] G --> B

    工作流：生成 -> 反思 -> 迭代¶

    1. 初步检索：从向量库获取 top-K 相关分块
    2. 生成回答：LLM 首次根据分块内容生成初步答案
    3. 反思评估 (Self-Reflection)
    4. 评估回答是否被分块支持（忠实度）
    5. 评估回答是否真的解决了用户问题（有用性）
    6. 评估检索到的分块是否与查询相关（相关性）
    7. 循环迭代：如果评估分数低于阈值，系统根据反馈（Feedback）进行精炼或重新检索

    反思令牌机制¶

    Self-RAG 引入三种反思令牌：

    令牌 值 含义
    REL 1/0 检索结果是否与查询相关
    RRF 1/0 回答是否被检索结果支持
    IS 1/0 回答是否完整、有用

    核心优化：DocStore 集成¶

    通过集成 DocStore，Self-RAG 在精炼（Refinement）阶段具有显著优势：

    • 召回全量父文档 (PDR)：如果由于分块过碎导致回答在反思阶段被判断为"不忠实"，系统可以通过 DocStore 获取分块所在的整篇 父文档 内容进行重写，从而确保回答的语义完整性
    • 上下文一致性验证：通过对比 DocStore 中的原始结构，确认生成的回复没有断章取义

    与其他 RAG 模式的区别¶

    特性 Native RAG Advanced RAG Agentic RAG Self-RAG
    检索方式 一次性 优化后 智能路由 自我反思循环
    质量评估 ❌ 无 ❌ 无 部分 ✅ 完整
    重新检索 ❌ 无 ❌ 无 ❌ 无 ✅ 支持
    幻觉减少 ❌ 无 部分 部分 ✅ 强
    迭代次数 1 次 1 次 多次 多次
    延迟 最低 中等 较高 最高

    CRAG 与 Self-RAG 的关系¶

    CRAG 与 Self-RAG 都属于 Agent 工作流的子集，是 Agent 模式在 RAG 领域的具体应用。

    核心区别¶

    模式 工作流类型 核心智能体特征
    标准 RAG 线性 (Sequential) 无
    CRAG 条件分支 (Conditional) 检索质量评估、工具切换
    Self-RAG 循环/反思 (Feedback Loop) 自我批判、内容精炼
    通用 Agentic RAG 多步推理 (Multi-hop/ReAct) 任务分解、动态规划、工具自由调用

    CRAG vs Self-RAG 对比¶

    • CRAG (Corrective RAG)：扮演"质量检测员"，评估检索到的文档与查询的相关性。如果不相关，会调用外部工具（如 Web Search）进行补充。具备"自我评估"和"外部工具调用"能力。

    • Self-RAG：扮演"评论家"和"编辑"，对检索到的分块进行打分（相关性），并对生成的答案进行多维度自我批判（忠实度、有用性）。具备"自我反思"和"递归精炼"能力。

    适用场景¶

    • 对答案质量要求极高的场景
    • 需要减少幻觉的应用
    • 事实准确性要求严格的领域（法律、医疗、金融）
    • 需要可解释答案来源的场景

    不适用场景¶

    • 简单闲聊或不需要高精度的场景
    • 延迟敏感场景（迭代循环增加额外延迟）
    • 资源受限环境

    使用示例¶

    基本用法¶

    import (
    "github.com/DotNetAge/gorag/pkg/retriever/selfrag"
)

retriever := selfrag.NewRetriever(
    vectorStore,
    embedder,
    evaluator,
    llm,
    selfrag.WithThreshold(0.8),
    selfrag.WithMaxRetries(3),
)

response, err := retriever.Retrieve(ctx, "法国的首都是什么？")

    启用 DocStore 增强¶

    retriever := selfrag.NewRetriever(
    vectorStore,
    embedder,
    evaluator,
    llm,
    selfrag.WithThreshold(0.8),
    selfrag.WithMaxRetries(3),
    selfrag.WithDocStore(docStore), // 启用 DocStore 增强
)

    通过顶层入口使用¶

    由于 GoRAG 目前没有 DefaultSelfRAG 便捷函数，请使用以下方式：

    import (
    "github.com/DotNetAge/gorag"
    "github.com/DotNetAge/gorag/pkg/retriever/selfrag"
)

// 方式一：使用默认配置
app, err := gorag.DefaultNativeRAG(...)

// 方式二：使用完整的 Self-RAG Retriever
retriever := selfrag.NewRetriever(
    vectorStore,
    embedder,
    evaluator,  // core.RAGEvaluator
    llm,
    selfrag.WithTopK(5),
    selfrag.WithThreshold(0.7),
    selfrag.WithMaxRetries(3),
)

    工作流程详解¶

    1. 检索阶段¶

    flowchart LR A[用户查询] --> B[Embedding] B --> C[向量数据库] C --> D[Top-K 相关文档块] D --> E[检查分块相关性 REL]

    2. 生成阶段¶

    flowchart LR A[相关文档块 + 查询] --> B[LLM 生成答案] B --> C[评估忠实度 RRF] C --> D[评估有用性 IS]

    3. 迭代精炼阶段¶

    flowchart TB A[初始答案 + 评估分数] --> B{分数 >= 阈值?} B -->|是| C[返回答案] B -->|否| D{剩余重试次数 > 0?} D -->|是| E[根据反馈精炼] E --> F[重新生成答案] F --> A D -->|否| G[返回最佳答案或拒答]

    评估指标详解¶

    相关性评估 (REL)¶

    评估检索到的文档块与用户查询的相关程度：

    得分 含义
    1 文档块与查询高度相关
    0 文档块与查询不相关

    忠实度评估 (RRF)¶

    评估生成的答案是否被检索到的文档块支持：

    得分 含义
    1 答案完全被文档支持
    0 答案与文档矛盾或未被支持

    有用性评估 (IS)¶

    评估生成的答案是否有助于回答用户问题：

    得分 含义
    1 答案完整、有用
    0 答案不完整或无用

    配置选项¶

    选项 类型 默认值 说明
    WithThreshold float 0.7 通过阈值
    WithMaxRetries int 3 最大重试次数
    WithDocStore DocStore nil DocStore 增强
    WithEvaluator Evaluator 内置 自定义评估器

    注意事项¶

    1. LLM 要求：Self-RAG 需要强大的 LLM 支持，建议使用 GPT-4 或同等能力的模型
    2. 延迟较高：由于自我反思循环，Self-RAG 比其他模式延迟高，不适合实时性要求高的场景
    3. 成本较高：每次迭代都需要额外的 LLM 调用
    4. 阈值调优：根据实际场景调整阈值，避免过度迭代或过早放弃
