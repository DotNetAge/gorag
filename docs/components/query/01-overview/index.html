    Query Processing - 查询处理¶

    什么是查询处理？¶

    查询处理是 RAG 系统的"智囊团"，负责在用户提出问题后、正式检索前，对问题进行一系列智能分析和转换。就像一位经验丰富的图书管理员，在帮你找书之前，会先理解你真正想问什么，然后用最合适的方式去查找。

    为什么需要查询处理？¶

    用户的提问往往是口语化、不完整、甚至带有歧义的。直接将这样的原始问题送去检索，效果往往不理想。

    用户原始问题 问题分析 处理后的问题
    "它去年的表现怎么样？" "它"指代不明 "XX公司 2023 年财务表现"
    "Go 咋样？" 过于简短、口语化 "Go 编程语言的优势和特点"
    "Transformer 是啥" 缩写、多义词 "Transformer 模型原理和应用"

    一句话理解¶

    查询处理 = 理解用户真正想问什么 → 用最合适的方式表达 → 提高检索质量

    为什么需要查询处理？¶

    用户问题的复杂性¶

    现实中的用户问题远比想象中复杂：

    flowchart LR subgraph 问题类型 A1[口语化] --> A[用户问题] A2[缩写] --> A A3[指代不明] --> A A4[表述不完整] --> A A5[多义词] --> A end A --> B[检索效果] B --> C[可能检索不到相关内容] style A fill:#ffcdd2 style C fill:#ffcdd2

    查询处理的四大价值¶

    价值 说明
    消除歧义 明确"它"、"那个"等指代词的真实含义
    补充信息 将缩写、简称扩展为完整表述
    优化表达 将口语转化为更适合检索的标准表述
    扩展视角 生成多个相关查询，扩大检索范围

    核心组件详解¶

    1. Intent Router（意图路由器）¶

    功能：判断用户的真实意图，选择最合适的处理流程。

    为什么重要？

    不是所有问题都需要检索。意图路由可以智能判断：

    flowchart TD A[用户问题] --> B[意图分类] B --> C{判断意图} C -->|闲聊| D[直接回答] C -->|常识| E[使用内部知识] C -->|需要检索| F[执行检索流程] style D fill:#e8f5e8 style E fill:#fff3e0 style F fill:#e1f5fe

    支持的意图类型：

    意图 说明 处理方式
    chat 闲聊、寒暄 跳过检索，直接生成友好回复
    common_fact 常识性问题 使用 LLM 内部知识回答
    domain_specific 领域知识查询 执行完整检索流程
    relational 关系推理查询 触发图检索
    fact_check 事实核查 精确检索 + 多方验证

    使用示例：

    import "github.com/DotNetAge/gorag/pkg/query"

router := query.NewIntentRouter(myLLM)

intent, err := router.Classify(ctx, "你好，今天天气怎么样？")
// intent = "chat" → 跳过检索

intent, err = router.Classify(ctx, "Go 语言是什么？")
// intent = "domain_specific" → 执行检索

    2. Entity Extractor（实体提取器）¶

    功能：从问题中识别和提取关键实体。

    为什么重要？

    实体是检索的关键锚点。准确提取实体可以大幅提升检索精度：

    原始问题 提取的实体
    "苹果公司的 CEO 是谁？" {"type": "ORG", "name": "苹果公司"}
    "张三和李四是什么关系？" {"type": "PERSON", "name": "张三"}, {"type": "PERSON", "name": "李四"}
    "iPhone 15 的价格" {"type": "PRODUCT", "name": "iPhone 15"}

    使用示例：

    extractor := query.NewEntityExtractor(myLLM)

entities, err := extractor.Extract(ctx, "特斯拉和比亚迪谁的市场份额更大？")
// 返回：["特斯拉", "比亚迪", "市场份额"]

    3. Decomposer（问题分解器）¶

    功能：将复杂问题分解为多个简单子问题。

    为什么重要？

    有些问题太复杂，直接检索效果差。分解后可以各个击破：

    flowchart LR subgraph 分解前 A["Go 和 Python 在并发处理上的对比？"] end subgraph 分解后 B1["Go 语言的并发特性？"] B2["Python 的并发处理方式？"] B3["两者在并发上的优缺点对比？"] end A --> B1 A --> B2 A --> B3

    使用示例：

    decomposer := query.NewDecomposer(myLLM)

subQueries, err := decomposer.Decompose(ctx, "收购 A 公司的 B 公司，其 CEO 是谁？")
// 返回：
// 1. "A 公司被哪家收购？"
// 2. "收购 A 公司的 CEO 是谁？"

    4. Rewriter（查询重写器）¶

    功能：将口语化、不完整的问题重写为更规范、更适合检索的表述。

    为什么重要？

    用户的表达方式往往与文档中的表述不同：

    原始问题 重写后问题 原因
    "它去年的表现" "XX公司 2023 年表现" 消除指代
    "Go 咋样" "Go 编程语言" 展开缩写
    "那个很厉害的公司" "特斯拉公司" 补充完整名称

    使用示例：

    rewriter := query.NewRewriter(myLLM)

rewritten, err := rewriter.Rewrite(ctx, "它去年的财报怎样？", map[string]string{"company": "苹果公司"})
// 返回："苹果公司 2023 年财务报告表现如何？"

    5. HyDE Generator（假设文档生成器）¶

    功能：根据问题生成一个假设性的答案文档，然后用这个假设文档去检索。

    为什么重要？

    这解决了"问题与文档表述不对称"的难题。假设文档包含了答案的表述方式，更容易找到真正相关的文档。

    工作原理：

    flowchart LR A[用户问题] --> B[生成假设答案] B --> C[假设答案向量] C --> D[向量检索] D --> E[找到真实相关文档]

    使用示例：

    hyde := query.NewHyDEGenerator(myLLM)

hypotheticalDoc, err := hyde.Generate(ctx, "什么是 Transformer？")
// 返回：
// "Transformer 是一种...（一段假设性的解释文档）"

hydeVector := embedder.Embed(hypotheticalDoc)
// 用假设文档的向量去检索

    6. StepBack Generator（退步提问生成器）¶

    功能：将具体问题"退步"为一个更高层次的抽象问题。

    为什么重要？

    有些问题过于具体，直接检索可能找不到答案。退步到更高层次可以召回更多背景知识：

    具体问题 StepBack 问题
    "为什么爱因斯坦在 1905 年发表相对论？" "爱因斯坦 1905 年的科学生涯和物理学背景"
    "GPT-4 和 BERT 哪个好？" "GPT 系列和 BERT 系列模型的技术区别"

    使用示例：

    stepback := query.NewStepBackGenerator(myLLM)

stepBackQuery, err := stepback.Generate(ctx, "为什么特斯拉的自动驾驶比 Waymo 差？")
// 返回："特斯拉自动驾驶技术和 Waymo 的技术路线对比"

    工作流程¶

    查询处理完整流程¶

    flowchart TD A[用户问题] --> B[意图分类] B --> C{需要检索？} C -->|否| D[直接生成答案] C -->|是| E[实体提取] E --> F{复杂问题？} F -->|是| G[问题分解] F -->|否| H[继续处理] G --> H H --> I[查询重写] I --> J[HyDE 生成] J --> K[StepBack 生成] K --> L[多路并发检索] L --> M[结果融合] M --> N[答案生成] D --> O[返回答案] N --> O

    组件组合策略¶

    不同场景下，组件的组合方式不同：

    场景 推荐组合
    简单问答 Router + Rewriter
    复杂多跳 Router + Decomposer + Rewriter
    语义模糊 Router + HyDE
    需要背景知识 Router + StepBack
    完整 Advanced RAG 全部组件

    在 GoRAG 中使用¶

    基本用法¶

    import (
    "github.com/DotNetAge/gorag/pkg/query"
    "github.com/DotNetAge/gorag/pkg/pipeline"
)

// 创建查询处理器
processor := query.NewProcessor(
    query.WithIntentRouter(myLLM),
    query.WithEntityExtractor(myLLM),
    query.WithDecomposer(myLLM),
    query.WithRewriter(myLLM),
    query.WithHyDE(myLLM),
    query.WithStepBack(myLLM),
)

// 创建 Pipeline
p := pipeline.New()
p.AddStep(query.NewProcessStep(processor))  // 查询处理
p.AddStep(vector.NewSearchStep(vecStore))   // 检索
p.AddStep(generate.NewGenerationStep(llm))  // 生成

    选择性使用组件¶

    如果不需要所有组件，可以选择性启用：

    processor := query.NewProcessor(
    query.WithIntentRouter(myLLM),  // 必须
    query.WithRewriter(myLLM),      // 可选
    // 其他组件按需添加
)

    配置选项¶

    组件 选项 类型 默认值 说明
    通用 WithTemperature float 0.7 生成温度
    IntentRouter WithIntentThreshold float 0.5 意图置信度阈值
    Decomposer WithMaxSubQueries int 5 最大子问题数
    HyDE WithHyDETemperature float 0.8 HyDE 生成温度
    StepBack WithStepBackLevel int 1 抽象层级

    适用场景¶

    ✅ 适合使用查询处理的场景¶

    场景 说明 核心组件
    口语化问题多 用户表达随意、多样 Rewriter
    多跳推理问题 需要分解复杂问题 Decomposer
    语义模糊查询 问题与文档表述差异大 HyDE
    需要背景知识 问题涉及专业概念 StepBack
    混合问答系统 同时处理闲聊和专业问题 IntentRouter

    ❌ 不适合使用查询处理的场景¶

    场景 原因 推荐替代
    简单直接问题 额外处理不必要 Native RAG
    延迟敏感场景 LLM 调用增加延迟 减少组件
    成本敏感场景 多次 LLM 调用成本高 使用小模型

    实现详解¶

    接口定义¶

    // 查询处理接口
type QueryProcessor interface {
    Process(ctx context.Context, query string) (*ProcessedQuery, error)
}

// 处理结果
type ProcessedQuery struct {
    Original   string   // 原始问题
    Rewritten  string   // 重写后的问题
    Entities   []string // 提取的实体
    SubQueries []string // 子问题（如果分解了）
    Intent     string   // 意图类型
    HyDEDoc    string   // HyDE 生成的假设文档
    StepBackQ  string   // StepBack 生成的问题
}
