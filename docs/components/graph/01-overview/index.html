    Knowledge Graph - 知识图谱¶

    是什么¶

    知识图谱是一种用图结构表示知识的技术。它由实体（Nodes）和关系（Edges）组成，能够捕捉文本中概念之间的关联信息。

    核心概念¶

    实体 (Node)¶

    graph LR A((Alice)) -->|KNOWS| B((Bob)) A -->|WORKS_AT| C((Google)) C -->|ACQUIRED| D((DeepMind)) style A fill:#e1f5fe style B fill:#e1f5fe style C fill:#e1f5fe style D fill:#e1f5fe
    类型 示例
    人物 (PERSON) Alice, Bob, 马斯克
    组织 (ORGANIZATION) Google, MIT, 苹果公司
    概念 (CONCEPT) AI, 机器学习, RAG
    地点 (LOCATION) 北京, 美国, 太平洋
    事件 (EVENT) 收购, 发布, 会议

    关系 (Edge)¶

    类型 示例
    WORKS_AT Alice -WORKS_AT→ Google
    KNOWS Alice -KNOWS→ Bob
    ACQUIRED 苹果 -ACQUIRED→ Beats
    DEVELOPS OpenAI -DEVELOPS→ GPT-4
    IS_A GPT-4 -IS_A→ 大语言模型

    GraphExtractor 图提取器¶

    核心功能¶

    从文本中提取实体和关系：

    import "github.com/DotNetAge/gorag/pkg/retrieval/graph"

extractor := graph.NewGraphExtractor(llmClient)
nodes, edges, err := extractor.Extract(ctx, chunk)

    提取结果示例¶

    输入：

    "Alice 在 Google 工作，她认识 Bob。Bob 是 MIT 的学生。"

    LLM 返回的 JSON：

    {
  "nodes": [
    {"id": "Alice", "type": "PERSON"},
    {"id": "Google", "type": "ORGANIZATION"},
    {"id": "Bob", "type": "PERSON"},
    {"id": "MIT", "type": "ORGANIZATION"}
  ],
  "edges": [
    {"source": "Alice", "target": "Google", "type": "WORKS_AT"},
    {"source": "Alice", "target": "Bob", "type": "KNOWS"},
    {"source": "Bob", "target": "MIT", "type": "STUDIES_AT"}
  ]
}

    核心类型¶

    type Node struct {
    ID         string         // 实体唯一标识
    Type       string         // 实体类型
    Properties map[string]any // 附加属性
}

type Edge struct {
    Source     string         // 源实体 ID
    Target     string         // 目标实体 ID
    Type       string         // 关系类型
    Properties map[string]any // 附加属性
}

    支持的图存储¶

    SQLite (嵌入式)¶

    推荐用于本地和轻量级应用：

    graphStore, _ := sqlite.NewGraphStore("./data/graph.db")

    BoltDB (嵌入式)¶

    纯 Go 实现，极高性能：

    graphStore, _ := bolt.NewGraphStore("./data/graph.db")

    Neo4j (工业级)¶

    支持大规模知识图谱：

    import "github.com/DotNetAge/gorag/pkg/indexing/store/neo4j"

graphStore, _ := neo4j.NewGraphStore(
    "bolt://localhost:7687",
    "neo4j",
    "password",
    "neo4j",
)

    使用 Cypher 模板¶

    Cypher 是图数据库的 SQL。如果说 SQL 是用来操作"表"的，那么 Cypher 就是用来操作"点"和"边"的。在 GoRAG 中，Cypher 模板的作用主要体现在以下三个维度：

    1. 实现"多跳推理"（Multi-hop Reasoning）¶

    向量检索（Vector Search）通常只能找到"长得像"的内容。但如果你问："张三入职公司的 CEO 是谁？"，向量检索可能找不到答案，因为文档里可能只写了"张三入职了 A 公司"和"B 是 A 公司的 CEO"。

    Cypher 模板的作用：

    MATCH (p:Person {name: $name})-[:WORKS_AT]->(c:Company)<-[:CEO_OF]-(ceo:Person)
RETURN ceo.name

    框架只需要把 $name 填入"张三"，就能在图中通过路径直接"跳"到答案。

    2. 标准化 GraphRAG 的"全局检索"¶

    微软的 GraphRAG 论文中提到了社区摘要（Community Summary）。它不是针对某个具体实体的检索，而是针对整个知识图谱的概括。

    Cypher 模板的作用：

    // 抓取第 2 层级的所有社区总结
MATCH (c:Community {level: 2})
RETURN c.summary

    这让 GoRAG 能够支持"这份文档集主要讲了哪几个核心观点？"这种全局性问题。

    3. 结构化过滤（Hybrid Search）¶

    有时候我们需要结合"语义"和"属性"。比如："查找 2024 年之后发布的、关于'深度学习'的论文。"

    Cypher 模板的作用：

    MATCH (d:Document)
WHERE d.publish_date > '2024-01-01' AND d.topic = 'AI'
RETURN d.content

    这比纯向量检索要精准得多，因为它排除了所有日期不符的干扰项。

    Cypher 模板的价值¶

    • 解耦查询逻辑：你不需要在 Go 代码里写死复杂的逻辑，只需要定义好 Cypher 字符串。
    • 让 LLM 生成查询：这是目前最高级的玩法（Text-to-Cypher）。用户输入自然语言，LLM 根据你提供的 Cypher 模板格式，生成一条查询语句，由 GoRAG 去执行，最后把结果返回。

    Cypher 模板就像是给 RAG 装上了一个"逻辑导航仪"： - 向量检索是负责"在大海里捞针" - Cypher 模板是负责"根据地图（图谱）找路"

    Go 代码中使用¶

    const ceoFinderTemplate = `
    MATCH (p:Entity {id: $id})-[:WORKS_AT]->(c:Entity)<-[:CEO_OF]-(ceo:Entity)
    RETURN ceo.id as name
`

retriever := graph.NewRetriever(
    vectorStore,
    graphStore,
    docStore,
    embedder,
    llm,
    graph.WithCustomStep(graph.NewCypherStep(graphStore, ceoFinderTemplate, logger)),
)

    GraphRAG 完整知识图谱查询¶

    传统的基于向量检索 (Vector RAG) 的系统在处理"特定事实"时表现优异，但在面对"全局性总结"（例如："这 100 份财报中所有公司在 AI 领域的共同投资趋势是什么？"）或"跨文档的多跳推理"时，往往会因为 Chunk 的碎片化而丢失全局上下文。

    GraphRAG（以微软 GraphRAG 和港大 LightRAG 为代表）通过引入知识图谱（实体与关系）完美弥补了这一缺陷。

    GraphRAG 核心模组¶

    图谱检索不仅仅是简单的查库，它通常分为局部实体检索和全局社区检索两种范式。

    1. 实体抽取与索引 (Graph Extraction)¶

    • 功能: 在文档入库时，利用 LLM 从文本 Chunk 中提取节点（实体 Entity，如人名、公司）和边（关系 Relationship，如"收购"、"竞争"），并存入 GraphStore。
    • 模块/Step: indexing.GraphExtractor / steps.NewGraphExtractStep()

    2. 局部搜索 (Local Graph Search)¶

    • 痛点: 用户询问"A公司和B公司的关系网络"，单纯的向量检索只能找到分别包含 A 和 B 的零散片段。
    • 机制:
    • 使用 LLM 或 NER（命名实体识别）从用户的 Query 中提取出核心实体（如 "A公司"）。
    • 在图数据库中定位该节点，并向外跳跃 (N-Hop) 检索其关联的邻居节点和关系描述。
    • 将召回的子图（Sub-graph）转化为文本描述作为上下文。

    3. 全局搜索 (Global Graph Search)¶

    • 痛点: 宏观问题（如"总结本书的核心主旨"）会导致向量检索召回大量毫不相干的 Top-K 结果。
    • 机制: 依赖图谱的社区检测算法（如 Leiden 算法），提前为图谱划分社区（Community）并由 LLM 生成"社区摘要"。查询时，系统会并发读取所有高层级的社区摘要进行 Map-Reduce 汇总，从而生成具备全局视野的答案。

    混合图谱管线¶

    在工业落地中，纯图谱检索是片面的。最强大的架构是 Hybrid GraphRAG（向量检索 + 图谱检索 双轨并行）。GoRAG 的 Pipeline 机制天然支持这种并发合并。

    flowchart TD Input[用户复杂 Query] --> Extractor[实体提取] Input --> Embed[文本 Embedding] subgraph VectorTrack[向量检索轨道] Embed --> VectorDB[(Vector Store)] VectorDB -->|Top-K Chunks| VectorRes[向量召回结果] end subgraph GraphTrack[图谱检索轨道] Extractor -->|提取 Entities| GraphDB[(Graph Store)] GraphDB -->|N-Hop 遍历| LocalGraph[局部关系文本] GraphDB -->|社区聚类| GlobalGraph[全局社区摘要] end VectorRes --> Fusion[RAG-Fusion 倒数排名融合] LocalGraph --> Fusion GlobalGraph -->|如果触发宏观查询| Fusion Fusion -->|去重打分后的综合上下文| Gen[LLM Generation] Gen --> Output[具备全局+细节视角的响应]

    管线编排模式¶

    模式 A: 纯局部图谱搜索 (Entity-Centric GraphRAG)¶

    适用于专门针对实体关系网络的查询应用（如公安图谱分析、企业股权穿透）：

    pipe := pipeline.NewPipeline()

pipe.AddSteps(
    steps.NewEntityExtractStep(llmClient),
    steps.NewGraphLocalSearchStep(graphStore, graph.LocalOptions{MaxHops: 2, TopK: 15}),
    steps.NewGenerationStep(llmClient),
)

    模式 B: 终极混合检索 (The Ultimate Hybrid GraphRAG)¶

    融合了稠密向量的"模糊语义召回"与图谱的"精准逻辑结构"：

    pipe := pipeline.NewPipeline()

// 并发执行双轨检索：
// 轨道 1: 传统的向量相似度检索 (捕捉语义细节)
vectorStep := steps.NewVectorSearchStep(vectorStore, 10)

// 轨道 2: 基于图谱的局部搜索 (捕捉实体关系逻辑)
graphLocalStep := steps.NewGraphLocalSearchStep(graphStore, graph.LocalOptions{MaxHops: 2})

// 组合为一个并行 Step，同时向底层的数据库发起请求，极大降低延迟
parallelSearchStep := steps.NewParallelStep(vectorStep, graphLocalStep)
pipe.AddSteps(
    parallelSearchStep,
    steps.NewRAGFusionStep(),
    steps.NewRerankStep(rerankerModel, 5),
    steps.NewGenerationStep(llmClient),
)
