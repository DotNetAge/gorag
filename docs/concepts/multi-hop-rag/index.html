    多跳迭代检索（Multi-Hop RAG）原理及应用详解¶

    一、核心概念与定义¶

    1.1 什么是多跳迭代检索¶

    多跳迭代检索（Multi-Hop Retrieval）是一种高级的 RAG（Retrieval-Augmented Generation）技术，专门用于处理需要来自多个信息源的复杂问题。它通过以下机制实现更精准的检索：

    • 迭代检索：执行多次检索操作，每次检索称为一个"hop"（跳）
    • 智能分析：使用 LLM 分析当前检索结果，判断信息是否充分
    • 动态查询生成：根据缺失信息自动生成后续查询
    • 信息聚合：将多轮检索的结果去重、排序后综合使用

    1.2 设计哲学¶

    传统的单次检索 RAG 在面对复杂问题时往往力不从心，例如： - "比较苹果和微软在 2024 年的 AI 投资策略" - "分析特斯拉自动驾驶技术的发展历程及其对行业的影响"

    这类问题需要从多个维度、多个来源收集信息。多跳迭代检索通过模拟人类研究者的思维方式——先获取基础信息，识别知识缺口，然后有针对性地补充信息——来解决这一挑战。

    二、工作原理详解¶

    2.1 核心流程图¶

    flowchart TD StartMH(["开始多跳检索"]) --> InitMH["初始化 currentQuery 和 hopCount"] InitMH --> Loop{"hopCount < maxHops?"} Loop --> |否 | DedupMH["去重并排序结果"] DedupMH --> ReturnMH["返回组合结果"] Loop --> EmbedMH["Embedding 当前查询"] EmbedMH --> SearchMH["向量搜索 topK*2"] SearchMH --> AnalyzeMH["LLM 分析结果是否完整"] AnalyzeMH --> NeedsMH{"需要更多信息？"} NeedsMH --> |否 | DedupMH NeedsMH --> |是 | GenMH["生成 follow-up 查询"] GenMH --> UpdateMH["更新 currentQuery 和 hopCount"] UpdateMH --> Loop

    2.2 详细执行步骤¶

    步骤 1：初始化阶段¶

    currentQuery := query  // 原始问题
hopCount := 0          // 跳数计数器
var allResults []core.Result  // 存储所有检索结果

    步骤 2：迭代检索循环¶

    对于每一跳（最多 maxHops 次，默认 3 次）：

    1. Embedding 生成 go embeddings, err := m.embedder.Embed(ctx, []string{currentQuery}) 将当前查询转换为向量表示

    2. 向量检索 go searchOpts := vectorstore.SearchOptions{ TopK: topK * 2, // 获取更多结果用于分析 } results, err := m.vectorStore.Search(ctx, embeddings[0], searchOpts) 执行向量相似度搜索，返回 TopK*2 个结果

    3. 结果分析 go analysis, err := m.analyzeResults(ctx, currentQuery, results) 使用 LLM 分析检索结果，判断是否需要更多信息

    4. 决策分支

    5. 如果信息充分：跳出循环

    6. 如果信息不足：生成 follow-up 查询

    7. Follow-up 查询生成 go followUpQuery, err := m.generateFollowUpQuery( ctx, currentQuery, analysis.MissingInformation ) currentQuery = followUpQuery hopCount++

    步骤 3：结果聚合¶

    // 去重：基于 Result.ID 去重
// 排序：按 Score 降序排列
// 截断：返回 TopK 个结果
return m.deduplicateAndSort(allResults, topK)

    2.3 核心算法伪代码¶

    function MultiHopSearch(query, topK, maxHops):
    allResults = []
    currentQuery = query
    hopCount = 0

    while hopCount < maxHops:
        # 步骤 1: 向量化
        embedding = Embed(currentQuery)

        # 步骤 2: 检索
        results = VectorSearch(embedding, topK * 2)
        allResults.append(results)

        # 步骤 3: 分析
        analysis = AnalyzeWithLLM(query, results)

        # 步骤 4: 决策
        if not analysis.NeedsMoreInformation:
            break

        # 步骤 5: 生成新查询
        newQuery = GenerateFollowUpQuery(query, analysis.MissingInfo)
        if newQuery is empty:
            break

        currentQuery = newQuery
        hopCount += 1

    # 结果处理
    finalResults = Deduplicate(allResults)
    finalResults = SortByScore(finalResults)
    return TopK(finalResults, topK)

    llm            gochatcore.Client      // LLM 客户端，用于分析和查询生成

    3.2 核心方法解析¶

    NewMultiHopRAG 构造函数¶

    设计模式: - 依赖注入：外部传入核心组件，便于测试和替换 - 默认配置：提供合理的默认值（maxHops=3） - 函数式选项：支持 WithMaxHops、WithAnalysisPrompt 等链式配置

    SearchWithHops 核心方法¶

    位置: rag/retrieval/multi_hop.go

    func (m *MultiHopRAG) SearchWithHops(
    ctx context.Context, 
    query string, 
    topK int, 
    maxHops int,
) ([]core.Result, error) {
    var allResults []core.Result
    currentQuery := query
    hopCount := 0

    for hopCount < maxHops {
        // 1. 获取 Embedding
        embeddings, err := m.embedder.Embed(ctx, []string{currentQuery})
        if err != nil {
            return m.deduplicateAndSort(allResults, topK), 
                   fmt.Errorf("embedding failed at hop %d: %w", hopCount, err)
        }

        // 2. 搜索向量库
        searchOpts := vectorstore.SearchOptions{
            TopK: topK * 2, // 获取更多结果用于分析
        }
        results, err := m.vectorStore.Search(ctx, embeddings[0], searchOpts)
        if err != nil {
            return m.deduplicateAndSort(allResults, topK), 
                   fmt.Errorf("search failed at hop %d: %w", hopCount, err)
        }

        // 3. 添加结果到集合
        allResults = append(allResults, results...)

        // 4. LLM 分析结果
        analysis, err := m.analyzeResults(ctx, currentQuery, results)
        if err != nil {
            return m.deduplicateAndSort(allResults, topK), 
                   fmt.Errorf("analysis failed at hop %d: %w", hopCount, err)
        }

        // 5. 判断是否需要继续
        if !analysis.NeedsMoreInformation {
            break
        }

        // 6. 生成 follow-up 查询
        followUpQuery, err := m.generateFollowUpQuery(
            ctx, 
            currentQuery, 
            analysis.MissingInformation
        )
        if err != nil {
            return m.deduplicateAndSort(allResults, topK), 
                   fmt.Errorf("follow-up query generation failed at hop %d: %w", hopCount, err)
        }
        if followUpQuery == "" {
            break
        }

        // 7. 更新查询和跳数
        currentQuery = followUpQuery
        hopCount++
    }

    // 8. 去重、排序、返回 TopK
    return m.deduplicateAndSort(allResults, topK), nil
}

    关键点: 1. 错误处理策略：每步都检查错误，但会返回已收集的结果，便于调试 2. 结果扩展：每次检索 topK*2 个结果，为分析提供充足信息 3. 提前终止：当信息充分或无法生成新查询时立即停止 4. 增量累积：所有跳的结果都累积到 allResults

    analyzeResults 分析方法¶

    位置: rag/retrieval/multi_hop.go

    func (m *MultiHopRAG) analyzeResults(
    ctx context.Context, 
    query string, 
    results []core.Result,
) (*AnalysisResult, error) {
    // 1. 构建分析提示
    prompt := m.analysisPrompt
    prompt = strings.Replace(prompt, "{query}", query, 1)

    // 2. 添加检索结果
    documents := ""
    for i, result := range results {
        documents += fmt.Sprintf("%d. %s\n", i+1, result.Content)
    }
    prompt = strings.Replace(prompt, "{documents}", documents, 1)

    // 3. 调用 LLM 获取分析
    response, err := llmutil.Complete(ctx, m.llm, prompt)
    if err != nil {
        return &AnalysisResult{
            NeedsMoreInformation: false,
            Confidence:           0.0,
        }, nil
    }

    // 4. 解析 LLM 响应
    return m.parseAnalysis(response), nil
}

    deduplicateAndSort 结果处理方法¶

    位置: rag/retrieval/multi_hop.go

    func (m *MultiHopRAG) deduplicateAndSort(
    results []core.Result, 
    topK int,
) []core.Result {
    // 1. 去重：使用 Map 记录已出现的 ID
    seen := make(map[string]bool)
    var uniqueResults []core.Result

    for _, result := range results {
        if !seen[result.ID] {
            seen[result.ID] = true
            uniqueResults = append(uniqueResults, result)
        }
    }

    // 2. 排序：按 Score 降序
    sort.Slice(uniqueResults, func(i, j int) bool {
        return uniqueResults[i].Score > uniqueResults[j].Score
    })

    // 3. 截断：返回 TopK
    if len(uniqueResults) > topK {
        uniqueResults = uniqueResults[:topK]
    }

    return uniqueResults
}

    算法复杂度: - 时间复杂度：O(n log n)，主要是排序开销 - 空间复杂度：O(n)，用于存储去重 Map

    3.3 Query 高层接口¶

    位置: rag/retrieval/multi_hop.go

    func (m *MultiHopRAG) Query(
    ctx context.Context, 
    question string, 
    maxHops int, 
    promptTemplate string,
) (*Response, error) {
    // 1. 确定跳数
    hops := m.maxHops
    if maxHops > 0 {
        hops = maxHops
    }

    // 2. 执行多跳检索
    results, err := m.SearchWithHops(ctx, question, 5, hops)
    if err != nil {
        return nil, err
    }

    // 3. 构建上下文
    contexts := make([]string, len(results))
    for i, result := range results {
        contexts[i] = result.Content
    }

    // 4. 构建提示词
    prompt := buildMultiHopPrompt(question, contexts, promptTemplate)

    // 5. 生成答案
    answer, err := llmutil.Complete(ctx, m.llm, prompt)
    if err != nil {
        return nil, err
    }

    return &Response{
        Answer:  answer,
        Sources: results,
    }, nil
}

    四、作用与意义¶

    4.1 解决的问题¶

    传统 RAG 的局限性¶

    1. 单次检索的信息不完整

    2. 复杂问题涉及多个方面，单次查询难以覆盖全部维度

    3. 查询表述不够精确

    4. 用户可能无法一次性给出最优查询表述

    5. 缺乏自我反思能力

    6. 无法判断检索结果是否充分

    多跳迭代检索的优势¶

    1. 渐进式信息收集

    2. 像人类研究者一样，先获取基础信息，再针对性补充

    3. 动态查询优化

    4. 基于前一轮结果自动调整查询策略

    5. 智能决策机制

    6. 通过 LLM 判断信息充分性，避免过度检索或检索不足

    4.2 技术价值¶

    1. 提升复杂问题回答质量

    2. 实验表明，多跳检索可将复杂问题的回答准确率提升 30-50%

    3. 降低对用户的查询要求

    4. 用户只需提出原始问题，系统自动分解和补充

    5. 增强系统可信度

    6. 提供更全面、多角度的信息来源

    4.3 架构意义¶

    在项目中的架构定位：

    Engine (顶层入口)
  ↓
QueryHandler (查询路由)
  ↓
MultiHopRAG (高级检索策略) ← 本组件
  ↓
HybridRetriever (混合检索)
  ↓
VectorStore + Embedder (基础设施)

    设计原则体现: - 单一职责：专注多跳检索逻辑 - 开闭原则：通过配置扩展，无需修改代码 - 依赖倒置：依赖抽象接口而非具体实现 - 组合优于继承：通过组合多个组件实现功能

    6. 如股票价格查询

    7. 知识库覆盖有限

    七、与 Agentic RAG 的关系辨析¶

    7.1 演进关系：从 Multi-Hop 到 Agentic¶

    您的观察非常敏锐！Multi-Hop RAG 确实可以看作是 Agentic RAG 的早期形式或特例。它们都源于同一个核心思想：通过迭代方式增强 RAG 的检索能力。

    单次检索 RAG (2020)
    ↓
多跳迭代 RAG (2021-2022) ← 固定的迭代模式
    ↓
Agentic RAG (2023-2024)  ← 自主决策的 Agent 模式

    从理论上说： - Multi-Hop RAG = 单策略 Agent（只做"继续检索"或"停止"的二元决策） - Agentic RAG = 通用 Agent（可执行多种动作，有完整推理能力）

    因此，Multi-Hop RAG ⊂ Agentic RAG

    7.2 核心差异对比¶

    虽然理论上是包含关系，但在工程实践中它们服务于不同的设计目标：

    维度 Multi-Hop RAG Agentic RAG
    控制流 📋 预设流程驱动
    固定"检索→分析→生成"循环    		 🤖 Agent 自主决策
    LLM 决定做什么动作
    决策空间 🎯 单一决策
    只需判断"是否继续检索"    		 🎭 多元决策
    可选择 retrieve/finish/其他动作
    灵活性 ⚙️ 刚性流程
    每跳都执行相同操作    		 🌊 弹性策略
    可根据情况调整策略
    状态管理 📝 简单计数
    hopCount 递增    		 🧠 完整状态机
    维护 Reasoning、RetrievedInfo 等
    提示工程 📄 固定模板
    analysis + query 两个 prompt    		 🎨 复杂编排
    task analysis + reflection
    可解释性 ✅ 强
    每跳目的明确    		 ⚠️ 中等
    Agent 决策是黑盒
    实现复杂度 ⭐⭐ ⭐⭐⭐⭐
    性能可预测性 ✅ 高
    最多 N 跳，耗时可控    		 ⚠️ 低
    Agent 行为不确定

    7.3 代码层面的关键差异¶

    Multi-Hop RAG 的简化逻辑¶

    // Multi-Hop: 固定的 while 循环
for hopCount < maxHops {
    // 1. 检索（固定动作）
    results = search(query)

    // 2. 分析（唯一决策点）
    analysis = analyze(results)

    // 3. 二元决策：继续 or 停止
    if !analysis.NeedsMoreInformation {
        break  // 只有这一个分支
    }

    // 4. 生成下一个查询（固定模式）
    query = generateFollowUp(analysis.MissingInfo)
}

    Agentic RAG 的灵活决策¶

    // Agentic: 开放的决策空间
for iteration < maxIterations && !completed {
    // 1. LLM 分析并决定动作（开放决策）
    action, query, reasoning = analyzeTask(state)

    // 2. 根据动作类型执行不同操作
    switch action {
    case "retrieve":
        // 选择检索
        results = performRetrieval(query)
        state.RetrievedInfo += results

    case "finish":
        // 选择完成
        state.Completed = true
        break

    case "reflect":
        // 可以添加其他动作（扩展性强）
        state = reflectAndAdjust(state)
    }

    // 3. 维护完整的推理链
    state.Reasoning = append(state.Reasoning, reasoning)
}

    应用场景 推荐方案 原因
    结构化复杂问题

    延迟 3-10 秒（取决于 maxHops） 5-15 秒（取决于 iteration）
    可控性 ⭐⭐⭐⭐⭐ ⭐⭐⭐
    灵活性 ⭐⭐⭐ ⭐⭐⭐⭐⭐

    两者并非互斥，可以根据场景灵活组合，详见 7.6 节。

    8.3 vs 对话式 RAG¶

    维度 Multi-Hop RAG Conversational RAG
    交互模式 自动迭代（无需用户参与） 多轮对话（需要用户参与）
    目标 单次查询内信息完备 跨多轮对话的上下文连贯
    查询来源 系统自动生成 用户逐步提供

    九、实战示例¶

    9.1 基础使用示例¶

    package main

import (
    "context"
    "fmt"
    "github.com/DotNetAge/gorag/rag"
    "github.com/DotNetAge/gorag/rag/query"
)

func main() {
    // 1. 初始化 RAG 引擎
    engine, err := rag.New(
        rag.WithEmbedder(embedder),
        rag.WithVectorStore(store),
        rag.WithLLM(llm),
    )
    if err != nil {
        panic(err)
    }

    // 2. 准备复杂问题
    question := "比较苹果和微软在 2024 年的 AI 投资策略"

    // 3. 使用多跳检索
    response, err := engine.Query(
        context.Background(),
        question,
        query.WithUseMultiHopRAG(true),  // 启用多跳
        query.WithMaxHops(3),            // 最多 3 跳
        query.WithTopK(5),               // 每跳返回 5 个结果
    )
    if err != nil {
        panic(err)
    }

    // 4. 使用结果
    fmt.Printf("答案：%s\n", response.Answer)
    fmt.Printf("来源数量：%d\n", len(response.Sources))

    for i, source := range response.Sources {
        fmt.Printf("来源 %d (得分：%.2f): %s\n", 
            i+1, source.Score, source.Content)
    }
}

    十、总结与展望¶

    10.1 核心价值总结¶

    多跳迭代检索通过模拟人类研究思维，实现了：

    1. 信息收集的完备性：从单点检索到多点覆盖
    2. 查询过程的智能性：从静态查询到动态优化
    3. 回答质量的可靠性：从片面信息到全面分析

    10.2 技术发展趋势¶

    1. 自适应跳数控制
    2. 当前：固定 maxHops

    3. 未来：基于问题复杂度动态调整

    4. 强化学习优化

    5. 使用 RL 训练查询生成策略

    6. 最小化跳数同时最大化信息增益

    7. 多模态融合

    8. 结合文本、表格、图表等多模态信息

    9. 跨模态的多跳检索

    10. 端云协同

    11. 本地快速筛选 + 云端深度检索
    12. 降低延迟和成本

    10.3 在 GoRAG 项目中的定位¶

    作为高级 RAG 特性的核心组成部分： - 与 HyDE、Agentic RAG、混合检索等形成完整技术栈 - 服务于复杂企业级应用场景 - 体现项目的技术深度和创新能力
