    RAG架构理论¶

    一、RAG 的基本原理与运行机制¶

    1.1 RAG 的核心概念¶

    检索增强生成（Retrieval-Augmented Generation，简称 RAG）是一种将参数化知识（存储在模型权重中）与非参数化知识（存储在外部文档库中）相结合的框架，旨在解决大型语言模型（LLM）的幻觉问题、知识更新滞后和领域特定信息缺失等核心局限性。

    数学形式化定义¶

    根据 Lewis 等人 (2020) 的开创性工作和 Sharma (2024) 的综合调研，RAG 的生成过程可以形式化为：

    P(y|x) = ∑_{d∈C} P(y|x,d) · P(d|x)

    其中： - x 是输入（如问题或提示） - d 是从语料库 C 中检索的文档 - y 是生成的响应

    在实际应用中，求和近似为 Top-K 个文档：

    P(y|x) ≈ ∑_{i=1}^{k} P(y|x,d_i) · P(d_i|x)

    这里包含两个关键概率： 1. P(d_i|x): 文档 d_i 相对于输入 x 的相关性得分（由检索器或重排序器计算） 2. P(y|x,d_i): 基于输入 x 和文档 d_i 生成输出 y 的条件概率（由语言模型建模）

    1.2 RAG 系统的三大核心组件¶

    根据 ACM TOIS 2024 年的综合调研，一个标准的 RAG 系统包含以下三个基本模块：

    1.2.1 查询编码器（Query Encoder）¶

    作用: 将输入 x 编码为查询表示 q，用于检索相关文档

    实现方式: - 神经编码器: 如 DPR (Dense Passage Retriever) 使用双编码器架构 - 规则模板: 基于关键词匹配的稀疏检索（如 BM25） - 混合方法: 结合稠密和稀疏表示

    在 GoRAG 中的实现:

    // embedding.Provider 接口负责将文本转换为向量表示
type Provider interface {
    Embed(ctx context.Context, texts []string) ([][]float32, error)
}

    1.2.2 检索器（Retriever）¶

    作用: 给定查询 q，从语料库 C 中检索并按相关性排序文档列表 d₁, d₂, ..., dₖ

    检索策略分类: 1. 稀疏检索 (Sparse Retrieval): - 基于词频统计（BM25, Robertson et al., 2009） - 优势：精确匹配、可解释性强 - 劣势：语义理解能力弱

    1. 稠密检索 (Dense Retrieval):
    2. 基于向量相似度（DPR, Karpukhin et al., 2020）
    3. 优势：语义匹配、泛化能力强

    4. 劣势：需要大量训练数据

    5. 混合检索 (Hybrid Retrieval):

    6. 结合稀疏和稠密检索的优势
    7. 使用 Reciprocal Rank Fusion (RRF) 或其他融合策略

    在 GoRAG 中的实现:

    // HybridRetriever 同时执行向量搜索和关键词搜索
func (r *HybridRetriever) Search(ctx context.Context, query string, topK int) ([]core.Result, error) {
    // 向量搜索
    vectorResults, _ := r.vectorStore.Search(ctx, embedding, SearchOptions{TopK: topK * 2})

    // 关键词搜索
    keywordResults, _ := r.keywordStore.Search(ctx, query, topK * 2)

    // 融合结果
    return r.combineResults(vectorResults, keywordResults, topK)
}

    1.2.3 生成器（Generator）¶

    作用: 基于输入 x 和检索到的文档 {d₁, ..., dₖ} 生成最终输出 y

    典型模型: - T5 (Raffel et al., 2020) - BART (Lewis et al., 2020) - GPT 系列 (Brown et al., 2020)

    在 GoRAG 中的实现:

    // LLM 客户端负责生成最终答案
answer, err := llmutil.Complete(ctx, q.llm, prompt)
// prompt 包含了检索到的上下文信息

    1.3 RAG 的工作流程¶

    基于 Figure 1 (Sharma, 2024) 的 RAG 工作流：

    用户查询 → 查询扩展/重写 → 检索器 → 文档重排序 → Top-K 文档 → 生成器 → 后处理 → 最终响应

    详细步骤:

    1. 查询处理阶段
    2. 接收用户查询 x
    3. 可选：使用 HyDE 或 RAG-Fusion 进行查询增强

    4. 生成增强后的查询表示 q'

    5. 检索阶段

    6. 使用 q' 在外部知识库 C 中检索

    7. 获取候选文档集合 {d₁, d₂, ..., dₙ} (n > k)

    8. 重排序阶段 (可选但推荐)

    9. 使用 LLM 或交叉编码器对候选文档重新评分

    10. 选择 Top-K 个最相关的文档

    11. 上下文优化阶段 (可选)

    12. 使用 ContextCompressor 压缩和优化上下文

    13. 去除冗余、截断过长内容、摘要提炼

    14. 生成阶段

    15. 构建提示：Prompt = f(x, d₁, ..., dₖ)
    16. 调用 LLM 生成响应 y

    17. 可选：后处理（事实核查、格式化等）

    18. 返回响应

    19. 返回答案 y 及来源引用

    1.4 RAG 的核心优势¶

    根据文献总结，RAG 相比纯 LLM 方案具有以下优势：

    维度 纯 LLM RAG 增强
    知识更新 静态（训练数据截止） 动态（实时更新知识库）
    幻觉抑制 容易产生虚假信息 基于检索证据，可验证
    领域适应性 需要微调 只需更新文档库
    可解释性 黑盒决策 可追溯信息来源
    长尾知识 记忆有限 理论上无限扩展
    成本效益 大模型训练昂贵 小模型 + 检索即可达到好效果

    二、GoRAG 各模块的作用与实现¶

    2.1 向量化模块（Embedding Provider）¶

    作用与意义¶

    向量化模块负责将离散的文本文档转换为连续的向量表示，使得计算机可以通过计算向量相似度来衡量文本之间的语义关系。这是稠密检索的基础设施。

    理论基础¶

    根据 Mikolov 等人 (2013) 的词向量理论和 Devlin (2018) 的 BERT 模型，文本向量化基于以下假设： - 分布假说: 语义相似的文本在向量空间中距离相近 - 语义相似性度量: 使用余弦相似度 cos(θ) = (v₁·v₂) / (||v₁||·||v₂||)

    在 GoRAG 中的实现¶

    接口定义:

    // embedding.Provider 定义了向量化提供者的接口
type Provider interface {
    // Embed 将文本列表转换为向量
    Embed(ctx context.Context, texts []string) ([][]float32, error)
}

    支持的提供商: 1. OpenAI Embedding: text-embedding-ada-002 等模型 2. Cohere: embed-multilingual-v2.0 3. Voyage AI: voyage-2 4. Ollama (本地): nomic-embed-text, mxbai-embed-large

    批处理优化:

    // BatchEmbed 批量处理多个文档，提高效率
func (p *Provider) BatchEmbed(ctx context.Context, docs []Document, batchSize int) {
    for i := 0; i < len(docs); i += batchSize {
        end := min(i + batchSize, len(docs))
        batch := docs[i:end]

        texts := extractTexts(batch)
        embeddings, _ := p.Embed(ctx, texts)

        // 将向量附加到文档
        for j, emb := range embeddings {
            batch[j].Embedding = emb
        }
    }
}

    1. 向量维度选择:

    2.2 上下文压缩模块（Context Compressor）¶

    作用与意义¶

    核心问题: 检索到的 Top-K 文档可能存在： - 信息冗余（多个文档表达相同内容） - 噪声干扰（包含不相关信息片段） - 超出 LLM 上下文窗口限制

    解决方案: ContextCompressor 通过过滤、去重、重排序、截断、摘要等策略，将原始检索结果优化为精炼、高信息密度的上下文，提升 LLM 生成质量并降低成本。

    理论支撑¶

    根据 Liu 等人 (2023) 的"Lost in the Middle"研究和 Jiang 等人 (2023) 的上下文压缩工作： - 位置偏差: LLM 对上下文中间部分的信息关注度较低 - 信息密度效应: 高密度、低冗余的上下文显著提升回答准确性 - 窗口限制: 需要在有限 token 数内最大化信息量

    3.1 多跳检索（Multi-Hop Retrieval）¶

    作用与意义¶

    解决的问题: 复杂问题需要从多个信息源拼凑完整答案，单次检索无法覆盖全部维度。

    典型场景: - "比较苹果和微软在 2024 年的 AI 投资策略" - "分析特斯拉自动驾驶技术的发展历程及其对行业的影响"

    这类问题涉及多个实体、多个时间点或多个维度，需要迭代式地收集和整合信息。

    理论基础¶

    根据 Min 等人 (2019) 的多跳推理工作和 Press 等人 (2022) 的迭代检索研究： - 分解 - 解决范式: 将复杂问题分解为多个子问题 sequentially 解决 - 信息增益最大化: 每跳都针对当前知识缺口进行检索 - 终止条件: 当信息充分或达到最大跳数时停止

    在 GoRAG 中的实现¶

    核心流程 (详见 multi_hop.go):

    func (m *MultiHopRAG) SearchWithHops(ctx context.Context, query string, topK int, maxHops int) ([]core.Result, error) {
    var allResults []core.Result
    currentQuery := query
    hopCount := 0

    for hopCount < maxHops {
        // 1. 向量化当前查询
        embeddings, _ := m.embedder.Embed(ctx, []string{currentQuery})

        // 2. 检索文档
        results, _ := m.vectorStore.Search(ctx, embeddings[0], SearchOptions{TopK: topK * 2})
        allResults = append(allResults, results...)

        // 3. LLM 分析结果是否充分
        analysis, _ := m.analyzeResults(ctx, currentQuery, results)

        // 4. 决策：继续 or 停止
        if !analysis.NeedsMoreInformation {
            break
        }

        // 5. 生成 follow-up 查询
        followUpQuery, _ := m.generateFollowUpQuery(ctx, currentQuery, analysis.MissingInformation)
        currentQuery = followUpQuery
        hopCount++
    }

    // 6. 去重、排序、返回 TopK
    return m.deduplicateAndSort(allResults, topK), nil
}

    3.2 Agentic 检索（Agentic RAG）¶

    作用与意义¶

    核心理念: 将检索过程嵌入到自主 Agent 的推理循环中，让 LLM 自主决定： 1. 是否需要检索 2. 检索什么信息 3. 检索结果的质量评估 4. 是否需要追加检索 5. 何时停止并生成答案

    与 Multi-Hop 的区别: - Multi-Hop: 固定"检索→分析→生成查询"循环，单一决策（继续/停止） - Agentic: 开放决策空间（retrieve/finish/reflect/其他动作），完整状态机

    理论基础¶

    根据 ReAct (Yao et al., 2022) 和 Reflexion (Shinn et al., 2023) 的 Agent 研究工作： - 推理 - 行动交替: Reasoning 和 Acting 在循环中相互促进 - 自我反思机制: Agent 可以评估自己的决策并进行调整 - 任务分解: 复杂任务自动拆解为可执行的子步骤

    在 GoRAG 中的实现¶

    Agent 状态机:

    type AgentState struct {
    Task           string        // 任务目标
    RetrievedInfo  []core.Result // 已获取信息
    Reasoning      []string      // 推理历史链
    IterationCount int           // 迭代次数
    Completed      bool          // 完成标志
    FinalAnswer    string        // 最终答案
}

    主循环逻辑:

    func (a *AgenticRAG) Search(ctx context.Context, task string, topK int) ([]core.Result, error) {
    state := &AgentState{
        Task: task,
        RetrievedInfo: []core.Result{},
        Reasoning: []string{},
        IterationCount: 0,
        Completed: false,
    }

    for state.IterationCount < a.maxIterations && !state.Completed {
        // Step 1: LLM 分析并决定行动
        action, query, reasoning := a.analyzeTask(ctx, state)
        state.Reasoning = append(state.Reasoning, reasoning)

        switch action {
        case "retrieve":
            // 执行检索
            results, _ := a.performRetrieval(ctx, query, topK)
            state.RetrievedInfo = append(state.RetrievedInfo, results...)

        case "finish":
            // 完成任务
            state.Completed = true

        case "reflect":
            // 反思并调整策略（可扩展）
            state = a.reflectAndAdjust(ctx, state)
        }

        state.IterationCount++
    }

    return a.deduplicateAndSort(state.RetrievedInfo, topK), nil
}

    灵活性 固定流程 开放策略
    可控性 ⭐⭐⭐⭐⭐ ⭐⭐⭐

    3.3 结果重排（Reranking）¶

    作用与意义¶

    核心问题: 向量检索和关键词检索返回的结果是基于查询 - 文档配对得分，而非查询 - 文档相对排序。这可能导致： - 高相似度但不直接相关的文档排在前面 - 低相似度但精准回答查询的文档被忽略 - 缺乏全局视角的相关性评估

    解决方案: Reranker 使用 LLM 的全局理解能力，对所有候选文档进行联合评估和重新排序，确保最相关的文档获得最高优先级。

    理论基础¶

    根据 Nogueira 和 Cho (2019) 的段落重排序研究，以及 Li 等人 (2022) 的 LLM 重排序工作： - Pointwise 方法: 独立评估每个文档的相关性分数 - Listwise 方法: 考虑文档间的相对关系进行整体排序 - 交叉编码器优势: 相比双编码器，交叉编码器能捕捉查询 - 文档间的细粒度交互

    在 GoRAG 中的实现¶

    核心结构:

    type Reranker struct {
    llm     gochatcore.Client
    topK    int               // 返回前 K 个结果
    prompt  string            // 重排序提示模板
}

    重排序流程:

    func (r *Reranker) Rerank(ctx context.Context, query string, results []core.Result) ([]core.Result, error) {
    if len(results) <= r.topK {
        return results, nil // 无需重排序
    }

    // Step 1: 构建重排序 Prompt
    prompt := r.buildRerankPrompt(query, results)
    // Prompt 示例："请为以下文档与查询的相关性打分 (0-1)..."

    // Step 2: LLM 给出所有文档的相关性分数
    scores, err := r.getRelevanceScores(ctx, prompt, len(results))

    // Step 3: 将分数赋给结果
    for i, score := range scores {
        results[i].Score = score
    }

    // Step 4: 按新分数排序
    sort.Slice(results, func(i, j int) bool {
        return results[i].Score > results[j].Score
    })

    // Step 5: 返回 TopK
    return results[:r.topK], nil
}

    多策略解析器 (鲁棒性设计):

    func (r *Reranker) parseScores(response string, expectedCount int) ([]float32, error) {
    // 策略 1: 逗号分隔 "0.9, 0.7, 0.3, 0.1"
    if parsed, ok := r.parseCommaSeparated(response, expectedCount); ok {
        return parsed, nil
    }

    // 策略 2: 编号列表 "1. 0.9\n2. 0.7\n3. 0.3\n4. 0.1"
    if parsed, ok := r.parseNumberedList(response, expectedCount); ok {
        return parsed, nil
    }

    // 策略 3: 提取所有数字
    if parsed, ok := r.parseAllNumbers(response, expectedCount); ok {
        return parsed, nil
    }

    return nil, fmt.Errorf("无法解析分数")
}

    根据项目实测数据： - 准确率提升: 20-30%（相比纯向量检索） - 延迟增加: 2-5 秒（LLM 调用开销） - 最佳实践: - 先用混合检索获取 20-50 个候选 - 再用 Reranker 精选 Top-5 - 性价比最高

    3.4 HyDE 虚拟文档嵌入技术（Hypothetical Document Embeddings）¶

    作用与意义¶

    核心洞察: Gao 等人 (2022) 在 HyDE 论文中发现： - 查询 - 文档语义鸿沟: 短查询难以表达完整语义意图 - 领域适配难题: 通用 embedding 模型在特定领域表现不佳 - 零样本场景: 无标注数据时检索质量大幅下降

    HyDE 的创新: 使用 LLM 生成假设性文档（即对查询的理想答案），用这个假设文档的向量来检索真实文档。即使假设文档包含事实错误（幻觉），其语义表示仍能有效匹配相关真实文档。

    数学原理¶

    传统方法:

    q → Embed(q) → Search(Embed(d))

    HyDE 方法:

    q → LLM(q) → h → Embed(h) → Search(Embed(d))

    其中 h 是假设文档（hypothetical document），关键观察： - Embed(h) ≈ Embed(d_relevant) 即使 h ≠ d_relevant - 这是因为 LLM 学会了相关性模式，尽管可能产生幻觉

    3.5 RAG-Fusion 多查询融合技术¶

    作用与意义¶

    核心思想: Raudaschl (2023) 提出的 RAG-Fusion 基于以下观察： - 单一查询局限性: 用户的一个问题可能有多种解读方式 - 视角多样性: 从多个角度检索可以覆盖更全面的信息 - 共识原则: 在多个查询下都排名靠前的文档更可能是高质量的

    技术路线: 1. 生成多个查询变体（多角度） 2. 对每个查询执行检索 3. 使用 Reciprocal Rank Fusion (RRF) 融合结果 4. 按融合分数重新排序

    理论基础¶

    Reciprocal Rank Fusion (RRF) 公式:

    score(d) = ∑_{q∈Queries} 1 / (rank(d, q) + k)

    其中： - rank(d, q) 是文档 d 在查询 q 下的排名 - k 是平滑常数（通常取 60，GoRAG 中用 fusionAlpha 默认为 0.1）

    RRF 的优势: - 归一化: 不同查询的排名可以直接比较 - 鲁棒性: 不受绝对分数影响，只关注相对顺序 - 理论保证: Cormack 等人 (2009) 证明了 RRF 在元分析中的最优性

    在 GoRAG 中的实现¶

    核心结构:

    type RAGFusion struct {
    llm          gochatcore.Client
    embedder     embedding.Provider
    vectorStore  vectorstore.Store
    keywordStore KeywordStore      // 可选：关键词搜索
    numQueries   int               // 生成查询数量（默认 4）
    fusionAlpha  float32           // RRF 平滑参数（默认 0.1）
}

    主流程:

    func (rf *RAGFusion) Search(ctx context.Context, originalQuery string, topK int) ([]core.Result, error) {
    // Step 1: 生成查询变体
    queries, err := rf.generateQueryVariations(ctx, originalQuery)
    queries = append(queries, originalQuery) // 保留原查询
    queries = rf.deduplicateQueries(queries)

    // Step 2: 对每个查询执行检索
    allResults := make(map[string]core.Result)
    ranks := make(map[string][]int)

    for _, query := range queries {
        // 获取向量
        embeddings, _ := rf.embedder.Embed(ctx, []string{query})

        // 向量搜索
        vectorResults, _ := rf.vectorStore.Search(ctx, embeddings[0], SearchOptions{TopK: topK * 2})

        // （可选）关键词搜索
        if rf.keywordStore != nil {
            keywordResults, _ := rf.keywordStore.Search(ctx, query, topK * 2)
            vectorResults = append(vectorResults, keywordResults...)
        }

        // 记录结果及其排名
        for rank, result := range vectorResults {
            allResults[result.ID] = result
            ranks[result.ID] = append(ranks[result.ID], rank + 1)
        }
    }

    // Step 3: RRF 融合
    fusedResults := rf.fuseResults(allResults, ranks)

    // Step 4: 返回 TopK
    return fusedResults[:topK], nil
}

    RRF 融合算法:

    func (rf *RAGFusion) fuseResults(results map[string]core.Result, 
                                   ranks map[string][]int) []core.Result {
    type scoredResult struct {
        core.Result
        score float32
    }

    var scoredResults []scoredResult

    for id, result := range results {
        // 计算 RRF 分数
        score := float32(0)
        for _, rank := range ranks[id] {
            score += 1.0 / (float32(rank) + rf.fusionAlpha)
        }

        scoredResults = append(scoredResults, scoredResult{
            Result: result,
            score:  score,
        })
    }

    // 按 RRF 分数排序
    sort.Slice(scoredResults, func(i, j int) bool {
        return scoredResults[i].score > scoredResults[j].score
    })

    // 转换回 core.Result
    fused := make([]core.Result, len(scoredResults))
    for i, sr := range scoredResults {
        fused[i] = sr.Result
    }

    return fused
}

    Agentic RAG
  └─ 包含 → 基础检索（Hybrid 或 Dense）
       └─ 包含 → 向量化（Embedding）

Multi-Hop RAG
  └─ 包含 → 基础检索（Dense）
       └─ 包含 → 向量化（Embedding）

RAG-Fusion
  └─ 包含 → 多次基础检索
       └─ 包含 → 向量化（Embedding）
       └─ 可选包含 → 关键词搜索

    4.2.3 依赖关系¶

    单向依赖（从外层到内层）:

    应用层（User Query）
  ↓
策略层（Multi-Hop / Agentic / RAG-Fusion）
  ↓
增强层（HyDE - 可选）
  ↓
检索层（Hybrid Retriever）
  ↓  
存储层（Vector Store + Keyword Store）
  ↓
基础设施层（Embedding Provider）

    具体依赖链:

    1. 标准检索路径: QueryHandler → HybridRetriever → VectorStore → Embedding Provider ↘ KeywordStore

    2. HyDE 增强路径: QueryHandler → HyDE → LLM (生成假设文档) ↓ HybridRetriever (使用增强后的查询)

    3. 后处理路径: HybridRetriever → Reranker → LLM (重排序评分) ↓ ContextCompressor → LLM (摘要生成)

    4.2.4 并行关系¶

    可并行执行的模块:

    // Hybrid Retriever 内部的并行
func (r *HybridRetriever) Search(ctx context.Context, query string, topK int) {
    // 并行执行向量搜索和关键词搜索
    done := make(chan struct{})

    go func() {
        vectorResults, _ = r.vectorStore.Search(ctx, embedding, opts)
        done <- struct{}{}
    }()

    go func() {
        keywordResults, _ = r.keywordStore.Search(ctx, query, opts)
        done <- struct{}{}
    }()

    <-done
    <-done

    // 融合结果
    return r.combineResults(vectorResults, keywordResults, topK)
}

    不可并行的模块: - HyDE 必须在检索之前（依赖查询增强） - Reranker 必须在检索之后（依赖检索结果） - ContextCompressor 必须在 Reranker 之后（依赖重排序结果）

    4.3 推荐的模块组合策略¶

    4.3.1 基础配置（适合 80% 场景）¶

    # config.yaml
retrieval:
  strategy: "hybrid"  # 混合检索
  topK: 5

reranking:
  enabled: true
  topK: 5

compression:
  enabled: true
  maxContextSize: 4000

    特点: - ✅ 平衡性能和成本 - ✅ 适合大多数问答场景 - ⚠️ 不适合高度复杂问题

    4.3.2 高精度配置（关键业务场景）¶

    retrieval:
  strategy: "rag-fusion"  # RAG-Fusion
  topK: 10
  ragFusionQueries: 4

reranking:
  enabled: true
  topK: 5

hyde:
  enabled: true  # HyDE 增强

compression:
  enabled: true

    特点: - ✅ 最高检索质量 - ✅ 适合复杂、关键问题 - ⚠️ 延迟较高（5-10 秒） - ⚠️ 成本较高（多次 LLM 调用）

    4.3.3 复杂推理配置¶

    advanced_rag:
  strategy: "multi-hop"  # 或 "agentic"
  maxHops: 3

retrieval:
  strategy: "hybrid"
  topK: 5

reranking:
  enabled: true

    特点: - ✅ 适合多跳推理问题 - ✅ 自动补充缺失信息 - ⚠️ 延迟最高（10-20 秒）

    4.3.4 低延迟配置（实时场景）¶

    retrieval:
  strategy: "dense"  # 仅向量检索
  topK: 5

reranking:
  enabled: false

compression:
  enabled: false

    特点: - ✅ 最快响应（<1 秒） - ✅ 适合对话系统 - ⚠️ 精度相对较低

    5.2 各层的职责与接口¶

    第 1 层：基础设施层 (Infrastructure Layer)¶

    职责: 提供基础的 AI 能力和数据处理能力

    核心组件: - Embedding Provider: 文本向量化 - LLM Provider: 文本生成 - Parser: 文档解析

    接口定义:

    // Embedding Provider
type Provider interface {
    Embed(ctx context.Context, texts []string) ([][]float32, error)
}

// LLM Client
type Client interface {
    Complete(ctx context.Context, prompt string) (string, error)
}

// Parser
type Parser interface {
    Parse(reader io.Reader) ([]core.Chunk, error)
}

    特点: - 🔧 可插拔设计（支持多个提供商） - 📦 标准化接口 - 🌐 外部服务依赖（OpenAI、Cohere 等）

    第 2 层：存储层 (Storage Layer)¶

    职责: 持久化和检索数据

    核心组件: - Vector Store: 向量相似度搜索 - Keyword Store: 关键词匹配搜索

    接口定义:

    type Store interface {
    AddDocuments(ctx context.Context, docs []Document) error
    Search(ctx context.Context, query []float32, opts SearchOptions) ([]Result, error)
    Delete(ctx context.Context, ids []string) error
}

    实现: - Memory Store（内存） - Milvus（专用向量数据库） - Qdrant - Pinecone - Weaviate

    特点: - 💾 数据持久化 - 🔍 高效检索（索引优化） - 📈 可扩展性（支持海量数据）

    第 3 层：检索层 (Retrieval Layer)¶

    职责: 实现核心检索逻辑和优化

    核心组件: - Hybrid Retriever: 混合检索（向量 + 关键词） - Reranker: 结果重排序 - ContextCompressor: 上下文压缩

    接口定义:

    type Retriever interface {
    Search(ctx context.Context, query string, topK int) ([]Result, error)
}

type Reranker interface {
    Rerank(ctx context.Context, query string, results []Result) ([]Result, error)
}

    特点: - 🎯 检索策略实现 - ⚙️ 性能优化（批处理、缓存） - 🔄 承上启下（对接策略层和存储层）

    第 4 层：策略层 (Strategy Layer)¶

    职责: 实现高级检索策略和智能决策

    核心组件: - Multi-Hop RAG: 多跳迭代检索 - Agentic RAG: 代理式自主检索 - RAG-Fusion: 多查询融合 - HyDE: 查询增强

    接口定义:

    type AdvancedRetriever interface {
    Query(ctx context.Context, question string, options Options) (*Response, error)
}

    特点: - 🧠 智能化（LLM 驱动决策） - 🎨 灵活性（可配置策略） - 📊 场景适配（不同问题类型）

    第 5 层：编排层 (Orchestration Layer)¶

    职责: 协调各个组件，实现完整的 RAG 流程

    核心组件: - RAG Engine: 顶层入口，资源管理 - QueryHandler: 查询处理流水线 - Indexer: 文档索引管理

    接口定义:

    type Engine interface {
    Index(ctx context.Context, paths ...string) error
    Query(ctx context.Context, question string, opts QueryOptions) (*Response, error)
}

    特点: - 🎼 流程编排 - 🔗 组件解耦 - 🛠️ 配置管理

    第 6 层：应用层 (Application Layer)¶

    职责: 面向最终用户的接口

    形式: - CLI 工具 (gorag query) - REST API - SDK 集成

    特点: - 👤 用户交互 - 📱 多端支持 - 🔌 插件扩展

    5.3 层间交互原则¶

    5.3.1 单向依赖原则¶

    上层 → 下层（依赖方向）
下层 ↛ 上层（反向禁止）

    好处: - ✅ 清晰的职责边界 - ✅ 易于测试和替换 - ✅ 降低耦合度

    5.3.2 接口抽象原则¶

    每层通过接口与相邻层通信，而非具体实现：

    // 第 4 层不依赖具体的 VectorStore 实现
type Retriever struct {
    store vectorstore.Store  // 依赖接口，而非具体实现
}

// 可以是 Memory、Milvus、Qdrant 等任意实现
store := milvus.NewStore(config)
// 或
store := memory.NewStore()

    5.3.3 关注点分离原则¶

    每层只关心自己的职责：

    第 4 层（策略层）: "是否需要多跳检索？"
  ↓
第 3 层（检索层）: "如何高效检索？"
  ↓
第 2 层（存储层）: "如何存储和索引？"
  ↓
第 1 层（基础设施）: "如何向量化文本？"

    六、总结与建议¶

    6.1 架构设计的核心优势¶

    GoRAG 的架构设计体现了以下软件工程最佳实践：

    1. 分层清晰: 每层职责明确，便于理解和维护
    2. 高内聚低耦合: 模块间接口简洁，依赖单向
    3. 可扩展性: 新增组件无需修改现有代码（开闭原则）
    4. 可测试性: 每层可独立测试（Mock 依赖）
    5. 生产就绪: 包含缓存、日志、指标、链路追踪

    6.2 选型建议矩阵¶
