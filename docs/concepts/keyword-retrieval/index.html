    关键词检索 (Keyword Retrieval / BM25)¶

    什么是关键词检索？¶

    关键词检索是信息检索领域最经典的技术之一，它通过匹配查询中的关键词与文档中的词来评估相关性。其中 BM25（Best Matching 25）是目前最广泛使用的关键词检索算法，被应用于 Elasticsearch、Solr 等主流搜索引擎。

    一句话理解¶

    关键词检索 = 找到包含查询关键词的文档

    关键词检索在 RAG 中的位置¶

    flowchart LR A[文档] --> B[分词] B --> C[倒排索引]:::orange D[用户查询] --> E[分词] E --> F[倒排索引查询]:::orange F --> G[BM25 打分] G --> H[排序返回] classDef orange fill:#fff3e0

    BM25 算法详解¶

    1. 算法起源¶

    BM25 于 1994 年由 Stephen Robertson 和 Karen Spärck Jones 提出 [1]，是 Okapi 信息检索系统的核心算法。"BM25"中的"25"表示这是该系列算法的第25个版本。

    2. 算法公式¶

    BM25 的核心公式如下 [1][2]：

    Score(Q, D) = Σ IDF(qi) · f(qi, D) · (k1 + 1) / (f(qi, D) + k1 · (1 - b + b · |D| / avgdl))

    其中： - Q = 查询关键词集合 {q1, q2, ..., qn} - D = 文档 - f(qi, D) = 关键词 qi 在文档 D 中的词频（TF） - |D| = 文档 D 的长度 - avgdl = 平均文档长度 - k1 = 词频饱和参数（通常取 1.2-2.0） - b = 文档长度归一化参数（通常取 0.75） - IDF(qi) = 逆文档频率

    3. 公式各部分详解¶

    3.1 IDF（逆文档频率）¶

    IDF 衡量一个词在整个文档集合中的重要性 [2]：

    IDF(qi) = log( (N - n(qi) + 0.5) / (n(qi) + 0.5) )

    其中： - N = 文档总数 - n(qi) = 包含关键词 qi 的文档数量

    IDF 的含义： - 词在越多文档中出现，IDF 越低（区分度低） - 词在越少文档中出现，IDF 越高（区分度高）

    flowchart TB A[词"的"] --> B[出现在 90% 文档中] B --> C[IDF ≈ 0] C --> D[不重要] E[词"RAG"] --> F[出现在 5% 文档中] F --> G[IDF ≈ 2] G --> H[很重要]

    3.2 词频饱和 (Term Frequency Saturation)¶

    BM25 通过参数 k1 控制词频饱和 [1][2]：

    TF 因子 = (k1 + 1) · tf / (k1 + tf)
    • 当 tf 较小时，因子近似线性增长
    • 当 tf 超过 k1 时，增长趋于平缓
    • k1 越大，饱和点越高
    flowchart TB A[词频 tf] --> B[饱和因子] B --> C[线性增长区] C --> D[饱和区]

    为什么需要饱和？ - 一个词在文档中出现 10 次并不比出现 5 次重要 2 倍 - 避免高频词主导评分

    3.3 文档长度归一化¶

    参数 b 控制文档长度对评分的影响 [2]：

    长度因子 = 1 - b + b · |D| / avgdl
    • b = 0：完全忽略文档长度
    • b = 1：完全根据文档长度调整
    • 通常 b = 0.75

    含义： - 长文档更容易包含查询词，但不应因此获得更高分 - 短文档如果包含查询词，应该获得相对更高的分数

    BM25 与 TF-IDF 的对比¶

    1. TF-IDF 公式¶

    TF-IDF 是 BM25 的前身 [3]：

    Score TF-IDF(qi, D) = TF(qi, D) · IDF(qi)

    2. 对比表格¶

    维度 TF-IDF BM25
    词频处理 线性增长 饱和处理
    文档长度 不考虑 归一化
    参数调优 无 k1, b 可调
    实际效果 基础 显著更好
    适用范围 小规模 大规模

    3. 为什么 BM25 更好？¶

    flowchart LR A[长文档] --> B[词频天然更高] B --> C[分数虚高] D[长文档] --> E[长度归一化] E --> F[分数更公平]

    根据 CSDN 技术博客的分析 [1][2]，BM25 的改进包括：

    1. 词频饱和：避免高频词主导结果
    2. 文档长度归一化：长文档不再天然占优
    3. 参数可调：可根据领域定制

    倒排索引 (Inverted Index)¶

    1. 什么是倒排索引？¶

    倒排索引是关键词检索的核心数据结构 [1]：

    flowchart LR A[Doc1] --> D["词A, 词B"] B[Doc2] --> E["词B, 词C"] C[Doc3] --> F["词A, 词C"] G[词A] --> J["Doc1, Doc3"] H[词B] --> K["Doc1, Doc2"] I[词C] --> L["Doc2, Doc3"]

    2. 倒排索引结构¶

    倒排表结构：
{
  "关键词": [
    { doc_id: 1, tf: 3, positions: [5, 12, 30] },
    { doc_id: 5, tf: 1, positions: [8] },
    ...
  ]
}

    3. 检索流程¶

    flowchart LR A[查询"机器 学习"] --> B[分词] B --> C["机器", "学习"] C --> D[查倒排表] D --> E[获取候选文档] E --> F[BM25 打分] F --> G[排序] G --> H[返回结果]

    BM25 在 RAG 中的应用¶

    1. 为什么 RAG 需要 BM25？¶

    尽管向量检索已成为主流，但 BM25 在 RAG 系统中仍有重要价值 [4][5]：

    场景 适用性 原因
    精确术语匹配 ⭐⭐⭐ 品牌名、代码等需要精确匹配
    混合检索 ⭐⭐⭐ 与向量检索互补
    冷启动 ⭐⭐⭐ 无需训练数据
    可解释性 ⭐⭐⭐ 明确知道哪个词匹配了

    2. 混合检索架构¶

    现代 RAG 系统通常结合 BM25 和向量检索 [5]：

    flowchart TB A[用户查询] --> B[向量化] A --> C[分词] B --> D[向量检索] C --> E[BM25 检索] D --> F[向量得分] E --> E1[BM25 得分] F --> G[分数融合] E1 --> G G --> H[RRF/加权融合] H --> I[最终排序]

    3. Elasticsearch 中的 BM25¶

    Elasticsearch 是最流行的开源搜索引擎，其底层使用 BM25 算法 [4]：

    // Elasticsearch 查询示例
{
  "query": {
    "match": {
      "content": "RAG 技术"
    }
  }
}

    BM25 参数调优¶

    1. 参数说明¶

    参数 默认值 作用 调优建议
    k1 1.2 词频饱和 短查询增大，长文档减小
    b 0.75 长度归一化 长文档增多，搜索质量好
    min_score 0 最低分数 过滤低相关结果

    2. 调优策略¶

    场景1：短查询

    查询："什么是 RAG"
建议：k1 = 1.5-2.0, b = 0.75

    场景2：长查询

    查询："请详细介绍一下检索增强生成技术的原理和应用场景"
建议：k1 = 1.0-1.2, b = 0.5-0.75

    场景3：长文档为主

    文档：技术文档、白皮书（平均 5000+ 字）
建议：k1 = 1.2, b = 0.5-0.6

    BM25 的局限性¶

    1. 无法处理语义¶

    BM25 能处理 BM25 不能处理
    精确关键词匹配 同义词（如"电脑"和"计算机"）
    词组匹配 语义相似（如"退款"和"退货"）
    短语顺序 复杂语义理解

    2. 中文处理挑战¶

    中文需要分词，面临以下问题：

    问题 影响 解决方案
    未登录词 新词无法识别 补充词典、时俱进
    歧义分词 错误切分影响召回 上下文消歧
    领域词汇 专业术语切分不当 领域词典

    总结¶

    BM25 是关键词检索的经典算法：

    1. 算法基础：基于词频统计和逆文档频率
    2. 核心改进：词频饱和和文档长度归一化
    3. 实际应用：Elasticsearch 等搜索引擎的核心
    4. RAG 价值：与向量检索混合使用，互补长短

    虽然 BM25 不如向量检索"智能"，但在精确匹配和混合检索场景中仍是不可替代的技术。
