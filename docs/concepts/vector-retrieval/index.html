    向量检索 (Vector Retrieval)¶

    什么是向量检索？¶

    向量检索是 RAG 系统中基于向量相似度进行信息检索的技术。它利用 Embedding 模型将查询和文档都转换为向量表示，然后在向量空间中通过计算相似度来找到与查询最相关的文档。

    一句话理解¶

    向量检索 = 在"语义空间"中找最相似的邻居

    向量检索在 RAG 中的位置¶

    flowchart LR A[文档] --> B[分块] B --> C[向量化] C --> D[向量索引]:::blue E[用户查询] --> F[向量化] F --> G[向量检索]:::blue G --> H[相似度计算] H --> I[返回 Top-K] classDef blue fill:#e3f2fd

    向量检索基础原理¶

    1. 核心思想¶

    向量检索的核心思想是将语义相似的内容映射到向量空间中相近的位置 [1]。这一思想基于分布假设（Distributional Hypothesis）："语义相似的词倾向于出现在相似的上下文中" [2]。

    flowchart TB A[""退款流程""] --- B[""退货政策""] B --- C[""售后服务""] C --- D[""客户支持""] D --- E[""产品咨询""] F[查询向量] --> G[语义空间] G --> H[距离最近的文档]

    2. 检索流程¶

    向量检索的完整流程包括以下步骤：

    flowchart LR A[用户查询] --> B[向量化] B --> C[构建查询向量] C --> D[ANN 索引搜索] D --> E[候选集] E --> F[精确距离计算] F --> G[排序] G --> H[返回 Top-K]

    步骤详解：

    1. 向量化：将用户查询通过 Embedding 模型转换为固定维度的向量
    2. 索引搜索：利用 ANN（近似最近邻）算法快速筛选候选集
    3. 距离计算：计算查询向量与候选向量的相似度
    4. 排序返回：按相似度降序排列，返回 Top-K 结果

    3. 相似度度量¶

    根据腾讯云的分析 [1]，向量检索中常用的相似度度量包括：

    度量方式 公式 特点 适用场景
    余弦相似度 cos(θ) = (A·B) / (||A||·||B||) 关注向量方向，不受长度影响 语义匹配、文本检索
    欧氏距离 √Σ(Ai-Bi)² 关注绝对距离 精确匹配、图像检索
    点积 A·B = ΣAiBi 兼具方向和 magnitude 推荐系统、排序

    为什么余弦相似度最常用：

    • 文本向量长度差异可能很大（长文档 vs 短查询）
    • 余弦相似度只比较方向，更稳定
    • 向量归一化后可与点积等价，计算效率高

    近似最近邻搜索 (ANN)¶

    1. 为什么需要 ANN？¶

    精确的最近邻搜索（KNN）需要遍历所有向量，对于亿级数据集来说计算量过大。根据百度百科的解释 [3]，ANN 算法通过牺牲一定的精度来换取显著的性能提升。

    flowchart TB A[100万向量] --> B{KNN vs ANN} B --> C[KNN 精确搜索] C --> D[遍历 100万次] D --> E[延迟: 秒级] B --> F[ANN 近似搜索] F --> G[遍历 ~1万次] G --> H[延迟: 毫秒级] style C fill:#ffcdd2 style F fill:#c8e6c9

    2. HNSW 算法详解¶

    HNSW（Hierarchical Navigable Small World）是当前最流行的 ANN 算法之一 [1]。它结合了跳表（Skip List）和小世界（Small World）网络的特性：

    核心概念：

    概念 说明
    节点 向量数据点
    边 节点间的连接
    层 分层结构，高层节点稀疏
    贪婪遍历 每步选择最近的邻居

    搜索过程（图解）：

    flowchart TD A[查询向量 Q] --> B[从最高层开始] B --> C[找到最近入口节点] C --> D{当前层还有更近的节点?} D -->|是| E[移动到更近的节点] D -->|否| F[跳到下一层] E --> D F --> G{到达最底层?} G -->|否| H[继续搜索] G -->|是| I[返回最终结果]

    参数配置建议：

    参数 说明 低延迟场景 高召回场景
    ef 搜索宽度 10-50 100-500
    M 每个节点的边数 8-16 16-64

    3. IVF 倒排索引¶

    IVF（Inverted File）是一种基于聚类的索引方法 [1]：

    原理： 1. 使用 K-Means 将向量聚类为 nlist 个簇 2. 构建倒排索引：每个簇记录属于它的向量 3. 搜索时先定位到最近的几个簇，再在簇内精确搜索

    flowchart LR A[所有向量] --> B[K-Means 聚类] B --> C[建立倒排索引] D[查询向量] --> E[找最近的 nprobe 个簇] E --> F[在簇内搜索]

    4. PQ 乘积量化¶

    PQ（Product Quantization）通过压缩向量来减少内存占用 [1]：

    原理： 1. 将高维向量分割为多个子向量 2. 对每个子向量空间进行独立的 K-Means 聚类 3. 用聚类中心的 ID 来表示原始向量

    flowchart TB A[原始向量 128维] --> B[分割为 4 个子向量] B --> C[每个子向量 32维] C --> D[分别量化] D --> E[压缩后的 ID]

    向量检索在 RAG 中的最佳实践¶

    1. 检索质量优化¶

    问题与解决方案：

    问题 原因 解决方案
    召回率低 Embedding 模型不匹配 更换或微调模型
    语义鸿沟 查询与文档表述差异 使用 HyDE 或查询扩展
    长文档稀释 分块太大 调整分块策略

    2. 分块策略对检索的影响¶

    根据 CSDN 技术博客的分析 [4]，分块策略直接影响检索效果：

    分块大小 优点 缺点 适用场景
    小 (100-300字) 语义聚焦 上下文缺失 精确问答
    中 (500-800字) 平衡 - 通用场景
    大 (1000+字) 上下文完整 语义稀释 摘要任务

    3. 检索参数调优¶

    参数 说明 调优建议
    top_k 返回结果数量 根据下游任务调整，通常 3-10
    score_threshold 相似度阈值 过滤低相关结果，通常 0.5-0.7
    ef HNSW 搜索宽度 越大召回越高，但延迟增加

    向量检索 vs 其他检索方式¶

    1. 与关键词检索对比¶

    维度 向量检索 关键词检索 (BM25)
    匹配方式 语义匹配 精确匹配
    同义词处理 ✅ 自动 ❌ 需要规则
    新词处理 ✅ 可处理 ❌ 可能遗漏
    计算成本 较高 较低
    可解释性 较低 较高

    2. 混合检索¶

    现代 RAG 系统通常结合向量检索和关键词检索 [5]：

    flowchart LR A[用户查询] --> B[向量化] A --> C[关键词提取] B --> D[向量检索] C --> E[BM25 检索] D --> F[结果融合] E --> F F --> G[最终结果]

    总结¶

    向量检索是 RAG 系统的核心技术：

    1. 核心原理：将查询和文档映射到向量空间，通过相似度计算找到相关内容
    2. 算法基础：HNSW、IVF 等 ANN 算法是实现高效检索的关键
    3. 质量优化：需要结合 Embedding 模型、分块策略、参数调优
    4. 最佳实践：与关键词检索混合使用，可以兼具体验和精确性
