    RAG 性能优化与调优指南¶

    RAG 系统从“Demo 级”走向“生产级”的分水岭在于其检索准确率和响应延迟的精细化调优。生产环境下的 RAG 面临着数据量大、问题模糊、并发高等挑战。

    1. 索引质量调优 (The Indexing Pipeline)¶

    分块策略 (Chunking Strategy)¶

    • 语义分块 (Semantic Chunking)：利用 NLP 模型（如 BERT）寻找语义变化的断点，优于固定大小切分。
    • 重叠比例 (Overlap)：建议保持 10%-20% 的重叠，以防止语义在切分点断裂，导致检索失败。
    • 层级分块 (Parent-Child Chunking)：在大块中保留小块，检索小块以匹配语义，喂给 LLM 大块以提供完整上下文。

    向量模型选择 (Embedding Model)¶

    • 维度权衡：1536 维（OpenAI）精度高但检索慢；768 维（BGE/Ollama）在本地化部署中更具性价比。
    • 特定领域微调：在法律、医疗、金融等强术语领域，建议对通用模型进行对比学习微调。

    2. 检索阶段优化 (Retrieval Optimization)¶

    混合检索 (Hybrid Search) 的权重调优¶

    公式：Score = α * VectorScore + (1-α) * KeywordScore - 建议值： - 侧重语义（如“如何处理...”）：α = 0.7 - 侧重术语（如“产品型号 XYZ-123”）：α = 0.3 - 动态权重：基于查询分类动态调整，如检测到特定实体词则自动增加关键词权重。

    向量数据库索引加速¶

    • HNSW 算法：通过构建多层图实现 ANN 搜索，牺牲极小精度获取毫秒级响应。
    • 标量量化 (Scalar Quantization)：将 float32 压缩为 int8，减少 75% 内存消耗，速度提升 2-4 倍。

    3. 响应延迟优化 (Latency Reduction)¶

    阶段 优化技术 预期效果
    查询增强 异步并行执行 HyDE / Multi-Query 减少串行等待时间
    检索 向量库索引预热 (Pre-warming) 减少首次检索冷启动延迟
    重排 级联过滤 (先过滤 50 个结果，再重排前 5 个) 降低 Cross-Encoder 的算力开销
    生成 推理流式输出 (Streaming) 显著降低首字延迟 (TTFT)

    4. 常见的“Bad Case”调优路线图¶

    • 检索不到：
      1. 检查解析器是否漏字、漏表。
      2. 增加 Top-K 召回数量。
      3. 检查分块是否太碎，导致语义不完整。
    • 检索到了但 LLM 没用上：
      1. 检查 Context Compressor 是否去重太狠。
      2. 检查 Rerank 分数是否过低。
      3. 优化 Prompt，明确指出“仅根据上下文回答”。
    • 回答存在幻觉 (Hallucination)：
      1. 启用 Self-RAG 或自我反思机制。
      2. 增加“如果无法找到答案，请诚实回答不知道”的强制约束。

    5. 核心指标监控 (KPIs)¶

    • 检索阶段：Recall@k, MRR (Mean Reciprocal Rank)。
    • 生成阶段：Faithfulness（忠实度）、Relevance（相关性）。
    • 工程指标：P95 Latency, Tokens Per Second (TPS), 成本消耗。
