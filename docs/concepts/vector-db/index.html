    向量数据库 (Vector Database)¶

    什么是向量数据库？¶

    向量数据库是专门用于存储和检索高维向量数据的数据库系统。在 RAG 系统中，向量数据库扮演着至关重要的角色，它存储了文档经过 Embedding 模型转换后的向量表示，支持基于语义相似度的快速检索。

    一句话理解¶

    向量数据库 = 专门存储"语义坐标"并支持快速相似度搜索的数据库

    向量数据库在 RAG 中的位置¶

    flowchart LR A[原始文档] --> B[分块] B --> C[向量化] C --> D[向量数据库]:::blue E[用户查询] --> F[向量化] F --> G[向量数据库] G --> H[相似度检索] H --> I[召回结果] classDef blue fill:#e3f2fd

    为什么需要向量数据库？¶

    传统数据库的局限¶

    传统数据库 局限 RAG 场景问题
    关系型数据库 精确匹配，无法处理语义 "退款"查不到"退货"
    全文检索 (Elasticsearch) 基于关键词，无法理解语义 同义词、多义词处理困难
    传统索引 无法支持高维向量检索 亿级向量检索延迟过高

    向量数据库的核心价值¶

    根据腾讯云开发者社区的分析 [1]，向量数据库相较传统数据库能够很好地处理以下场景：

    价值 说明
    大规模数据处理 支持亿级向量的高效存储和检索
    低时延高并发 毫秒级响应，支持高并发查询
    语义理解 支持基于向量相似度的语义检索
    多模态支持 可存储文本、图像、音频等多种向量化数据

    向量数据库核心概念¶

    1. 近似最近邻搜索 (ANN)¶

    由于高维空间中精确搜索的计算复杂度呈指数级增长，向量数据库通常采用近似最近邻（Approximate Nearest Neighbor, ANN）算法来平衡精度和性能 [1]。

    flowchart TB A[查询向量] --> B[遍历所有向量] B --> C[计算距离] C --> D[排序] D --> E[返回精确结果] F[查询向量] --> G[ ANN 索引] G --> H[快速过滤] H --> I[返回近似结果] style A fill:#e8f5e9 style F fill:#fff3e0

    2. 主流 ANN 算法¶

    根据搜索结果 [1]，主流的 ANN 算法包括：

    算法 全称 原理 特点
    HNSW Hierarchical Navigable Small World 构建分层导航小世界图 搜索快、精度高、内存占用大
    IVF Inverted File 倒排索引 + 聚类 实现简单、适合大规模数据
    PQ Product Quantization 乘积量化压缩向量 内存占用极低、精度有所下降
    LSH Locality-Sensitive Hashing 局部敏感哈希 对近似重复数据效果好

    HNSW 算法详解¶

    HNSW（Hierarchical Navigable Small World）是当前最流行的向量索引算法之一，由 Yury Malkov 等人于 2016 年提出 [1]。其核心思想是构建一个分层的 navigable small world 图结构：

                        ┌─────────┐
                    │ Layer 3 │  ← 顶层：少量节点，长跳跃
              ┌─────┴─────────┴─────┐
              │                     │
        ┌─────┴─────┐         ┌─────┴─────┐
        │  Layer 2 │         │  Layer 2 │
  ┌─────┴─────┐     │   ┌─────┴─────┐     │
  │  Layer 1  │     │   │  Layer 1  │     │
  │ (多数节点) │     │   │           │     │
  └───────────┘     │   └───────────┘     │
        │           │         │           │
        └───────────┴─────────┴───────────┘
                    Layer 0            ← 底层：所有节点，短距离连接

    搜索过程： 1. 从最高层开始，找到与查询向量最近的入口节点 2. 沿当前层的边贪心遍历，找到当前层的最近邻 3. 跳到下一层，从上一步的最近邻开始继续搜索 4. 重复直到最底层，返回最终结果

    参数调优：

    参数 说明 建议值
    M 每个节点的边数 16-64
    efConstruction 建索引时的搜索宽度 200-500
    ef 查询时的搜索宽度 50-200

    IVF-PQ 混合索引¶

    IVF-PQ 是结合了倒排索引和乘积量化的混合算法：

    flowchart LR A[原始向量] --> B[IVF 聚类] B --> C[每个簇内] C --> D[PQ 压缩] D --> E[存储压缩向量] F[查询向量] --> G[先找最近簇] G --> H[在簇内解压搜索]

    3. 距离度量¶

    向量数据库支持多种距离度量方式 [1]：

    度量方式 公式 特点 适用场景
    余弦相似度 cos(θ) = A·B/(||A||·||B||) 关注方向 语义匹配
    欧氏距离 ||A - B|| 关注绝对距离 精确匹配
    点积 A·B 综合考虑 排序推荐

    向量数据库核心操作¶

    1. 数据写入流程¶

    flowchart LR A[原始数据] --> B[预处理] B --> C[向量化] C --> D[构建索引] D --> E[存储到数据库] F[id] G[文档内容] H[分块信息] I[时间戳] E -.-> F E -.-> G E -.-> H E -.-> I

    2. 相似度检索流程¶

    flowchart LR A[用户查询] --> B[向量化] B --> C[ANN 索引搜索] C --> D[距离计算] D --> E[Top-K 排序] E --> F[返回结果] G[元数据过滤] -.-> C G --> H[重排序]

    3. 过滤查询¶

    向量数据库通常支持元数据过滤：

    # 示例：带过滤条件的查询
results = collection.search(
    query_vector=[0.1, 0.2, ...],
    filter="category == '技术文档' AND year >= 2024",
    limit=10
)

    向量数据库最佳实践¶

    1. 索引构建策略¶

    数据规模 推荐索引 参数建议
    < 100万 HNSW M=16, ef=100
    100万-1000万 HNSW M=32, ef=200
    > 1000万 IVF-PQ nlist=1024, nprobe=32

    2. 性能优化技巧¶

    批量写入：

    # 批量写入比单条写入快 10-100 倍
vectors = [[...], [...], ...]  # 批量向量
collection.add(vectors)

    预计算 + 缓存： - 热门查询的向量可以预计算并缓存 - 使用向量数据库的近似结果 + 精确重排

    内存管理： - HNSW 需要将全部向量加载到内存 - PQ 可以大幅降低内存占用

    3. 常见问题与解决¶

    问题1：检索召回率低

    解决： - 调整 HNSW 的 ef 参数（增大） - 降低 M 参数以构建更密的图 - 检查 Embedding 模型是否适合你的数据

    问题2：内存占用过高

    解决： - 使用 IVF-PQ 代替纯 HNSW - 对向量进行量化（INT8/FLOAT16） - 分片部署到多台机器

    问题3：过滤查询慢

    解决： - 尽量在向量检索前完成过滤（Pre-filter） - 建立适当的索引（如倒排索引） - 使用支持原生过滤的数据库

    选择要点¶

    考量因素 建议
    数据规模 亿级选 Milvus/Pinecone，百万级可选轻量方案
    延迟要求 毫秒级选 Qdrant 或 Redis
    部署方式 云服务选 Pinecone，自部署选 Milvus/Qdrant
    团队熟悉度 PostgreSQL 团队优先选 pgvector
    成本 开源方案（Milvus/Qdrant）vs 云服务（Pinecone）

    总结¶

    向量数据库是 RAG 系统的核心基础设施：

    1. 核心能力：支持高维向量的高效存储和相似度检索
    2. 算法基础：HNSW、IVF-PQ 等 ANN 算法是性能关键
    3. 产品选择：根据数据规模、延迟要求、部署方式选择
    4. 最佳实践：批量操作、合理调参、适当过滤

    选择合适的向量数据库和正确的索引参数，能够显著提升 RAG 系统的检索质量和响应速度。
