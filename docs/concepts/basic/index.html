    RAG 工作原理¶

    什么是 RAG？¶

    RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型生成相结合的技术架构。简单来说，RAG 让 AI 在回答问题前先去"查阅资料"，然后基于资料内容生成答案。

    一句话理解¶

    RAG = 先查阅资料再回答问题

    核心价值¶

    问题 RAG 解决方案
    幻觉 检索真实文档作为依据，答案有据可查
    时效性 检索最新文档，获取实时信息
    知识不足 利用外部知识库，弥补模型知识缺陷
    不可解释 提供文档来源，支持答案溯源

    为什么需要 RAG？¶

    大语言模型的局限性¶

    尽管大语言模型（LLM）能力强大，但仍存在根本性问题：

    flowchart LR A1["知识截止日期"]:::red A2["幻觉问题"]:::red A3["专业领域知识不足"]:::red classDef red fill:#ffcdd2
    问题 说明 示例
    知识截止 模型知识停留在训练时间点 GPT-4 不知道 2024 年发生的事
    幻觉 模型可能生成看似正确但错误的内容 自信地说"爱因斯坦出生在 1900 年"
    专业性不足 垂直领域知识有限 医疗、法律等专业问题回答不准确

    RAG 的解决思路¶

    RAG 通过"先检索再生成"的模式，从根本上解决这些问题：

    用户问："苹果公司 2024 年的营收如何？"
        ↓
    [RAG 系统]
        ↓
    检索：找到苹果公司 2024 年财报
        ↓
    生成：基于真实财报数据生成答案
        ↓
    答案："根据苹果公司 2024 年财报，营收为..."

    RAG 工作原理¶

    完整工作流程¶

    RAG 系统分为两个主要阶段：索引阶段和查询阶段。

    flowchart LR D[原始文档] --> P[解析文档] P --> C[分块处理] C --> E[向量化] E --> S[(向量数据库)] Q[用户问题] --> EQ[查询向量化] EQ --> R[相似度检索] R --> F[构建上下文] F --> G[LLM 生成] G --> A[返回答案]

    索引阶段详解¶

    索引阶段负责将文档转化为可检索格式：

    步骤 说明 产出
    解析 提取 PDF、Word 等格式的文本 纯文本
    分块 将长文档切分为小段落 文本块
    向量化 使用 Embedder 将文本转为向量 向量
    存储 将向量存入向量数据库 索引完成

    查询阶段详解¶

    查询阶段负责回答用户问题：

    步骤 说明 产出
    查询向量化 将用户问题转为向量 查询向量
    相似度检索 在向量库中找最相似的文档块 相关文档
    构建上下文 将检索结果组合成提示词 完整上下文
    LLM 生成 基于上下文生成答案 最终答案

    检索与生成的关系¶

    检索（Retrieval）¶

    核心问题：如何找到"真正相关"的文档？

    检索面临的核心挑战是"语义鸿沟"——用户的问题表达与文档内容可能存在很大差异：

    用户问题 文档表述 问题
    "怎么退钱？" "退款流程说明" 表述不同
    "它去年的表现" "2023 年年度报告" 指代不明
    "Go 咋样" "Go 语言的优势" 口语化

    解决方案： - 向量检索（语义相似度） - 关键词检索（BM25） - 混合检索（两者结合）

    生成（Generation）¶

    核心问题：如何"有效利用"检索结果生成高质量答案？

    flowchart LR A[用户问题] --> B[上下文] B --> C[Prompt 组装] C --> D[LLM 生成] D --> E[答案输出] B -->|"检索到的文档块"| C

    关键考虑： - 上下文长度控制（不能超过 LLM 上下文限制） - 噪声过滤（移除不相关的检索结果） - 答案引用（标注答案来源）

    RAG 的技术演进¶

    第一代：Naive RAG¶

    最简单的 RAG 实现：

    用户问题 → 向量检索 → 直接生成答案

    优点：简单、直接 缺点：检索质量差，生成效果有限

    第二代：Advanced RAG¶

    在检索和生成之间增加优化：

    用户问题 → [查询优化] → 向量检索 → [结果优化] → 生成答案

    优化手段： - Query Rewrite（查询重写） - HyDE（假设文档嵌入） - Rerank（重排序）

    第三代：Modular RAG¶

    模块化设计，自由组合：

    用户问题 → [路由] → [查询规划] → [并行检索] → [融合] → 生成答案

    特点：高度灵活，可针对不同场景定制

    第四代：Agentic RAG¶

    引入 AI 智能体，具备自主决策能力：

    用户问题 → [智能体决策] → [多步推理] → [自我反思] → 最终答案

    能力： - 判断是否需要检索 - 规划检索策略 - 评估结果质量 - 必要时迭代优化

    RAG 与 Fine-tuning 的对比¶

    维度 RAG Fine-tuning
    知识更新 实时，更换文档即可 需要重新训练
    成本 低，无需训练 高，需要 GPU 训练
    可解释性 高，答案可溯源 低，知识隐含在模型中
    定制程度 中等，受限于检索质量 高，完全定制模型行为
    幻觉问题 较少，有据可查 仍然存在

    选择建议： - 知识频繁更新 → RAG - 需要定制对话风格 → Fine-tuning - 需要高可解释性 → RAG - 垂直领域深度定制 → 两者结合

    RAG 系统评估¶

    三大核心指标¶

    指标 说明 重要性
    Faithfulness 答案是否忠实于检索内容 ⭐⭐⭐⭐⭐
    Answer Relevance 答案是否回答了问题 ⭐⭐⭐⭐
    Context Precision 检索结果排序是否合理 ⭐⭐⭐

    评估方法¶

    flowchart LR A[Query + Chunks + Answer] --> B[LLM Judge] B --> C[Faithfulness] B --> D[Relevance] B --> E[Precision]
