    HyDE 原理及应用详解¶

    概述¶

    HyDE（Hypothetical Document Embeddings，假设文档嵌入）是一种用于查询转换（Query Transformation）的创新方法，特别适用于 RAG 系统中的复杂问题处理。该方法由论文 Precise Zero-Shot Dense Retrieval without Relevance Labels 提出。

    核心思想¶

    HyDE 的核心思想是：不直接对用户查询进行嵌入，而是先让 LLM 生成一个"假设文档"来回答这个问题，然后对这个假设文档进行嵌入，最后用假设文档的嵌入去检索相似的真实文档。

    工作原理¶

    传统 RAG 检索流程¶

    用户查询 → 向量化 → 向量检索 → 返回结果

    HyDE 检索流程¶

    用户查询 → LLM生成假设文档 → 向量化假设文档 → 向量检索真实文档 → 返回结果

    实现步骤¶

    步骤 1：用户输入提问¶

    用户提出一个自然语言问题，例如："什么是机器学习？"

    步骤 2：生成假设文档¶

    使用 LLM（通常是 GPT 等指令跟随型模型）生成一个能回答该问题的假设文档。这个假设文档：

    • 可能是虚构的
    • 可能包含错误的细节
    • 但包含与问题相关的语义信息

    例如，LLM 可能生成：

    "机器学习是人工智能的一个分支，它使计算机能够从数据中学习并改进性能，而无需明确编程。机器学习算法通过识别模式来进行预测或决策..."

    步骤 3：编码假设文档¶

    使用无监督的稠密检索模型（如 Contriever）将生成的假设文档编码为稠密向量。

    步骤 4：检索真实文档¶

    使用假设文档的向量，在向量数据库中找到相似的真实文档，作为最终检索结果返回。

    技术架构¶

    HyDE 将密集检索分解为两个任务：

    1. 指令跟随的语言模型生成任务：由 LLM 生成假设文档
    2. 对比编码器的文档相似性任务：由检索模型执行向量相似度匹配

    实验结果¶

    根据论文和社区实验：

    • HyDE 在各个方面都对 Contriever 带来了显著提升
    • 虽然无监督的 Contriever 在性能上可能不及经典的 BM25 方法
    • 但 HyDE 大幅超越了 BM25

    优点¶

    1. 零样本检索：HyDE 可以"开箱即用"，无需依赖大量标注示例的数据集
    2. 跨语言：它在各种语言中表现良好，适用于多语言搜索应用
    3. 灵活性：HyDE 的方法使其能够适应不同的任务，而无需进行大量的微调
    4. 语义桥接：有效弥合查询与文档嵌入空间之间的语义差距

    适用场景¶

    HyDE 特别适合以下场景：

    • 专业术语：当一个概念有多种表述方式时（如医学术语）
    • 语义模糊：用户查询表述模糊，需要扩展语义理解
    • 跨语言检索：多语言环境下的检索任务
    • 冷启动：没有足够标注数据的场景

    与其他查询转换技术的对比¶

    技术 方法 适用场景
    HyDE 生成假设文档进行检索 语义扩展、多语言
    Rewrite-Retrieve-Read 重写查询再检索 查询表述不清
    Step-Back Prompting 抽象化问题 复杂推理任务
    Query2doc 生成伪文档扩展查询 语义增强
