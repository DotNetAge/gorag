    存储后端选择¶

    本指南围绕 GoRAG 的存储后端选择展开，系统梳理文档存储、图谱存储与向量存储的接口设计、实现差异、性能特征与适用场景，并给出安装配置要点、连接参数、成本与扩展性分析以及在不同环境中的选型建议。读者可据此在本地开发、测试与生产环境中做出稳健的存储方案决策。

    项目结构¶

    GoRAG 将“文档存储”“图谱存储”“向量存储”“索引存储”“会话记忆”等能力抽象为统一接口，具体实现分别提供本地嵌入式数据库（Bolt、SQLite）、云原生向量数据库（Milvus、Pinecone、Qdrant、Weaviate）以及图数据库（Neo4j）。下图概览了核心存储接口与实现的关系：

    核心组件¶

    • 文档存储（DocStore）
    • 负责持久化完整文档与分块（Chunk），支持按文档或分块检索，是高级 RAG（如父文档检索、多跳）的基础。
    • 图谱存储（GraphStore）
    • 支持节点（实体）与边（关系）的增删改查、邻居查询、语义属性查询、社区摘要读取，支撑 GraphRAG。
    • 向量存储（VectorStore）
    • 提供向量插入、相似度搜索、过滤、删除与关闭；不同实现具备不同的索引策略、指标类型与元数据承载方式。
    • 索引存储（IndexStore）
    • 记录已构建索引的拓扑结构与配置，用于增量更新与路由。
    • 会话记忆（ChatMemory）
    • 维护对话历史缓冲，适配多轮与智能体 RAG。

    详细组件分析¶

    文档存储实现对比¶

    • Bolt 文档存储
    • 特点：嵌入式键值数据库，单机部署、零运维；适合小规模、开发/测试场景。
    • 关键行为：初始化桶、序列化写入、基于游标扫描分块。
    • 适用场景：本地原型、CI 测试、低并发。
    • SQLite 文档存储
    • 特点：关系型表结构，事务安全、索引支持；适合中小规模、需要结构化查询的场景。
    • 关键行为：建表、事务批量写入、按文档 ID 查询分块。
    • 适用场景：中小体量知识库、需要 SQL 查询辅助治理。
    • 选择建议
    • 开发/测试：优先 Bolt；中试/预生产：SQLite。
    • 生产：若需强一致与 SQL 辅助治理，SQLite 更合适；若追求极致简单，Bolt 可继续保留。

    图谱存储实现对比¶

    • Bolt 图谱存储
    • 特点：KV 存储，通过邻接列表维护出入边；不支持复杂查询（如 Cypher）。
    • 关键行为：节点/边 Upsert、邻接列表更新、广度优先邻居遍历、社区摘要读取。
    • 适用场景：轻量图谱、无需复杂查询的场景。
    • SQLite 图谱存储
    • 特点：关系型表结构，支持递归 CTE 进行邻居查询；可存储社区摘要。
    • 关键行为：节点/边 Upsert、递归 CTE 邻居查询、原生 SQL 查询、社区摘要读取。
    • 适用场景：需要 SQL 查询与社区摘要的场景。
    • Neo4j 图谱存储
    • 特点：原生图数据库，Cypher 查询语言、成熟生态；适合复杂图分析与大规模图谱。
    • 关键行为：节点/边 MERGE 写入、Cypher 邻居查询、任意 Cypher 查询、社区摘要查询。
    • 适用场景：企业级 GraphRAG、复杂关系推理。
    • 选择建议
    • 小规模/轻量：Bolt 或 SQLite；中大规模/复杂查询：Neo4j。

    向量存储实现对比¶

    • govector
    • 特点：本地嵌入式向量存储，使用 HNSW 索引；默认维度 1536，适合本地测试。
    • 关键行为：集合创建、Upsert、Search（支持过滤）、Delete、Close。
    • 适用场景：本地开发、离线/私有化部署。
    • Milvus
    • 特点：分布式向量数据库，支持多种索引与度量；自动建集合、索引创建与加载。
    • 关键行为：客户端重试、集合存在性检查、Upsert（批量写入与 Flush）、Search（表达式过滤）、Delete、Close。
    • 适用场景：生产级向量检索、高吞吐/低延迟。
    • Pinecone
    • 特点：托管向量服务，按需弹性扩缩；支持元数据过滤。
    • 关键行为：客户端初始化、索引描述与连接、Upsert（向量+元数据）、Search（向量+过滤）、Delete、Close。
    • 适用场景：快速上线、SaaS 化、弹性成本。
    • Qdrant
    • 特点：开源向量引擎，支持多值过滤；默认本地部署。
    • 关键行为：客户端初始化、Upsert（向量+元数据）、Search（Filter 构造）、Delete、Close。
    • 适用场景：自建/私有化、对过滤类型要求高的场景。
    • Weaviate
    • 特点：图+向量混合存储，支持 GraphQL 查询；可将元数据作为属性或 JSON 字段。
    • 关键行为：集合确保与属性定义、Upsert（向量+属性）、Search（NearVector + Where 过滤）、Delete、Close。
    • 适用场景：需要图/向量一体化的场景。
    • 选择建议
    • 本地/私有化：govector；SaaS/弹性：Pinecone；生产/高可用：Milvus；自建/过滤复杂：Qdrant；图向一体化：Weaviate。

    性能考量¶

    • 文档存储
    • Bolt：顺序扫描分块时可能成为瓶颈；适合小数据集。
    • SQLite：建立索引后按文档 ID 查询分块效率较高；事务批量写入吞吐可观。
    • 图谱存储
    • Bolt：邻接列表维护，遍历邻居较高效；不支持复杂查询。
    • SQLite：递归 CTE 查询邻居，SQL 引擎优化较好；适合中小规模。
    • Neo4j：原生图遍历与 Cypher 优化，适合复杂关系与深度查询。
    • 向量存储
    • govector：本地磁盘 IO，适合小规模；HNSW 提升检索效率。
    • Milvus：分布式索引、可调索引参数；吞吐与延迟取决于集群规模与网络。
    • Pinecone：托管弹性，延迟稳定；成本与写入/查询量相关。
    • Qdrant：本地部署，过滤类型丰富；吞吐受 CPU/内存影响。
    • Weaviate：图向一体，查询路径复杂度高；适合混合场景。
    • 成本与扩展性
    • 本地/嵌入式：govector 最低成本；Bolt/SQLite 仅需磁盘空间。
    • SaaS/托管：Pinecone 按请求计费，弹性扩容；Milvus 需自建/托管集群，成本与规模相关。
    • 自建/私有化：Qdrant/Weaviate 需要运维资源；Neo4j 需要数据库实例管理。
    • 选型建议
    • 小规模/开发：govector/Bolt/SQLite
    • 中小规模/需要 SQL：SQLite/Neo4j
    • 生产/高吞吐：Milvus/Pinecone
    • 私有化/过滤复杂：Qdrant
    • 图向一体化：Weaviate

    故障排除指南¶

    • 文档存储
    • Bolt/SQLite 打不开数据库：确认路径存在且权限正确；Bolt 需确保目录存在；SQLite 需确保文件可写。
    • 查询不到文档/分块：确认 ID 是否正确；SQLite 使用按文档 ID 查询分块；Bolt 需检查桶是否存在。
    • 图谱存储
    • Bolt：复杂查询未实现，需自行扩展；邻居查询受深度与限制影响。
    • SQLite：递归 CTE 查询受限于数据库方言；注意 LIMIT 与性能。
    • Neo4j：认证失败、数据库名错误、Cypher 语法错误；确保驱动连接参数正确。
    • 向量存储
    • govector：存储路径不可写、集合初始化失败；确认 dbPath 与维度设置。
    • Milvus：客户端连接超时、集合不存在；确认地址可达、HasCollection 与 CreateCollection 重试逻辑。
    • Pinecone：API Key 错误、索引描述失败；确认密钥与索引名。
    • Qdrant：客户端初始化失败、Upsert/Query 报错；确认主机/端口与集合存在。
    • Weaviate：集合不存在、属性未定义；确保 ensureCollectionExists 成功。
    • 通用排查
    • 网络连通性：Milvus/Pinecone/Qdrant/Weaviate 均需检查网络与防火墙。
    • 元数据格式：Pinecone/Qdrant/Weaviate 需确保元数据结构与类型正确。
    • 日志与指标：结合项目日志与指标模块定位问题。

    结论¶

    • 文档存储：本地/测试用 Bolt，中小规模用 SQLite，复杂治理用 Neo4j。
    • 图谱存储：轻量用 Bolt，需要 SQL 用 SQLite，复杂关系用 Neo4j。
    • 向量存储：本地/私有化用 govector，SaaS 用 Pinecone，生产用 Milvus，自建/过滤复杂用 Qdrant，图向一体化用 Weaviate。
    • 选型应综合数据规模、查询复杂度、成本预算、运维能力与扩展需求，必要时采用组合方案（如 SQLite 文档 + Milvus 向量 + Neo4j 图谱）。

    附录¶

    • 安装与配置要点
    • govector：设置 dbPath、dimension、collection 名称与 HNSW 开关。
    • Milvus：提供地址，自动建集合与索引；注意网络可达与重试。
    • Pinecone：提供 API Key 与索引名；自动连接索引 Host。
    • Qdrant：提供集合名、维度、主机与端口。
    • Weaviate：提供集合名、维度、地址与可选 API Key；自动创建类与属性。
    • Bolt/SQLite：确保路径存在与权限正确；Bolt 初始化桶，SQLite 建表。
    • 连接参数
    • govector：dbPath、collection、dimension、useHNSW
    • Milvus：addr、collection、dimension、indexType、metricType
    • Pinecone：apiKey、indexName、environment、dimension、namespace
    • Qdrant：collection、dimension、host、port
    • Weaviate：collection、dimension、addr、apiKey
    • Bolt/SQLite：文件路径
