    RAG应用类型对比¶

    的五类应用类型：Native RAG、Advanced RAG、Graph RAG、Agentic RAG 与 CRAG。内容涵盖架构差异、默认配置、性能特征、资源消耗、扩展能力与典型使用场景，并提供可视化图表与决策指南，帮助读者基于业务目标做出正确选型。

    项目结构¶

    GoRAG 采用模块化分层设计，核心接口集中在 core 包，具体检索器按类型分布在 pkg/retriever 下，应用入口通过 gorag.go 的工厂函数统一构建。整体结构清晰、职责分离，便于替换向量库、图库、嵌入模型与 LLM。

    graph TB subgraph "应用层" APP["RAG 应用<br/>gorag.go"] end subgraph "检索器层" NATIVE["Native RAG<br/>pkg/retriever/native"] ADV["Advanced RAG<br/>pkg/retriever/advanced"] GRAPH["Graph RAG<br/>pkg/retriever/graph"] AGENTIC["Agentic RAG<br/>pkg/retriever/agentic"] CRAG["CRAG<br/>pkg/retriever/crag"] end subgraph "核心接口层" CORE["核心接口与类型<br/>pkg/core"] end subgraph "基础设施" DI["依赖注入容器<br/>pkg/di"] EVAL["评估器<br/>pkg/generation/evaluator"] end APP --> NATIVE APP --> ADV APP --> GRAPH APP --> AGENTIC APP --> CRAG NATIVE --> CORE ADV --> CORE GRAPH --> CORE AGENTIC --> CORE CRAG --> CORE APP --> DI CRAG --> EVAL

    核心组件¶

    • RAG 接口与默认实现
    • RAG 接口定义了索引与检索的统一入口，defaultRAG 将 indexer 与 retriever 组合为应用实例。
    • 核心类型与上下文
    • Query、RetrievalResult、CRAGEvaluation、IntentType 等类型定义了检索流程中的输入输出与评估标签。
    • RetrievalContext 作为管道执行上下文，承载查询、中间结果、代理状态与可观测性信息。
    • 依赖注入容器
    • 提供服务注册与解析机制，支持单例/瞬态模式，便于在不同环境注入具体实现。

    详细组件分析¶

    Native RAG（原生语义检索）¶

    • 特点
    • 单一向量检索链路，适合标准问答与本地知识库。
    • 默认 TopK=5，轻量、低延迟、低资源占用。
    • 默认配置
    • 工作目录 ./data；向量库 govector；维度 1536；TopK 5。
    • 处理逻辑
    • 步骤：向量搜索 → 生成答案。
    • 性能与资源
    • 计算与存储开销最小；适合中小规模知识库与边缘部署。
    • 适用场景
    • AI 助手、FAQ、本地文档问答。
    flowchart TD S["开始"] --> V["向量搜索 TopK=5"] V --> G["生成答案"] G --> E["结束"]

    Advanced RAG（高级融合检索）¶

    • 特点
    • 实现 RAG-Fusion：查询分解 + 多路检索 + RRF 融合，提升召回质量。
    • 默认 TopK=10，适合复杂或多角度问题。
    • 默认配置
    • 工作目录 ./data；向量库 govector；维度 1536；TopK 10。
    • 处理逻辑
    • 步骤：查询分解 → 向量搜索 → RRF 融合 → 生成答案。
    • 性能与资源
    • 搜索次数增加，吞吐略降但准确性显著提升；适合企业级高召回场景。
    • 适用场景
    • 对比分析、跨领域综合问答、技术文档检索。
    flowchart TD S["开始"] --> D["查询分解"] D --> VS["向量搜索 TopK=10"] VS --> F["RRF 融合"] F --> G["生成答案"] G --> E["结束"]

    Graph RAG（图谱关系推理）¶

    • 特点
    • 结合实体抽取、图邻域搜索与向量混合检索，支持多跳关系推理。
    • 默认 TopK=5；可配置深度 depth 与节点限制 limit。
    • 默认配置
    • 工作目录 ./data；向量库 govector；维度 1536；TopK 5；深度 1；限制 10。
    • 处理逻辑
    • 步骤：实体抽取 → 图邻域搜索 → 可选文档增强 → 向量混合 → 生成答案。
    • 性能与资源
    • 图查询与 LLM 交互带来额外开销；适合关系密集的知识库。
    • 适用场景
    • 企业知识图谱问答、合规审查、产品关联分析。
    flowchart TD S["开始"] --> EE["实体抽取"] EE --> GS["图邻域搜索(depth,limit)"] GS --> VE["向量混合检索 TopK=5"] VE --> EN["可选: 文档增强"] EN --> GEN["生成答案"] GEN --> E["结束"]

    Agentic RAG（智能体驱动检索）¶

    • 特点
    • 以智能体为核心，结合工具调用与记忆会话，实现动态推理与多步检索。
    • 通过智能路由将不同意图分派到 Native/Advanced/Graph 等子检索器。
    • 默认配置
    • 工作目录 ./data；向量库 govector；维度 1536；TopK 5。
    • 路由规则：聊天→Native；事实核查→Advanced；关系→Graph；垂直领域→Advanced。
    • 处理逻辑
    • 步骤：智能体执行 → 生成答案（可携带 agent 步骤元数据）。
    • 性能与资源
    • 依赖 LLM 推理成本较高；适合需要强交互与上下文记忆的应用。
    • 适用场景
    • 智能客服、垂直领域咨询、多轮对话与任务编排。
    sequenceDiagram participant U as "用户" participant APP as "AgenticRAG" participant CLS as "意图分类器" participant R1 as "Native" participant R2 as "Advanced" participant R3 as "Graph" U->>APP : 提交查询 APP->>CLS : 分类意图 CLS-->>APP : 返回 IntentType alt 聊天/事实核查/关系/垂直领域 APP->>R1 : 路由到 Native APP->>R2 : 路由到 Advanced APP->>R3 : 路由到 Graph end R1-->>APP : 检索结果 R2-->>APP : 检索结果 R3-->>APP : 检索结果 APP-->>U : 综合答案

    CRAG（自纠错检索）¶

    • 特点
    • 引入评估与回退机制：先向量检索，再由评估器判断相关性，必要时回退到网络搜索。
    • 支持文档增强（PDR），提升回答完整性。
    • 默认配置
    • 工作目录 ./data；向量库 govector；维度 1536；TopK 5。
    • 处理逻辑
    • 步骤：向量搜索 → CRAG 评估 → 文档增强（可选）→ 网络搜索回退（可选）→ 生成答案。
    • 性能与资源
    • 评估与回退引入额外调用；在“模糊/无关/歧义”场景显著提升质量。
    • 适用场景
    • 需要高质量与可解释性的问答系统、搜索引擎补充。
    flowchart TD S["开始"] --> VS["向量搜索 TopK=5"] VS --> EVAL["CRAG 评估"] EVAL --> DEC{"相关?"} DEC -- 是 --> GEN["生成答案"] DEC -- 否 --> WEB["网络搜索回退"] WEB --> GEN GEN --> E["结束"]

    依赖分析¶

    • 组件耦合
    • 应用层通过工厂函数统一装配检索器，降低上层对具体实现的耦合。
    • 检索器均依赖 core 接口，保证可替换性与测试友好性。
    • 外部依赖
    • 向量库：govector（默认）、milvus、weaviate、qdrant 等（通过选项切换）。
    • 图库：neo4j、sqlite（零 CGO）等（GraphRAG 专用）。
    • LLM/嵌入：通过 Provider/Client 注入，便于替换。
    • 循环依赖
    • 未发现直接循环依赖；模块间通过接口解耦。
    graph LR APP["gorag.go"] --> IF["core 接口<br/>interfaces.go"] NATIVE["native"] --> IF ADV["advanced"] --> IF GRAPH["graph"] --> IF AGENTIC["agentic"] --> IF CRAG["crag"] --> IF CRAG --> EVAL["评估器<br/>generation/evaluator"]

    性能考量¶

    • 计算与吞吐
    • Native 最低；Advanced 因多路搜索与融合略降；Graph 因图查询与 LLM 抽取成本较高；Agentic 受 LLM 推理影响最大；CRAG 在回退场景有额外网络开销。
    • 存储与内存
    • govector 本地嵌入存储，适合小中型知识库；企业级可切换 Milvus/Weaviate/Qdrant 等分布式向量库。
    • GraphRAG 额外需要图存储（Neo4j/SQLite），注意图构建与查询的资源开销。
    • 可扩展性
    • 通过选项与依赖注入容器可灵活替换实现；建议在高并发场景启用并发 worker 与流式解析。

    故障排查指南¶

    • 常见问题定位
    • 向量库初始化失败：检查工作目录权限与向量库参数（路径、维度）。
    • 图库不可用：确认图存储连接参数与可用性。
    • LLM 推理失败：检查 API Key、模型参数与超时设置。
    • CRAG 未触发回退：确认是否提供了 WebSearcher 实例以及评估标签。
    • 观测性
    • 开启日志与分布式追踪，利用 Span 标签与事件记录定位瓶颈。
    • 上下文与状态
    • 检查 RetrievalContext 中的中间结果与 AgenticContext 元数据，辅助复现与调试。

    结论¶

    • 若追求极致速度与低成本：选择 Native RAG。
    • 若强调高召回与多视角问答：选择 Advanced RAG。
    • 若需要关系推理与多跳推断：选择 Graph RAG。
    • 若需要智能体驱动与上下文记忆：选择 Agentic RAG。
    • 若需要高质量与可解释性，且存在模糊/无关/歧义问题：选择 CRAG。

    最终选型应综合业务目标、数据形态、资源预算与 SLA 要求，必要时组合使用（如 AgenticRAG 的智能路由）。

    技术对比表¶

    • Native RAG
    • 架构：向量搜索 → 生成
    • 默认 TopK：5
    • 适用：标准问答、本地知识库
    • 资源：低
    • Advanced RAG
    • 架构：查询分解 → 多路搜索 → RRF 融合 → 生成
    • 默认 TopK：10
    • 适用：对比分析、跨领域问答
    • 资源：中高
    • Graph RAG
    • 架构：实体抽取 → 图邻域 → 向量混合 → 生成
    • 默认 TopK：5；深度/限制可配
    • 适用：关系推理、合规审查
    • 资源：高
    • Agentic RAG
    • 架构：智能体执行 → 生成（可带 agent 步骤）
    • 默认 TopK：5
    • 适用：智能客服、多轮对话
    • 资源：高
    • CRAG
    • 架构：向量搜索 → 评估 → 文档增强 → 网络回退 → 生成
    • 默认 TopK：5
    • 适用：高质量问答、搜索引擎补充
    • 资源：中高
