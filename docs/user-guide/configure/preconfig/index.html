    预配置应用详解¶

    本文件系统性阐述 GoRAG 的预配置应用（DefaultNativeRAG、DefaultAdvancedRAG、DefaultAgenticRAG、DefaultGraphRAG）的默认行为、配置参数、初始化流程、默认存储设置、向量维度与检索参数，并提供完整的使用示例与自定义配置转换方法。目标是帮助开发者快速理解并正确使用这些预设，同时在需要时进行灵活扩展。

    项目结构¶

    GoRAG 将“索引”和“检索”封装为统一的 RAG 应用接口，通过工厂函数提供四种工业级预配置应用： - DefaultNativeRAG：本地优先、轻量级，适合 AI Agent 与本地知识库 - DefaultAdvancedRAG：企业级召回增强，融合查询分解与 RRF 融合 - DefaultAgenticRAG：智能路由到多种检索器，按意图分类 - DefaultGraphRAG：图谱增强检索，结合实体抽取与图邻域搜索

    graph TB subgraph "应用层" RAGApp["RAG 应用接口<br/>RAG 接口"] end subgraph "索引层" Indexer["Indexer 接口<br/>默认索引器"] Parsers["解析器集合"] Chunker["语义分块器"] Embedder["嵌入模型提供者"] VStore["向量存储"] DStore["文档存储"] GStore["图存储"] end subgraph "检索层" NR["原生检索器"] AR["高级检索器"] GR["图检索器"] Router["智能路由器"] end RAGApp --> Indexer RAGApp --> NR RAGApp --> AR RAGApp --> GR RAGApp --> Router Indexer --> Parsers Indexer --> Chunker Indexer --> Embedder Indexer --> VStore Indexer --> DStore Indexer --> GStore Router --> NR Router --> AR Router --> GR

    核心组件¶

    • RAG 接口：统一的索引与检索入口，包含 IndexFile、IndexDirectory、Search 方法
    • RAGConfig：扁平化配置对象，包含工作目录、向量数据库类型、图数据库类型、嵌入模型类型、LLM 类型、API Key、集合名、维度、TopK 等
    • RAGOption：可变配置函数，用于覆盖默认配置
    • defaultRAG：RAG 接口的具体实现，组合了 Indexer 和 Retriever

    架构总览¶

    预配置应用的核心构建流程如下： - 解析 RAGOption，生成 RAGConfig - 根据 VectorDBType 初始化向量存储（默认 govector） - 根据 GraphDBType 初始化图存储（默认 sqlite） - 基于模式选择对应的 Indexer 与 Retriever - 返回 defaultRAG 实例

F->>C : 创建默认配置并应用 RAGOption F->>B : 传入配置与模式 B->>VS : 按 VectorDBType 初始化 B->>DS : 按 GraphDBType 初始化 B->>IDX : 选择对应 DefaultIndexer B->>RET : 选择对应 DefaultRetriever B-->>F : 返回 defaultRAG F-->>U : 返回可用的 RAG 应用

    详细组件分析¶

    DefaultNativeRAG（原生检索）¶

    • 默认行为
    • 工作目录：./data
    • 向量数据库：govector
    • 向量维度：1536（OpenAI 默认）
    • TopK：5
    • 初始化流程
    • 使用 govector.NewStore 初始化本地向量存储
    • 使用 sqlite.NewDocStore 初始化本地文档存储
    • 选择 DefaultNativeIndexer
    • 选择 DefaultNativeRetriever，TopK 来自配置
    • 存储设置
    • 向量存储路径：{WorkDir}/vectors.db
    • 文档存储路径：{WorkDir}/docs.db
    • 检索参数
    • TopK：默认 5，可通过 WithTopK 覆盖
    • 维度：默认 1536，可通过 WithDimension 覆盖

    DefaultAdvancedRAG（高级检索）¶

    • 默认行为
    • 工作目录：./data
    • 向量数据库：govector
    • 向量维度：1536
    • TopK：10
    • 初始化流程
    • 使用 govector.NewStore 初始化本地向量存储
    • 使用 sqlite.NewDocStore 初始化本地文档存储
    • 选择 DefaultAdvancedIndexer
    • 选择 DefaultAdvancedRetriever，TopK 来自配置
    • 检索参数
    • TopK：默认 10，可通过 WithTopK 覆盖
    • 维度：默认 1536，可通过 WithDimension 覆盖
    • 企业级能力
    • 支持 WithMilvus、WithQdrant、WithWeaviate、WithPinecone 等外部向量库
    • 支持 WithOpenTelemetryTracer、WithPrometheusMetrics 等可观测性配置

    DefaultAgenticRAG（智能路由）¶

    • 默认行为
    • 工作目录：./data
    • 向量数据库：govector
    • 向量维度：1536
    • TopK：5
    • 初始化流程
    • 使用 govector.NewStore 初始化本地向量存储
    • 使用 sqlite.NewDocStore 初始化本地文档存储
    • 选择 DefaultAdvancedIndexer（用于支持多检索器）
    • 组装三个子检索器：原生检索器、高级检索器、图检索器
    • 使用 NewSmartRouter 按意图路由到不同检索器
    • 意图映射
    • Chat → 原生检索器
    • FactCheck → 高级检索器
    • Relational → 图检索器
    • DomainSpecific → 高级检索器
    • 检索参数
    • TopK：默认 5，可通过 WithTopK 覆盖
    • 维度：默认 1536，可通过 WithDimension 覆盖
    classDiagram class RouterRetriever { -classifier IntentClassifier -mapping map[IntentType]Retriever -defaultRet Retriever +Retrieve(ctx, queries, topK) []*RetrievalResult } class NativeRetriever class AdvancedRetriever class GraphRetriever RouterRetriever --> NativeRetriever : "Chat" RouterRetriever --> AdvancedRetriever : "FactCheck, DomainSpecific" RouterRetriever --> GraphRetriever : "Relational"

    DefaultGraphRAG（图增强检索）¶

    • 默认行为
    • 工作目录：./data
    • 向量数据库：govector
    • 向量维度：1536
    • TopK：5
    • 初始化流程
    • 使用 govector.NewStore 初始化本地向量存储
    • 使用 sqlite.NewDocStore 初始化本地文档存储
    • 选择 DefaultGraphIndexer
    • 选择 DefaultGraphRetriever，支持实体抽取、图邻域搜索、向量混合检索、答案生成
    • 检索参数
    • TopK：默认 5，可通过 WithTopK 覆盖
    • 维度：默认 1536，可通过 WithDimension 覆盖
    • 图存储
    • 支持 WithNeo4j、WithBoltDoc、WithSQLDoc 等图存储配置

    依赖关系分析¶

    • 预配置应用依赖关系
    • gorag.go 中的 DefaultXxxRAG 依赖各 retriever 包的 DefaultXxxRetriever
    • buildRAG 根据模式选择不同的 Indexer 与 Retriever
    • 默认存储由 govector 与 sqlite 提供，图存储可选 neo4j 或 sqlite/bolt
    • 接口契约
    • Retriever、VectorStore、DocStore、GraphStore 等接口定义于 core 包
    • Indexer 与 Retriever 均通过工厂函数与选项模式进行装配

    性能考量¶

    • 默认 TopK 设置
    • NativeRAG：5，适合快速响应与低延迟
    • AdvancedRAG：10，提升召回率，适合复杂查询
    • 并发与工作线程
    • 默认 AdvancedIndexer/GraphIndexer 使用并发与较多工作线程，适合批量索引
    • 向量维度
    • 默认 1536，适配 OpenAI 嵌入模型；可根据实际模型调整
    • 可观测性
    • 支持 Prometheus 指标与 OpenTelemetry 追踪，便于生产环境监控

    故障排查指南¶

    • 常见问题
    • 无法连接外部向量库（如 Milvus、Qdrant、Weaviate、Pinecone）：检查 WithMilvus/WithQdrant/WithWeaviate/WithPinecone 参数是否正确
    • 图存储未启用：GraphRAG 需要 GraphStore，确保 WithNeo4j/WithBoltDoc/WithSQLDoc 已配置
    • 日志输出：使用 WithZapLogger/WithFileLogger/WithConsoleLogger 定位问题
    • 建议步骤
    • 先用本地 govector+sqlite 验证流程
    • 再逐步替换为外部存储
    • 开启可观测性以捕获错误与耗时

    结论¶

    GoRAG 的预配置应用提供了从本地到企业级的完整方案。通过 RAGOption 与 Indexer/Retriever 的选项模式，可以在不修改核心逻辑的前提下灵活切换存储后端、调整检索参数与启用可观测性。建议先以 DefaultNativeRAG 快速验证，再根据场景选择 Advanced/Graph/Agentic 方案，并按需扩展自定义配置。

    自定义配置与预配置的关系与转换方法¶

    • RAGOption 到 IndexerOption 的映射
    • WithWorkDir → 通过 WithZapLogger/WithDefaultGoVector/WithDefaultSQLiteDoc 等组合实现
    • WithDimension → 通过 govector.WithDimension 或 milvus.WithDimension 等传递给底层存储
    • WithTopK → 通过 retriever 的 WithTopK 选项传递
    • 企业级存储替换
    • 用 WithMilvus/WithQdrant/WithWeaviate/WithPinecone 替换默认 govector
    • 用 WithNeo4j/WithBoltDoc/WithSQLDoc 替换默认 sqlite 图存储
    • 可观测性与日志
    • 通过 WithZapLogger/WithConsoleLogger/WithPrometheusMetrics/WithOpenTelemetryTracer 启用
