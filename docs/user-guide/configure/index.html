    应用配置¶

    本文件面向希望在生产环境中部署与调优 GoRAG 的工程师与架构师，系统性地阐述应用配置体系，覆盖以下主题： - 不同类型的 RAG 应用（Native、Advanced、Graph、Agentic）特点与适用场景 - 预配置应用的默认行为与可定制选项 - 完整配置参数说明（工作目录、向量数据库类型、嵌入器类型、LLM 类型等） - 使用 RAGOption 函数进行自定义配置的方法 - 存储后端选择指南与性能调优建议 - 实际配置示例与最佳实践 - 常见配置问题与故障排除方法

    项目结构¶

    GoRAG 采用模块化分层设计，核心由“索引器”“检索器”“管道步骤”“存储后端”“依赖注入容器”组成。应用通过一组预设工厂函数快速构建不同策略的 RAG 应用，并支持通过选项函数进行细粒度定制。

    核心组件¶

    • RAGConfig 与 RAGOption：统一的配置结构体与选项函数模式，用于工厂函数构建不同预设。
    • 预设工厂：DefaultNativeRAG、DefaultAdvancedRAG、DefaultAgenticRAG、DefaultGraphRAG，分别对应不同的检索策略与默认参数。
    • buildRAG 工厂：根据模式选择索引器与检索器组合，并装配向量/文档/图存储。
    • 检索器模块：Native、Advanced（RAG-Fusion）、Graph、Agentic，各自提供 Option 以注入嵌入器、LLM、Tracer、日志等。
    • 索引器选项：集中管理向量/文档/图存储、日志、指标、分布式追踪、嵌入器等配置。
    • 存储后端：本地 GoVector、Milvus、SQLite/Bolt 文档/图存储。
    • 依赖注入容器：提供服务注册、解析与全局默认容器能力。

    详细组件分析¶

    RAG 应用类型与适用场景¶

    • Native RAG
    • 特点：纯向量检索，轻量、低延迟、成本低。
    • 适用：标准语义搜索、本地知识库、对推理关系不敏感的场景。
    • Advanced RAG（RAG-Fusion）
    • 特点：查询分解 + 多视角检索 + RRF 融合，提升召回质量。
    • 适用：企业级高召回需求、复杂多面查询。
    • Graph RAG
    • 特点：结合实体抽取、图邻域遍历与向量混合检索，支持多跳关系推理。
    • 适用：强关系推理、知识图谱驱动的问答。
    • Agentic RAG
    • 特点：基于智能体的意图路由与工具使用，适合动态工作负载与复杂交互。
    • 适用：需要自动决策与工具编排的高级 Agent 场景。

    预配置应用与默认行为¶

    • 默认工作目录：./data
    • 默认向量维度：1536（适配主流嵌入模型）
    • 默认 TopK：
    • Native/Agentic：5
    • Advanced：10
    • 默认向量存储：govector（本地嵌入式存储）
    • 默认文档存储：SQLite（用于文档/片段持久化）
    • 可选图存储：SQLite（零 CGO），或通过索引器选项接入 Neo4j

    RAGOption 与配置参数详解¶

    • RAGConfig 字段
    • WorkDir：工作目录，决定日志、数据库文件位置
    • VectorDBType：向量库类型，如 "govector"、"milvus"
    • GraphDBType：图库类型，如 "neo4j"、"sqlite"
    • EmbedderType：嵌入器类型（通过索引器选项配置）
    • LLMType：大语言模型类型（通过索引器选项配置）
    • APIKey：外部服务密钥（如 OpenAI/Milvus 等）
    • Collection：集合/索引名称
    • Dimension：向量维度
    • TopK：检索返回数量
    • RAGOption 函数
    • WithWorkDir(path)：设置工作目录
    • WithAPIKey(key)：设置 API 密钥
    • 使用方式：将 RAGOption 传入 DefaultNativeRAG/DefaultAdvancedRAG/DefaultAgenticRAG/DefaultGraphRAG，覆盖默认配置

    检索器配置与可定制项¶

    • Native 检索器
    • Options：Logger、Tracer、Embedder、LLM、TopK、WorkDir、VectorStore、DocStore
    • Option：WithVectorStore、WithDocStore、WithWorkDir、WithLogger、WithTracer、WithEmbedder、WithLLM、WithTopK
    • Advanced 检索器（RAG-Fusion）
    • Options：Logger、Embedder、LLM、TopK、VectorStore
    • Option：WithLogger、WithEmbedder、WithLLM、WithTopK、WithStore
    • Graph 检索器
    • Options：TopK、Depth、Limit、PromptTemplate、DocStore、Logger、Tracer、Embedder、LLM、VectorStore、GraphStore、CustomSteps
    • Option：WithEmbedder、WithLLM、WithTopK、WithDepth、WithLimit、WithDocStore、WithPromptTemplate、WithLogger、WithTracer、WithCustomStep
    • Agentic 检索器
    • Options：Logger
    • Option：WithLogger

    索引器选项与存储后端¶

    • 向量存储
    • WithDefaultGoVector：本地 govector，默认集合名 gorag，维度 1536
    • WithGoVector(collection, path, dimension)：自定义集合、路径与维度
    • WithMilvus(collection, addr, dimension)：Milvus 远程集群
    • WithQdrant(collection, host, port, dimension)：Qdrant
    • WithWeaviate(collection, addr, apiKey, dimension)：Weaviate
    • WithPinecone(indexName, apiKey, dimension)：Pinecone
    • 文档存储
    • WithDefaultSQLiteDoc：SQLite 默认文档存储
    • WithSQLDoc(path)：自定义 SQLite 路径
    • WithBoltDoc(path)：BoltDB 文档存储
    • 图存储
    • WithNeoGraph(uri, username, password, dbName)：Neo4j
    • WithSQLDoc(path)：SQLite 图存储（默认）
    • 嵌入器
    • WithBGE(modelPath)、WithBert(modelPath)、WithClip(modelPath)
    • WithEmbedding(embedder)：显式注入嵌入器实例
    • 观测性
    • WithZapLogger(path, maxSizeMB, maxDays, maxBackups, console)：日志
    • WithPrometheusMetrics(addr)：指标
    • WithOpenTelemetryTracer(ctx, endpoint, serviceName)：分布式追踪
    • 并发与解析
    • WithConcurrency(enabled)、WithWorkers(workers)
    • WithParsers(...)、WithAllParsers()、ClearParsers()
    • WithDefaultSemanticChunker()、WithCharacterChunker(...)、WithTokenChunker(..., model)

    存储后端选择指南¶

    • 本地开发/小规模数据
    • 向量：govector（零依赖、易部署）
    • 文档：SQLite（零 CGO）
    • 图：SQLite（零 CGO）
    • 企业级/大规模
    • 向量：Milvus、Qdrant、Weaviate、Pinecone
    • 文档：SQLite（本地）、BoltDB（本地）
    • 图：Neo4j（关系建模强）
    • 性能与一致性权衡
    • govector：低延迟、单机、适合原型与小规模
    • Milvus/Qdrant：分布式、高吞吐、需运维
    • SQLite/Bolt：零 CGO、跨平台友好、适合嵌入式部署

    配置示例与最佳实践¶

    • 快速本地 Native RAG
    • 使用默认 govector 与 SQLite，设置工作目录即可
    • 参考路径：gorag.go:66-70
    • 企业 Advanced RAG（Milvus + OpenAI）
    • 通过索引器选项连接 Milvus 与嵌入器，再构建 Advanced 检索器
    • 参考路径：indexer_options.go:182-208
    • Graph RAG（Neo4j + 向量混合）
    • 通过 WithNeoGraph 与 WithGoVector 组合，启用实体抽取与图邻域搜索
    • 参考路径：indexer_options.go:154-158
    • 日志/指标/追踪
    • 使用 WithZapLogger、WithPrometheusMetrics、WithOpenTelemetryTracer
    • 参考路径：indexer_options.go:138-244

    依赖注入容器（DI）¶

    • 容器能力
    • 注册单例/瞬态服务、解析服务、全局默认容器
    • 提供类型安全解析与实例注册
    • 适用场景
    • 在复杂检索器（如 Agentic）中按需注入服务
    • 测试环境重置与隔离

    依赖分析¶

    • 组件耦合
    • buildRAG 将 RAGConfig 与检索器工厂解耦，通过模式分支选择不同组合
    • 检索器内部通过 Options 注入依赖，降低对外部配置的直接依赖
    • 外部依赖
    • 向量库：govector、milvus-sdk-go、qdrant、weaviate、pinecone
    • 图存储：neo4j、sqlite
    • 日志/观测：zap、prometheus、opentelemetry
    • 循环依赖
    • 未发现直接循环依赖；检索器通过接口与管道解耦
    graph LR CFG["RAGConfig"] --> BUILD["buildRAG"] BUILD --> IDX["索引器"] BUILD --> RET["检索器"] RET --> PIPE["管道步骤"] PIPE --> STORE["存储后端"]

    性能考虑¶

    • 向量维度与索引
    • 维度越高，内存与计算开销越大；默认 1536 适配主流模型
    • govector 支持 HNSW 开关；Milvus 可配置索引类型与度量
    • TopK 与并发
    • TopK 增大提升召回但增加延迟；建议按场景调参
    • WithWorkers 与 WithConcurrency 控制并发度
    • 缓存与降噪
    • 语义缓存阈值可调，减少重复请求
    • 分解器最大子查询数限制，避免过度分解
    • 稳定性
    • 为关键步骤添加熔断器包装，出现异常时快速回退

    故障排除指南¶

    • Milvus 连接失败
    • 现象：初始化客户端超时或集合不存在
    • 处理：检查地址、网络连通性；确认集合存在或允许自动创建
    • 参考路径：milvus_store.go:44-121
    • govector 存储打开失败
    • 现象：数据库文件无法打开或集合初始化失败
    • 处理：确认工作目录权限、磁盘空间；检查路径是否正确
    • 参考路径：govector_store.go:64-89
    • SQLite/Bolt 文档/图存储初始化失败
    • 现象：表创建失败或数据库打开失败
    • 处理：确认目录存在且有写权限；检查驱动导入
    • 参考路径：sqlite_docstore.go:26-64、sqlite_graphstore.go:26-38、bolt_docstore.go:29-60
    • 嵌入器未配置
    • 现象：Advanced/Graph 检索器缺少 Embedder 导致初始化失败
    • 处理：通过 WithEmbedding 或 WithBGE/Bert/Clip 显式注入
    • 参考路径：indexer_options.go:246-278
    • LLM/提示词问题（Graph）
    • 现象：Graph 生成步骤报错
    • 处理：检查 PromptTemplate 是否为空；确保 LLM 正常可用
    • 参考路径：retriever_graph.go:262-319

    结论¶

    GoRAG 提供了从本地到企业级的完整配置路径：通过 RAGOption 与索引器选项，可在不修改业务代码的前提下灵活切换存储后端、嵌入器与检索策略。结合内置观测性与稳定性组件，可在生产环境中实现高性能、可观测、可维护的 RAG 应用。
