    数据模型¶

    本文件系统性梳理 GoRAG 的核心数据模型与处理流程，围绕 Document（文档）、Chunk（块）、Vector（向量）三大实体，解释其定义、属性与关系；阐述文档解析到向量化的数据结构转换过程；介绍向量化表示与相似度计算的基础原理；说明元数据管理与上下文组织方式；给出不同存储后端的映射关系；并提供序列化/反序列化与数据完整性校验的实践建议。

    核心组件¶

    • Document（文档）
    • 唯一标识、内容、元数据、创建/更新时间、来源、内容类型等字段构成文档实体。
    • 提供构造函数与更新方法，确保时间戳一致性。
    • Chunk（块）
    • 文档分块实体，支持父子层级（用于细粒度检索与父文档回溯），包含起止索引、向量引用等。
    • 提供构造函数与设置向量 ID 的方法。
    • Vector（向量）
    • 向量化表示，包含向量值、对应块 ID 与元数据。
    • 提供构造函数，便于从嵌入结果快速生成向量实体。
    • 元数据与检索结果
    • Query、RetrievalResult 等类型承载查询与检索结果，便于后续重排、评估与回答生成。
    • 接口契约
    • Parser、Embedder、VectorStore、DocStore、IndexStore 等接口定义了数据模型与外部系统的交互边界。

    详细组件分析¶

    Document（文档）模型¶

    • 定义与职责
    • 表示一个完整的文档，包含唯一 ID、内容、元数据、来源与内容类型等。
    • 提供构造与更新方法，自动维护创建/更新时间。
    • 关键属性
    • 字段包括：ID、Content、Metadata、CreatedAt、UpdatedAt、Source、ContentType。
    • 使用场景
    • 作为解析器输出的统一载体，贯穿索引与检索阶段。
    • 序列化/反序列化
    • JSON 标签用于序列化；反序列化时注意时间字段与元数据的兼容性。

    Chunk（块）模型¶

    • 定义与职责
    • 表示已分块的文档片段，支持父子层级与上下文注入，便于细粒度检索与父文档回溯。
    • 关键属性
    • 字段包括：ID、DocumentID、ParentID、Level、Content、Metadata、CreatedAt、StartIndex、EndIndex、VectorID。
    • 分块策略
    • 支持基础分块与语义分块（HierarchicalChunk、ContextualChunk），用于高级 RAG 场景。
    • 使用场景
    • 作为向量化输入单元，并在检索后用于上下文增强与答案生成。

    Vector（向量）模型¶

    • 定义与职责
    • 表示块的向量化表示，包含向量值、对应块 ID 与元数据。
    • 关键属性
    • 字段包括：ID、Values、ChunkID、Metadata。
    • 生成流程
    • 由嵌入步骤根据块内容批量生成，同时回填块的向量引用。

    元数据与检索结果¶

    • Query、RetrievalResult 等类型用于承载查询文本、检索结果、答案与元数据，支撑后续重排与评估。
    • 检索结果中包含块列表与对应分数，便于排序与融合。

    解析到向量化的数据结构转换¶

    • 多解析器路由
    • 根据文件扩展名选择合适的解析器，构建元数据并流式解析为文档通道。
    • 文档到块
    • 将文档内容切分为多个块，保留起止索引与层级信息。
    • 块到向量
    • 批量嵌入块内容，生成向量实体并回填块的向量引用。
    • 写入存储
    • 文档与块写入 DocStore，向量写入 VectorStore。
    flowchart TD A["开始"] --> B["选择解析器"] B --> C["流式解析为Document"] C --> D["分块为Chunk"] D --> E["批量嵌入生成Vector"] E --> F["写入VectorStore"] E --> G["回填Chunk.VectorID"] D --> H["写入DocStore"] H --> I["结束"] F --> I

    向量化表示与相似度计算¶

    • 向量表示
    • 通过 Embedder 将文本编码为固定维度的浮点向量。
    • 相似度计算
    • 内存向量存储使用余弦相似度，并预计算向量 L2 范数以加速计算。
    • 支持基于元数据的过滤与 TopK 返回。
    • 过滤与排序
    • 在搜索前应用元数据过滤，随后对候选集进行 TopK 选择与排序。
    flowchart TD Q["查询向量"] --> N1["计算查询范数"] N1 --> S["遍历向量库"] S --> M{"元数据过滤?"} M --> |否| S2["跳过"] M --> |是| F["匹配过滤条件"] F --> CS["计算余弦相似度"] CS --> T["TopK 选择与排序"] S2 --> CS T --> O["返回向量与分数"]

    元数据管理与上下文信息组织¶

    • 元数据来源
    • 解析阶段将文件元数据（如来源、文件名、大小、修改时间）注入文档/块元数据。
    • 上下文增强
    • 检索后可从 DocStore 获取完整文档内容，附加到块元数据中，供生成器决定如何组织上下文。
    • 索引结构跟踪
    • IndexStore 记录索引拓扑与配置，支持增量更新与路由决策。

    数据模型在不同存储后端的映射¶

    • DocStore 映射
    • 文档表：id、content、metadata（JSON 序列化）
    • 块表：id、document_id、content、metadata（JSON 序列化）
    • 索引：块表按 document_id 建立索引，支持按文档 ID 查询块集合。
    • VectorStore 映射
    • 内存向量存储：以 map 存储向量与预计算范数，支持元数据过滤与 TopK 排序。
    • 实际生产中可替换为具体向量数据库实现（如 Milvus、Pinecone、Weaviate 等）。

    序列化与反序列化示例¶

    • 文档/块/向量的 JSON 标签可用于序列化与反序列化。
    • DocStore 中使用 JSON 对元数据进行序列化与反序列化，确保跨存储的一致性。
    • 嵌入步骤在生成向量时直接使用块元数据作为向量元数据，保持上下文一致。

    数据完整性检查与验证机制¶

    • 解析器测试
    • JSON/数据库模式解析器测试覆盖空对象、类型断言、元数据字段正确性等，确保输出文档具备 ID、内容与类型标记。
    • 流式解析健壮性
    • 多解析器路由根据扩展名选择解析器，避免不支持格式导致的失败。
    • 嵌入步骤校验
    • 嵌入步骤检查嵌入器与块通道有效性，失败时返回错误并记录指标。
    • DocStore 读写一致性
    • 文档与块的 JSON 序列化/反序列化保证跨存储一致性；删除操作采用事务以确保级联删除的原子性。

    依赖分析¶

    • 组件耦合
    • Document/Chunk/Vector 三者通过 ID/DocumentID/ChunkID 形成清晰的引用关系，降低跨模块耦合。
    • 接口层（Parser、Embedder、VectorStore、DocStore、IndexStore）解耦具体实现，便于替换与扩展。
    • 直接与间接依赖
    • 索引流水线依赖解析器与嵌入器；向量存储依赖嵌入器输出；DocStore 依赖文档/块实体。
    • 循环依赖
    • 当前设计未见循环依赖，接口抽象清晰。
    classDiagram class Document { +string ID +string Content +map~string,any~ Metadata +time.Time CreatedAt +time.Time UpdatedAt +string Source +string ContentType } class Chunk { +string ID +string DocumentID +string ParentID +int Level +string Content +map~string,any~ Metadata +time.Time CreatedAt +int StartIndex +int EndIndex +string VectorID } class Vector { +string ID +[]float32 Values +string ChunkID +map~string,any~ Metadata } Document "1" --> "*" Chunk : "包含" Chunk "1" --> "1" Vector : "被嵌入"

    性能考虑¶

    • 向量相似度优化
    • 预计算向量与查询的 L2 范数，减少重复开销；使用快速选择与排序算法选取 TopK。
    • 批量嵌入
    • 嵌入步骤批量处理块，减少网络/模型调用次数，提升吞吐。
    • 存储索引
    • DocStore 为块表建立按文档 ID 的索引，加速按文档查询块集合。
    • 元数据过滤
    • 在向量搜索前进行元数据过滤，缩小候选集，提高检索效率。

    故障排查指南¶

    • 解析失败
    • 检查文件扩展名是否受支持；确认解析器选择逻辑与文件路径正确。
    • 嵌入失败
    • 确认嵌入器已配置且可用；检查块通道是否为空；关注返回的错误信息与指标记录。
    • 存储异常
    • DocStore 删除文档时采用事务，确保文档与块级联删除；若出现不一致，检查事务提交与回滚逻辑。
    • 相似度异常
    • 检查向量维度与查询维度是否一致；确认范数计算与余弦相似度实现；核对元数据过滤条件。

    结论¶

    GoRAG 的数据模型以 Document/Chunk/Vector 为核心，配合解析器、嵌入器与存储接口，形成从“解析—分块—向量化—存储—检索—增强”的完整链路。通过明确的元数据管理、上下文增强与索引结构跟踪，系统能够支撑高级 RAG 场景。内存向量存储与 SQLite 文档存储提供了轻量级实现示例，便于扩展至生产级向量数据库与图数据库。
