    评估指标体系¶

    本文件系统化阐述 GoRAG 的评估指标体系，重点覆盖以下方面： - RAG 系统质量评估的重要性与评估维度 - RAGAS 评估指标的组成与计算方法 - 忠实度、答案相关性、上下文精准度等关键指标 - CRAG 评估与传统 RAG 评估的区别与适用场景 - 评估结果的统计分析与趋势监控 - 评估指标的配置与自定义方法 - 评估报告的解读与改进建议

    GoRAG 在 README 中明确指出内置了“企业级评测协议”，涵盖忠实度、答案相关性与上下文精准度（RAGAS 风格），并提供 CRAG 评估能力，用于处理模糊查询与质量评估，并可回退至网络搜索。

    核心组件¶

    • RAGAS 评估器：基于 LLM 的忠实度、答案相关性、上下文精准度评分，通过标准提示词模板与响应解析实现。
    • 传统 RAG 评估器：对整体 RAG 输出进行忠实度与相关性评估，返回综合分数与通过判定。
    • CRAG 评估器：针对检索上下文进行相关性评估，输出标签、理由与分数，支持后续回退策略。
    • 评估步骤：将评估器集成到检索生成流水线中，负责构建上下文、调用评估器、记录日志与指标。
    • 核心类型与接口：定义评估结果的数据结构、评估接口以及检索上下文状态，支撑评估流程与可观测性。

    详细组件分析¶

    RAGAS 评估器（忠实度、答案相关性、上下文精准度）¶

    • 设计要点
    • 通过 LLM 作为“裁判”对生成答案进行评分，分别评估忠实度、答案相关性与上下文精准度。
    • 采用固定格式的提示词模板，确保 LLM 输出可解析。
    • 解析 LLM 响应中的分数与理由，返回浮点分数与解释文本。
    • 关键指标
    • 忠实度：衡量答案是否严格基于检索到的上下文，避免幻觉。
    • 答案相关性：衡量答案是否直接回答用户问题。
    • 上下文精准度：衡量检索到的上下文是否包含回答所需信息。
    • 计算方法
    • 将检索到的上下文拼接为统一文本，按固定格式构造提示词，调用 LLM 后解析“Score: x.x”和“Reason: …”。
    • 若解析失败或格式不正确，采用默认值与默认理由。
    • 错误处理
    • LLM 调用失败时返回错误；响应解析失败时使用默认理由，分数默认为 0.0。

    传统 RAG 评估器（整体质量评估）¶

    • 设计要点
    • 对给定的查询、答案与上下文进行整体评估，返回忠实度、相关性与总体分数，并给出是否通过的判定。
    • 返回结构包含分数与反馈信息，便于下游使用。
    • 使用场景
    • 适用于需要快速评估 RAG 整体质量的场景，便于与业务阈值结合。
    classDiagram class RAGEvaluator { <<interface>> +Evaluate(ctx, query, answer, context) (*RAGEvaluation, error) } class ragEvaluator { -llm : chat.Client +Evaluate(...) } RAGEvaluator <|.. ragEvaluator

    CRAG 评估器（检索质量评估）¶

    • 设计要点
    • 针对检索到的上下文进行相关性评估，输出标签（相关/不相关/模糊）、理由与分数。
    • 评估结果可用于后续决策（如是否回退到网络搜索）。
    • 使用场景
    • 适用于处理模糊查询、需要对检索质量进行量化评估的场景。
    classDiagram class CRAGEvaluator { <<interface>> +Evaluate(ctx, query, chunks) (*CRAGEvaluation, error) } class cragEvaluator { -llm : chat.Client +Evaluate(...) } CRAGEvaluator <|.. cragEvaluator

    评估步骤（集成到流水线）¶

    • RAG 评估步骤
    • 从检索上下文中提取查询、答案与上下文，调用 RAG 评估器，将评估结果写入代理状态，并记录指标。
    • CRAG 评估步骤
    • 从检索上下文中收集所有检索块，调用 CRAG 评估器，将评估结果写入代理状态，并记录指标。
    • 日志与可观测性
    • 步骤在执行前后记录日志，包含查询、答案长度、评估分数等信息。
    • 指标接口支持记录评估结果数量等指标。

    数据模型与接口¶

    • 评估结果数据结构
    • RAGEvaluation：包含忠实度、相关性、总体分数、是否通过、反馈信息。
    • CRAGEvaluation：包含标签、理由、分数。
    • 评估接口
    • RAGEvaluator、CRAGEvaluator 定义评估器的统一接口，便于替换与扩展。
    • 检索上下文
    • AgenticContext 提供自定义字段，用于存放评估结果，便于后续步骤使用。

    依赖关系分析¶

    • 评估器依赖
    • RAGAS 评估器依赖 LLM 客户端进行评分与解析。
    • 传统与 CRAG 评估器同样依赖 LLM 客户端。
    • 评估步骤依赖
    • 评估步骤依赖检索上下文与可观测性接口，将评估结果写入上下文并记录指标。
    • 接口与类型耦合
    • 评估器通过统一接口与类型解耦，便于替换不同实现（如更换 LLM、自定义评分规则）。

    性能考量¶

    • LLM 调用成本
    • 评估器每次评估均需调用 LLM，建议在批量评估时合并请求或使用缓存策略，减少重复调用。
    • 响应解析开销
    • 解析 LLM 输出为分数与理由存在字符串处理成本，建议在上游进行必要的格式约束与校验。
    • 指标记录频率
    • 评估步骤会记录指标，建议根据业务需求控制记录频率，避免指标风暴影响系统性能。
    • 上下文拼接
    • RAGAS 评估器会将多个检索块拼接为上下文文本，建议限制最大上下文长度，避免 LLM 输入超限。

    故障排查指南¶

    • LLM 调用失败
    • 现象：评估步骤报错，无法获取评估结果。
    • 处理：检查 LLM 客户端配置与网络连通性；必要时降级为默认分数与理由。
    • 响应格式不正确
    • 现象：解析分数或理由失败，返回默认值。
    • 处理：确认提示词模板与 LLM 输出格式一致；在测试中验证解析逻辑。
    • 缺少上下文或答案
    • 现象：评估步骤跳过评估，记录调试日志。
    • 处理：检查上游检索与生成步骤是否正常产生上下文与答案。
    • 指标未记录
    • 现象：评估完成后未见指标上报。
    • 处理：确认传入的指标收集器非空，且实现已启用。

    结论¶

    GoRAG 的评估指标体系以 RAGAS 为核心，结合传统 RAG 与 CRAG 评估器，形成覆盖忠实度、答案相关性、上下文精准度与检索质量的完整评估闭环。通过评估步骤与可观测性接口，评估结果可被记录与追踪，为统计分析与趋势监控提供基础。实践中应关注 LLM 调用成本、响应解析稳定性与上下文长度控制，以获得稳定高效的评估体验。

    附录¶

    评估指标与计算方法对照¶

    • 忠实度（Faithfulness）
    • 定义：答案是否严格基于检索上下文，避免幻觉。
    • 计算：通过 RAGAS 评估器对答案与上下文进行评分。
    • 答案相关性（Answer Relevance）
    • 定义：答案是否直接回答用户问题。
    • 计算：通过 RAGAS 评估器对答案与查询进行评分。
    • 上下文精准度（Context Precision）
    • 定义：检索上下文是否包含回答所需信息。
    • 计算：通过 RAGAS 评估器对上下文与查询进行评分。
    • 传统 RAG 评估
    • 定义：对整体 RAG 输出进行忠实度与相关性评估。
    • 计算：返回综合分数与通过判定。
    • CRAG 评估
    • 定义：对检索上下文进行相关性评估，输出标签与分数。
    • 计算：返回标签、理由与分数，支持后续决策。

    • 替换 LLM 客户端
    • 通过注入不同的 chat.Client 实现，适配不同供应商或版本。
    • 自定义提示词模板
    • 在 RAGAS 评估器中调整提示词模板，以适配特定领域或风格。
    • 自定义解析逻辑
    • 在 parseEvalResponse 中扩展解析规则，支持更多输出格式。
    • 自定义评估器
    • 实现 RAGEvaluator 或 CRAGEvaluator 接口，替换为基于规则或外部服务的评估器。
    • 指标收集器
    • 通过实现 Metrics 接口，接入 Prometheus/OpenTelemetry 等监控系统。

    评估报告解读与改进建议¶

    • 忠实度低
    • 可能原因：检索上下文不足或包含噪声；答案过度依赖外部知识。
    • 改进建议：优化检索策略、引入上下文裁剪与过滤、加强检索块质量。
    • 答案相关性差
    • 可能原因：查询重写不准确、检索结果与问题不匹配。
    • 改进建议：使用查询重写与分解技术，提升检索质量。
    • 上下文精准度低
    • 可能原因：检索算法召回过多无关内容。
    • 改进建议：引入重排与融合策略，提高上下文相关性。
    • CRAG 评估结果偏向不相关
    • 可能原因：查询模糊或检索质量不佳。
    • 改进建议：结合 CRAG 评估结果回退至网络搜索或调整检索参数。
