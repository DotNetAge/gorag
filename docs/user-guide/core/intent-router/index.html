    意图分类与路由¶

    本章节面向“意图分类与路由”主题，系统阐述 GoRAG 的查询意图识别、查询分解与复杂问题简化策略，以及基于意图的智能路由机制。文档重点包括： - 查询意图识别的重要性与实现方法 - 不同类型查询的分类标准与分类算法 - 查询分解与复杂问题简化策略 - 智能路由机制如何选择最优检索策略 - 路由决策的上下文感知与动态调整能力 - 意图分类器的配置与优化指南 - 实际的分类示例与路由决策流程图

    核心组件¶

    • 意图分类器：对用户查询进行意图分类，输出意图类型、置信度与理由。
    • 查询分解器：将复杂问题拆分为若干子问题，便于并行检索与融合。
    • 实体抽取器：从查询中提取关键实体，辅助关系检索与过滤。
    • 查询改写器：将口语化或模糊查询改写为更利于向量检索的表达。
    • Step-back 生成器：生成更高层背景问题，先检索基础概念再回答具体问题。
    • 智能路由器：根据意图将查询分发到最适合的检索器（向量、图谱、全局等）。
    • 检索上下文与代理状态：承载查询、中间结果、子查询、缓存命中等上下文信息。
    • 缓存步骤与分解步骤：在管道中插入缓存检查与查询分解步骤，提升整体效率与质量。

    详细组件分析¶

    意图分类器（Intent Classifier）¶

    • 功能概述
    • 接收用户查询，调用 LLM 输出 JSON 结构的意图分类结果，包含意图类型、置信度与理由。
    • 支持自定义提示词模板、默认意图与最低置信度阈值，具备降级与可观测性记录能力。
    • 分类标准与算法
    • 分类标签：聊天对话、领域知识（向量）、关系（图谱）、全局摘要、事实核查。
    • 算法流程：构造提示词 → 发送消息给 LLM → 解析 JSON → 校验置信度 → 应用默认意图。
    • 关键参数与配置
    • 自定义提示词模板、默认意图、最小置信度阈值、日志与指标收集器。
    • 典型使用场景
    • 将日常闲聊路由到快速回答；将需要实体关系的问题路由到图谱检索；将需要实时信息的问题路由到外部验证或事实核查。

    查询分解器（Query Decomposer）¶

    • 功能概述
    • 将复杂问题拆分为 2-5 个独立且可回答的子问题，同时给出分解理由与复杂度标记。
    • 支持最大子查询数量限制与降级回退策略。
    • 分解策略
    • 子问题需独立可答、覆盖原问题不同方面、保持清晰具体。
    • 典型使用场景
    • 在智能路由前对复杂问题进行并行化处理，提高检索效率与答案质量。

    实体抽取器（Entity Extractor）¶

    • 功能概述
    • 从查询中抽取关键实体（人名、地名、组织、概念等），返回实体数组。
    • 用于后续的关系检索、过滤与上下文增强。
    • 典型使用场景
    • 配合图谱检索器定位实体关系；结合过滤器进行检索范围限定。

    查询改写器（Rewriter）¶

    • 功能概述
    • 将口语化、含代词或模糊的查询改写为更清晰、更适合向量检索的表达。
    • 典型使用场景
    • 在向量检索前进行预处理，减少噪声，提升检索质量。

    Step-back 生成器（Step-back）¶

    • 功能概述
    • 将具体问题抽象为更高层的背景问题，先检索基础概念，再回答具体问题。
    • 典型使用场景
    • 对于需要背景知识支撑的具体问题，先检索原理与历史背景，再进行具体回答。

    智能路由器（Smart Router）¶

    • 功能概述
    • 基于意图分类结果将查询分发到对应的检索器（向量、图谱、全局），若无映射则回退到默认检索器。
    • 将意图信息写入检索结果元数据，便于追踪与评估。
    • 决策流程
    • 对每个查询进行意图分类 → 查找映射检索器 → 若无映射则使用默认检索器 → 执行检索并将意图信息附加到结果元数据。
    • 上下文感知与动态调整
    • 通过日志与指标记录路由行为，便于监控与优化；可根据业务反馈调整映射表与阈值。

    检索上下文与代理状态¶

    • 检索上下文（RetrievalContext）
    • 保存原始查询、当前查询、检索中间产物、并行结果、重排分数、过滤条件、代理状态与最终答案。
    • 代理状态（AgenticContext）
    • 记录下一步动作、意图、历史消息、子查询、假设文档、Step-back 查询、缓存命中、过滤条件等。
    • 典型使用场景
    • 在查询分解、缓存检查、路由决策与生成阶段共享上下文，确保流程连贯与可追踪。

    缓存步骤与查询分解步骤¶

    • 缓存步骤（SemanticCacheCheck/Store）
    • 在检索前检查语义缓存命中，命中则直接返回答案；在生成后将答案写入缓存。
    • 查询分解步骤（Decompose）
    • 将复杂查询分解为子查询并写入检索上下文，供后续步骤使用。

    依赖关系分析¶

    • 接口与实现
    • 意图分类器、查询分解器、实体抽取器、查询改写器、Step-back 生成器均实现 core 包中的对应接口。
    • 智能路由器依赖 IntentClassifier 接口与 Retriever 接口，通过映射表将意图与检索器解耦。
    • 类型与上下文
    • Query、IntentResult、DecompositionResult、RetrievalResult 等类型贯穿整个流程，保证数据一致性。
    • RetrievalContext 与 AgenticContext 为流程提供上下文与中间状态。

    性能考量¶

    • LLM 调用成本控制
    • 通过最小置信度阈值与默认意图降级，避免不必要的 LLM 调用。
    • 在分类与分解失败时采用回退策略，降低端到端延迟。
    • 并行化与流水线
    • 查询分解后可并行执行多个子查询，提升吞吐。
    • 缓存命中可直接短路后续检索与生成步骤。
    • 指标与可观测性
    • 记录意图分类耗时、查询分解耗时等指标，便于性能分析与优化。

    故障排查指南¶

    • 意图分类失败
    • 现象：分类器返回错误或解析 JSON 失败。
    • 处理：检查 LLM 客户端配置、提示词模板、日志与指标；必要时降低最小置信度阈值或调整默认意图。
    • 查询分解失败
    • 现象：分解器返回空子查询或超出最大数量。
    • 处理：检查提示词模板与 LLM 输出格式；设置合理的最大子查询数量上限。
    • 路由失败
    • 现象：无映射检索器可用或默认检索器未配置。
    • 处理：完善映射表；确保默认检索器可用；检查日志与指标定位问题。
    • 缓存未命中
    • 现象：预期命中但未命中。
    • 处理：确认缓存键策略、相似度阈值与缓存存储配置；检查语义嵌入一致性。

    结论¶

    GoRAG 的意图分类与路由机制通过“意图识别 + 查询分解 + 智能路由”的组合，实现了对不同查询类型的精准分流与高效检索。该机制具备良好的扩展性与可观测性，能够根据业务需求动态调整映射表与阈值，从而在准确性与性能之间取得平衡。

    实际分类示例与路由决策流程图¶

    • 示例 1：聊天类查询
    • 输入：你好，今天天气如何？
    • 分类：chat
    • 路由：直接返回快速回答或事实核查（如需实时数据）
    • 示例 2：领域知识查询
    • 输入：公司退款政策是什么？
    • 分类：domain_specific
    • 路由：向量检索
    • 示例 3：关系查询
    • 输入：CEO 与 CTO 的汇报关系？
    • 分类：relational
    • 路由：图谱检索
    • 示例 4：全局摘要查询
    • 输入：项目核心技术有哪些？
    • 分类：global
    • 路由：全局检索
    • 示例 5：复杂问题
    • 输入：对比 A 与 B 的架构差异并说明优缺点
    • 分类：domain_specific 或 relational（取决于侧重）
