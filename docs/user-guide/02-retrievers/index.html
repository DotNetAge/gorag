    查询阶段¶

    是什么¶

    Retriever 是 GoRAG 中负责检索的核心组件，每个 Retriever 实现特定的检索策略。

    查询阶段全景¶

    GoRAG 的查询阶段设计遵循"实用原子模块 + 自由管线编排"的哲学。我们不强加固定的复杂业务流，而是提供当前 RAG 领域最前沿、被工业界验证有效的增强检索技术模块。

    查询阶段核心增强模组¶

    1. Query Rewriting (查询重写)¶

    • 痛点: 用户提问往往口语化、包含指代词（如"它去年的财报怎样？"），直接进行向量匹配效果极差。
    • 功能: 使用轻量级 LLM 将原始 Query 转化为对搜索引擎/向量库更友好的标准陈述句。
    • 模块: query.Rewriter / steps.NewQueryRewriteStep()

    2. Step-back Prompting (退步提问/抽象化检索)¶

    • 痛点: 用户提问过于具体（如"为什么爱因斯坦在1905年发表了狭义相对论？"），直接检索可能找不到原话。
    • 功能: 在检索前，用 LLM 将具体问题"退步"抽象为一个更高维度的背景问题（"爱因斯坦的生平轨迹和1905年的物理学背景是什么？"）。用这个抽象问题去检索，往往能召回更丰富、正确的底层逻辑。
    • 模块: query.StepBackGenerator / steps.NewStepBackStep()

    3. HyDE (Hypothetical Document Embeddings - 假设性文档嵌入)¶

    • 痛点: "问题"和"答案"在向量空间中往往不相似（非对称语义）。例如提问很短，而包含答案的文档很长。
    • 功能: 先让 LLM 根据问题"盲答"生成一篇假设性文档，然后对这篇假设性文档进行 Embedding，拿着它的向量去库里搜索真实的 Chunk。
    • 模块: query.HyDEGenerator / steps.NewHyDEStep()

    4. Metadata Pre-filtering (元数据前置过滤)¶

    • 痛点: 在海量企业知识库中，用户常带有明确的结构化条件（如"找一下 2023 年关于新能源的财报"）。纯向量检索可能会把 2022 年或其他的文档也召回，导致致命错误。
    • 功能: 使用小模型（或规则引擎）从 Query 中提取出结构化过滤条件（year=2023, tag=新能源）。在调用向量库搜索时，将这些条件作为硬性约束（Pre-filtering）传入，确保召回结果的绝对边界。
    • 模块: query.FilterExtractor / steps.NewQueryToFilterStep()

    5. RAG-Fusion & Multi-Query (多路召回与融合)¶

    • 痛点: 单一的查询可能无法覆盖复杂的知识面；稠密向量容易漏掉精准的关键词。
    • 功能: 将查询改写为多个相似子问题并发检索；同时使用向量检索和关键词检索（BM25）；最后使用 RRF (倒数排名融合) 算法合并去重。
    • 模块: retrieval.FusionEngine / steps.NewRAGFusionStep()

    6. Re-ranking (二次重排)¶

    • 痛点: 向量检索是粗筛，Top 10 结果往往排序混乱。
    • 功能: 引入 Cross-Encoder 模型。在召回 50 个 Chunk 后进行深度交叉打分，截取真正最相关的 Top 5。这是目前提升 RAG 质量性价比最高的技术。
    • 模块: retrieval.Reranker / steps.NewRerankStep()

    7. Citation & Grounding (引用注入与溯源追踪)¶

    • 痛点: 企业级问答必须有理有据，避免模型凭空捏造（幻觉），且用户需要点击链接查看原文。
    • 功能: 在构建上下文时，自动为每个 Chunk 注入角标（如 [doc_1]）；在生成阶段，通过 Prompt 强制要求模型使用角标引用；最后通过流式拦截器解析这些标签，返回带有原文出处的结构化响应。
    • 模块: generation.CitationInjector / steps.NewCitationGenerationStep()

    管线编排模式¶

    模式 A: 标准基础 RAG¶

    pipe := pipeline.NewPipeline()
pipe.AddSteps(
    steps.NewEmbeddingStep(embedder),
    steps.NewVectorSearchStep(store, topK=5),
    steps.NewGenerationStep(llm),
)

    模式 B: 高精度检索管线 (Advanced Precision RAG)¶

    pipe := pipeline.NewPipeline()
pipe.AddSteps(
    steps.NewQueryRewriteStep(llmClient),             // 1. 消除代词，补全问题
    steps.NewMultiWaySearchStep(vectorDb, bm25Db),   // 2. 向量 + 关键词 双路召回
    steps.NewRAGFusionStep(),                         // 3. RRF 算法合并结果
    steps.NewRerankStep(rerankerModel, topK=5),     // 4. 精确重排，截取前 5
    steps.NewGenerationStep(llmClient),               // 5. 最终生成
)

    模式 C: 假设性/抽象增强管线 (HyDE & Step-back)¶

    pipe := pipeline.NewPipeline()
pipe.AddSteps(
    steps.NewStepBackStep(llmClient),       // 将具体问题抽象化
    steps.NewHyDEStep(llmClient),           // 为抽象问题生成假答案
    steps.NewEmbeddingStep(embedder),       // 对假答案进行向量化
    steps.NewVectorSearchStep(store),
    steps.NewGenerationStep(llmClient),
)

    模式 D: 企业级精准溯源管线 (Enterprise RAG with Filters & Citations)¶

    结合了元数据硬过滤与输出溯源的严谨管线：

    pipe := pipeline.NewPipeline()
pipe.AddSteps(
    // 1. 提取条件 (e.g., year >= 2023, author = "John")
    steps.NewQueryToFilterStep(llmClient), 

    // 2. 检索时自动应用上一环提取的硬性 Filter
    steps.NewVectorSearchStep(store, topK=10), 

    // 3. 将结果格式化为带有 [doc_x] 标签的 Context，并强制生成时引用
    steps.NewCitationGenerationStep(llmClient), 
)

    内置 Retriever¶

    按类型分类¶

    基础检索¶

    Retriever 说明 适用场景
    native.Retriever 基础向量检索 简单问答、快速原型
    advanced.Retriever 高级检索 复杂查询、高精度
    agentic.Retriever Agent 检索 智能路由、多策略
    graph.Retriever 图检索 关系查询
    selfrag.Retriever 自我反思检索 高质量答案
    crag.Retriever 纠正型检索 开放域问答

    按功能分类¶

    类别 Retriever
    简单检索 native.Retriever
    优化检索 advanced.Retriever
    智能检索 agentic.Retriever
    图检索 graph.Retriever
    自评检索 selfrag.Retriever
    纠正检索 crag.Retriever

    选择指南¶

    按场景选择¶

    场景 推荐 Retriever
    简单文档问答 native.Retriever
    复杂查询优化 advanced.Retriever
    智能路由 agentic.Retriever
    关系推理 graph.Retriever
    高质量生成 selfrag.Retriever
    动态知识 crag.Retriever

    按需求选择¶

    需求 推荐
    性能优先 native.Retriever
    精度优先 advanced.Retriever
    智能优先 agentic.Retriever
    关系优先 graph.Retriever

    通用配置¶

    向量存储¶

    所有 Retriever 都需要 VectorStore：

    import "github.com/DotNetAge/gorag/pkg/indexing/vectorstore/govector"

vecStore, err := govector.NewStore(
    govector.WithDBPath("./data/vectors.db"),
    govector.WithDimension(1536),
)

    LLM¶

    高级 Retriever 需要 LLM：

    import "github.com/DotNetAge/gochat/pkg/llm/openai"

llm, err := openai.NewChat(
    openai.WithAPIKey("your-api-key"),
)

    创建 Retriever 示例¶

    Native Retriever¶

    import "github.com/DotNetAge/gorag/pkg/retriever/native"

retriever, err := native.NewRetriever(vecStore)

    Advanced Retriever¶

    import "github.com/DotNetAge/gorag/pkg/retriever/advanced"

retriever, err := advanced.NewRetriever(
    vecStore,
    advanced.WithLLM(llm),
    advanced.WithEmbedder(embedder),
)

    Agentic Retriever¶

    import "github.com/DotNetAge/gorag/pkg/retriever/agentic"

retriever, err := agentic.NewRetriever(
    vecStore,
    agentic.WithLLM(llm),
)

    注意事项¶

    1. 选择合适的 Retriever：根据场景选择，不要过度设计。

    2. 配置 LLM：高级 Retriever 需要 LLM 支持。

    3. 性能考虑：复杂 Retriever 延迟更高。

    4. 组合使用：可以在 Pipeline 中组合多个 Retriever。
