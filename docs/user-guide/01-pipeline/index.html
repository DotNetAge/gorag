    管线架构¶

    Pipeline 是 GoRAG 提供的管线框架，用于将多个 Step 按顺序组合，实现复杂的 RAG 流程。它是 GoRAG 的核心抽象，让开发者像搭积木一样灵活组合检索和处理步骤。

    GoRAG 将“索引”和“检索”两大流程抽象为两条独立但可组合的管道： - 索引管道：负责文档发现、解析、分块、嵌入、存储等步骤，形成向量与文档的持久化 - 检索管道：负责查询预处理、向量检索、结果融合/重排、上下文压缩、答案生成等步骤，产出最终回答

    graph TB subgraph "索引管道" D["Discover<br/>发现文件"] P["MultiParser<br/>多解析器"] C["Chunk<br/>语义分块"] E["Embed<br/>批量嵌入"] S["MultiStore<br/>多存储写入"] D --> P --> C --> E --> S end
    graph TB subgraph "检索管道" Q["QueryPrep<br/>查询预处理"] V["VectorSearch<br/>向量检索"] F["Fuse/Rerank<br/>融合/重排"] G["Generate<br/>生成答案"] Q --> V --> F --> G end

    核心组件¶

    • RAG 应用接口与默认实现：统一对外提供索引与检索能力，并按模式装配不同索引器与检索器
    • 管道与步骤：基于通用 pipeline 库构建，每个步骤实现 Step 接口，接收状态对象并向前推进
    • 检索器：封装向量检索、并发执行、指标记录与错误处理
    • 接口契约：Retriever、Generator、Embedder、VectorStore、Metrics 等，确保可替换性与可观测性
    Pipeline = 多个 Step 的有序组合

Step 1 → Step 2 → Step 3 → ... → 结果

    管道化架构与执行流程¶

    • 索引阶段
    • 解析文档 → 切分 Chunk → 嵌入向量 → 写入向量存储。
    • 检索阶段
    • 查询进入，可能经意图分类与路由 → 多路检索（向量/图/全局）→ 融合/重排序 → 生成答案。
    • 生成与评估
    • 基于上下文生成答案，使用 LLM Judge 进行 RAGAS 指标评估。
    flowchart TD S["开始"] --> P1["解析文档<br/>Parser"] P1 --> P2["切分 Chunk<br/>Chunker"] P2 --> P3["嵌入向量<br/>Embedder"] P3 --> P4["写入向量库<br/>VectorStore"] P4 --> Q["接收查询"] Q --> C["意图分类<br/>IntentClassifier"] C --> R["智能路由<br/>Router"] R --> V["向量检索<br/>VectorSearch"] V --> F["融合/重排序<br/>Fusion/Rerank"] F --> G["生成答案<br/>Generator"] G --> E["质量评估<br/>RAGAS"] E --> END["结束"]

    Step 接口¶

    type Step[T any] interface {
    Name() string
    Execute(ctx context.Context, state T) error
}

    Pipeline 创建¶

    添加 Step¶

    p.AddStep(step1)

    执行 Pipeline¶

    完整客户端示例¶

    基本 Pipeline¶

    package main

import (
    "context"
    "fmt"
    "log"

    "github.com/DotNetAge/gorag/pkg/core"
    "github.com/DotNetAge/gorag/pkg/pipeline"
    "github.com/DotNetAge/gorag/pkg/steps/rewrite"
    "github.com/DotNetAge/gorag/pkg/steps/vector"
    "github.com/DotNetAge/gorag/pkg/steps/rerank"
)

func main() {
    ctx := context.Background()

    // 创建 Pipeline
    p := pipeline.New[*core.RetrievalContext]()

    // 添加 Step
    p.AddStep(rewrite.NewQueryRewriter(llm))    // 查询重写
    p.AddStep(vector.NewSearchStep(vecStore))   // 向量检索
    p.AddStep(rerank.NewRerankStep(crossEncoder)) // 重排序

    // 创建状态
    state := &core.RetrievalContext{
        Query: &core.Query{Text: "Go 语言的优势"},
    }

    // 执行
    err := p.Execute(ctx, state)
    if err != nil {
        log.Fatal(err)
    }

    fmt.Println("检索结果:", len(state.Chunks), "个文档块")
}

    Pipeline 复杂度选择¶

    复杂度 适用场景 示例
    ⭐ 简单检索 Search → Result
    ⭐⭐ 标准 RAG Rewrite → Search → Rerank → Result
    ⭐⭐⭐ 高级 RAG 多策略 → 融合 → 评估 → 生成

    注意事项¶

    1. Step 顺序：Step 的执行顺序很重要，一般遵循 查询处理 → 检索 → 优化 → 结果 的顺序。

    2. 状态共享：所有 Step 共享同一个状态对象，确保数据正确传递。

    3. 错误处理：可以在 Pipeline 中添加错误处理 Step，或者使用 Fallback Step 处理错误。

    4. 性能考虑：过多的 Step 会增加延迟，按需添加。

    5. 测试建议：复杂 Pipeline 建议编写集成测试，确保各 Step 正确协作。

    生产级辅助增强模块¶

    一个真实可用的 RAG 系统，不仅仅是检索和生成，更需要应对高并发、网络抖动、LLM API 速率限制、成本控制以及评估追踪。GoRAG 提供了一套完整的辅助增强模组，基于 pipeline 的机制，这些模组被设计为非侵入式的中间件 (Middleware)、装饰器 (Decorator) 或 钩子 (Hooks)。

    查询阶段辅助模组¶

    1. 语义缓存 (Semantic Cache)¶

    • 痛点: 用户经常提问相似的问题（例如"你们的退款政策是什么？"和"怎么退款？"），每次都去调 Embedding 和 LLM 会造成巨大的延迟和成本浪费。
    • 机制: 拦截 Query，计算其 Embedding，并在本地内存/Redis 中查找相似度极高（如 Cosine > 0.98）的历史问题。如果命中，直接返回缓存的生成答案，整个查询耗时降至毫秒级。
    • 用法: 作为 Pipeline 的头部 Step，命中即阻断后续执行。

    2. LLM 熔断与优雅降级 (Circuit Breaker & Fallback)¶

    • 痛点: 外部大模型 API（如 GPT-4）可能由于网络波动、官方宕机导致长时间超时或报错。如果无脑等待，会拖垮整个系统的 Goroutine 池。
    • 机制: 引入熔断器。当 Generation Step 连续失败超过阈值时，触发熔断。此时系统优雅降级 (Fallback)：快速调用备用模型（如本地部署的 Llama-3 或云端廉价模型），或者返回友好的预设话术。

    3. 管线遥测与可观测性 (Telemetry & Observability)¶

    • 痛点: 在复杂的管线中，究竟是检索慢还是生成慢？消耗了多少 Token？
    • 机制: 利用 gochat/pipeline 现有的 Hook 机制（OnStepStart, OnStepEnd）。GoRAG 提供标准的 OpenTelemetryHook，它会自动捕获每一个 Step 的耗时、输入输出摘要和 Token 消耗，并将其暴露为 Prometheus 指标或推送到 Jaeger 链路追踪系统。

    索引阶段辅助模组¶

    1. 动态限流器 (Rate Limiter)¶

    • 痛点: 大规模并发解析文档时，瞬间向 Embedding 模型或 LLM 发送海量请求，极易触发 429 Too Many Requests 错误。
    • 机制: 在核心的 Worker 前挂载基于令牌桶 (Token Bucket) 的限流器。支持动态调整并发速率，确保请求平滑输出。

    2. 断点续传与去重缓存 (Checkpoint & Deduplication)¶

    • 痛点: 索引一个包含 1000 个 PDF 的目录，在中途崩溃了。重启后如果从头再来，成本巨大。此外，很多文档内容未修改但被重复索引。
    • 机制:
    • File Hash 去重: 在 Parser 之前，计算文件的 SHA256，对比 KV 状态表。如未变动，直接跳过。
    • Chunk Hash 去重: 在写入 VectorStore 前，对 Chunk 内容进行 Hash。如果已存在，则跳过 Embedding 步骤。

    装饰器模式使用¶

    GoRAG 提倡使用装饰器模式 (Decorator Pattern)，在不修改原有核心 Step 代码的前提下，赋予其高可用特性：

    // 1. 实例化核心组件
baseGenStep := steps.NewGenerationStep(llmClient)

// 2. 为 Generation Step 穿上"熔断降级"护甲
safeGenStep := resilience.WithCircuitBreakerAndFallback(
    baseGenStep, 
    fallbackClient,
    resilience.BreakerOptions{ErrorThreshold: 3, Timeout: time.Second * 10},
)

// 3. 实例化 Embedding 步骤，并穿上"限流"护甲
limitedEmbedStep := resilience.WithRateLimiter(
    steps.NewEmbeddingStep(embedder),
    rate.Limit(50), 
)

// 4. 组装坚不可摧的生产级管线
pipe := pipeline.NewPipeline()

// 挂载全局链路追踪 Hook
pipe.AddHook(telemetry.NewOTelPipelineHook("production-query-pipeline"))

pipe.AddSteps(
    steps.NewSemanticCacheStep(redisCache, 0.98), // [头部挂载] 语义缓存
    limitedEmbedStep,                             // [核心流程] 带限流的向量化
    steps.NewVectorSearchStep(store),
    safeGenStep,                                  // 带熔断降级的生成
)
