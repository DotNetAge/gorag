    依赖注入扩展¶

    本指南面向希望在 GoRAG 中扩展依赖注入（DI）系统的开发者，目标是帮助你： - 理解 DI 容器的架构与生命周期管理 - 掌握服务注册、替换与装饰器模式的实现方式 - 学会扩展容器以支持新的服务类型与依赖关系 - 了解最佳实践与性能优化技巧，确保扩展的兼容性与稳定性

    GoRAG 的 DI 容器采用"接口 + 工厂函数"的轻量设计，支持单例与瞬态两种生命周期，并提供全局默认容器与选项化构建能力，便于在不同模块中统一管理依赖。最新版本增强了资源池化和生命周期管理功能，支持IoC容器的完整生命周期管理。

    项目结构¶

    围绕 DI 扩展的相关文件与职责如下： - pkg/di/container.go：DI 容器主体，提供注册、解析、生命周期管理等核心能力 - pkg/di/options.go：容器选项（WithSingleton、WithTransient、WithInstance、NewWithOptions） - pkg/di/container_test.go：对容器行为的单元测试，覆盖注册、解析、生命周期、错误处理等 - pkg/core/interfaces.go：核心接口定义，用于指导服务注册与替换 - pkg/resilience/decorators.go：装饰器模式示例（限于 pipeline 步骤包装），可借鉴到 DI 场景 - gorag.go：应用入口与预设构建逻辑，展示如何在业务层组织服务

    核心组件¶

    • Container：容器主体，内部以接口类型为键存储服务元数据（工厂函数、是否单例、是否已初始化）
    • service：服务元数据，包含实例、工厂、单例标记与初始化状态
    • 全局默认容器：通过 Default() 提供单例容器，配合 Register/Resolve 等便捷函数
    • Options：以函数式选项的方式批量注册服务，支持单例、瞬态与已有实例
    • 新增：Close方法支持IoC容器的完整生命周期管理，自动关闭实现了io.Closer接口的单例服务

    关键能力与特性： - 注册阶段校验接口类型，支持指针到接口类型的自动解包 - 解析阶段区分单例与瞬态；单例采用双检锁与懒初始化 - 提供类型安全的解析与断言工具（ResolveTyped、MustResolveTyped） - 支持批量注册 RegisterServices 与 RegisterDefaultServices - 提供 Clear 清空与 IsRegistered 检查 - 新增：支持IoC容器生命周期管理，自动关闭实现了io.Closer接口的单例服务 - 新增：泛型注册和解析方法，提供更好的类型安全

    详细组件分析¶

    容器类与生命周期¶

    • 类型与字段
    • Container：持有服务映射与读写锁
    • service：封装实例、工厂、单例标记与初始化状态
    • 关键流程
    • Register：注册接口与工厂，记录 isSingleton
    • RegisterInstance：注册现有实例，标记 isSingleton=true 且 isInitialized=true
    • Resolve：根据接口类型查找服务；单例需懒初始化并支持双检锁；瞬态每次调用工厂
    • ResolveTyped/MustResolveTyped：类型安全解析与断言
    • RegisterServices：批量注册
    • Clear/IsRegistered：清空与查询
    • 新增：Close：关闭所有实现了io.Closer接口的单例服务

    选项化注册与全局默认容器¶

    • Options
    • WithSingleton/WithTransient：以 Option 形式注册单例或瞬态
    • WithInstance：注册已有实例
    • NewWithOptions：一次性应用多个 Option 创建容器
    • 全局默认容器
    • Default：延迟初始化全局容器
    • Register/Resolve/IsRegistered/Clear/RegisterServices 等便捷函数均委托给默认容器
    • 新增：RegisterDefaultSingletonInstance：全局默认容器的泛型单例实例注册
    flowchart TD Start(["开始"]) --> NewOpt["NewWithOptions(opts...)"] NewOpt --> Apply["逐个应用 Option 到容器"] Apply --> Done(["容器就绪"])

    自定义服务注册与配置¶

    • 注册单例服务
    • 使用 RegisterSingleton 或 WithSingleton
    • 工厂函数负责创建并可能依赖其他服务（通过容器参数）
    • 注册瞬态服务
    • 使用 RegisterTransient 或 WithTransient
    • 每次解析都会重新创建实例
    • 注册已有实例
    • 使用 RegisterInstance 或 WithInstance
    • 适用于外部库实例或测试桩
    • 批量注册
    • 使用 RegisterServices 或 RegisterDefaultServices
    • 将多个服务注册函数组合为一组
    • 新增：RegisterSingletonInstance：泛型单例实例注册
    • 新增：RegisterDefaultSingletonInstance：全局默认容器的泛型单例实例注册

    服务替换与装饰器模式¶

    • 服务替换
    • 在同一接口上重复注册会覆盖旧实现（以最后注册为准）
    • 建议在应用启动时集中注册，避免运行时动态替换导致不可控
    • 装饰器模式
    • 可参考 resilience/decorators.go 中的 RateLimiterStepWrapper 与 CircuitBreakerStepWrapper
    • 在 DI 中可将真实服务作为被装饰对象，通过工厂函数返回包装后的实例
    • 注意：装饰器应保持接口不变，避免破坏容器解析契约
    sequenceDiagram participant Client as "客户端" participant DI as "DI 容器" participant Real as "真实服务" participant Wrap as "装饰器包装" Client->>DI : "Resolve(接口)" DI->>Real : "创建真实服务" DI->>Wrap : "用装饰器包装真实服务" DI-->>Client : "返回包装后的实例"

    生命周期管理与并发安全¶

    • 单例懒加载与双检锁
    • 首次解析单例时，先读锁检查，再加写锁进行初始化，再次检查防止重复初始化
    • 并发控制
    • 读多写少场景使用 RWMutex，保证解析高并发下的性能
    • 错误处理
    • 工厂函数返回 error 时，解析阶段会传播错误
    • MustResolve/MustResolveTyped 失败时会 panic，适合启动期严格校验
    • 新增：IoC容器生命周期管理
    • Close方法自动关闭实现了io.Closer接口的所有单例服务
    • 支持多错误聚合，返回详细的关闭失败信息
    • 仅关闭已初始化的单例服务

    应用层集成示例（从 gorag.go 观察）¶

    • 应用层通过 RAGConfig 与多种模式（native/advanced/agentic/graph）组合服务
    • DI 容器可用于集中注册这些服务，替代或补充当前直接构造方式
    • 建议将 VectorStore、DocStore、Retriever 等抽象为接口，通过 DI 注入
    • 新增：利用Close方法实现资源池化和生命周期管理

    依赖关系分析¶

    • 内部耦合
    • Container 仅依赖标准库（反射、同步原语），内聚度高
    • Options 与 Container 为组合关系，通过函数式选项降低耦合
    • 对外依赖
    • 通过接口隔离对外实现细节，便于替换与测试
    • 循环依赖风险
    • 工厂函数参数为容器本身，若不当使用可能导致循环依赖；建议通过接口分层与延迟初始化规避
    • 新增：IoC容器生命周期管理
    • 依赖io.Closer接口，支持资源池化和自动清理

    性能考量¶

    • 单例懒加载与双检锁
    • 减少不必要的初始化开销，提升并发解析性能
    • 读写锁策略
    • 读多写少场景下，RWMutex 提升吞吐
    • 工厂函数幂等性
    • 工厂应尽量无副作用，避免在解析过程中产生不可预期的资源竞争
    • 批量注册
    • 使用 RegisterServices/ RegisterDefaultServices 合并多次注册，减少锁竞争
    • 类型安全解析
    • ResolveTyped/MustResolveTyped 可减少类型断言成本与错误路径
    • 新增：IoC容器生命周期管理性能
    • Close方法采用批量关闭策略，避免逐个检查的开销
    • 仅对已初始化的单例服务执行关闭操作

    故障排查指南¶

    常见问题与定位要点： - 注册接口类型错误 - Register/ RegisterInstance 期望接口或指针到接口；否则触发 panic - RegisterInstance 实例必须实现对应接口，否则触发 panic - 解析失败 - Resolve 返回错误：接口类型非法、未注册、工厂返回错误 - MustResolve/MustResolveTyped 失败会 panic，适合启动期严格校验 - 生命周期异常 - 单例未初始化或被意外 Clear 导致后续解析失败 - 新增：Close方法返回的多错误聚合信息，便于定位具体失败的服务 - 测试环境 - ResetForTesting 仅用于测试，生产禁止调用 - 新增：IoC容器生命周期管理问题 - 关闭失败的服务会累积到错误列表中 - 仅实现了io.Closer接口的服务才会被自动关闭

    结论¶

    GoRAG 的 DI 容器提供了简洁而强大的依赖管理能力：以接口为中心的服务注册、单例与瞬态生命周期、类型安全解析与全局默认容器，使得在复杂业务场景中也能保持清晰的依赖关系与良好的可维护性。最新版本增强了IoC容器的完整生命周期管理功能，支持资源池化和自动清理，通过Close方法统一管理实现了io.Closer接口的单例服务。通过选项化注册、批量注册与装饰器模式，可以进一步增强扩展性与可测试性。建议在应用层以接口抽象为核心，结合 DI 容器实现松耦合、可替换的服务体系。

    附录¶

    最佳实践清单¶

    • 明确接口边界：所有服务以接口暴露，避免直接依赖具体实现
    • 启动期严格校验：使用 MustResolveTyped/MustResolve 在应用启动阶段快速发现配置错误
    • 单例优先：共享昂贵资源（如数据库连接、向量库）使用单例
    • 瞬态隔离：需要隔离状态或线程不安全的组件使用瞬态
    • 装饰器透明：装饰器包装后仍保持接口一致，避免破坏容器契约
    • 批量注册：将相关服务聚合为注册函数，使用 RegisterServices/ RegisterDefaultServices
    • 测试隔离：使用 ResetForTesting 清理全局状态，确保测试互不影响
    • 新增：资源池化：实现io.Closer接口的服务会在应用关闭时自动清理
    • 新增：生命周期管理：合理使用Close方法管理IoC容器的完整生命周期

    性能优化建议¶

    • 减少锁竞争：批量注册、避免频繁 Clear
    • 工厂幂等：工厂函数应无副作用，避免在解析时产生额外开销
    • 单例懒加载：充分利用容器的单例懒加载机制
    • 类型安全：优先使用 ResolveTyped/MustResolveTyped，减少运行时断言
    • 新增：IoC容器优化：合理设计服务的生命周期，避免在Close中执行耗时操作
    • 新增：资源池化优化：对于昂贵的资源，考虑使用单例模式配合Close方法统一管理

    IoC容器生命周期管理指南¶

    • 实现io.Closer接口：对于需要清理的资源（数据库连接、文件句柄等），实现Close方法
    • 单例注册：将实现了Close方法的服务注册为单例，以便容器统一管理
    • 应用关闭：在应用退出时调用container.Close()，容器会自动关闭所有实现了io.Closer接口的单例服务
    • 错误处理：Close方法返回的错误列表包含了所有关闭失败的服务信息，便于诊断问题
    • 资源池化：通过单例模式实现资源池化，避免重复创建昂贵的资源
