    代理和工具系统¶

    本指南围绕 GoRAG 的"代理与工具系统"，系统阐述以下主题： - Agent 接口的设计理念与实现模式：包括 AgentActionType、AgentAction 的数据结构与职责边界。 - 工具（Tool）开发的完整示例：如何创建自定义工具与代理行为，以及工具输入输出规范。 - 代理推理与决策机制：AgentReasoner 与 AgentActionSelector 的职责与协作方式。 - 代理系统的配置与调度机制：通过 RAG 预设与 DI 容器进行模块化装配。 - 代理工具的测试方法与集成策略：确保代理系统的可靠性与可扩展性。

    项目结构¶

    GoRAG 采用分层与模块化的组织方式： - 核心抽象与类型位于 pkg/core，定义了检索、生成、意图分类、工具与代理等接口与数据结构。 - 代理与工具位于 pkg/core/agent，提供 Agent 与 Tool 的抽象及数据模型。 - 调度与执行通过 pkg/retriever/agentic 将 Agent 注入到检索管线中，实现"代理增强型检索"。 - 依赖注入容器位于 pkg/di，提供全局默认容器与注册/解析能力，便于在运行时装配服务。 - 应用入口与预设配置位于 gorag.go，提供多种 RAG 模式（Native、Advanced、Agentic、Graph）的工厂函数。

    graph TB subgraph "应用层" APP["gorag.go<br/>RAG 接口与预设工厂"] end subgraph "核心抽象" CORE_TYPES["types.go<br/>查询/结果/评估等类型"] CORE_IF["interfaces.go<br/>检索/生成/意图/代理接口"] end subgraph "代理与工具" AGENT_IF["agent.go<br/>Agent 接口与响应结构"] TOOL_IF["tool.go<br/>Tool 接口与结果结构"] end subgraph "调度与执行" AGENTIC_RET["retriever.go<br/>Agentic 检索器与代理步骤"] end subgraph "依赖注入" DI["container.go<br/>DI 容器与默认实例"] end APP --> AGENTIC_RET AGENTIC_RET --> AGENT_IF AGENT_IF --> TOOL_IF CORE_IF --> AGENTIC_RET CORE_TYPES --> AGENTIC_RET DI --> APP

    核心组件¶

    • Agent 接口：定义代理的名称、添加工具、聊天对话（ReAct 循环）、记忆存储等能力。
    • Tool 接口：定义工具的名称、描述、输入模式（JSON Schema）、调用逻辑与结果封装。
    • AgentActionType 与 AgentAction：定义代理可执行的动作类型与动作载体（如检索、反思、结束）。
    • AgentReasoner 与 AgentActionSelector：分别负责"基于当前状态的推理"和"基于推理选择下一步动作"。

    组件详解¶

    Agent 接口与数据结构¶

    • AgentStep：记录中间思考、动作、动作输入与观察，用于可追踪的 ReAct 流程。
    • AgentResponse：封装最终回答与多步 AgentStep 列表，便于回放与审计。
    • Agent 接口：提供 Name、AddTool、Chat、Memory 等能力，支持动态扩展与记忆管理。
    classDiagram class Agent { +Name() string +AddTool(tool) +Chat(ctx, query, history) *AgentResponse +Memory() ChatMemory } class AgentStep { +string Thought +string Action +ToolInput ActionInput +string Observation } class AgentResponse { +string Response +[]AgentStep Steps } Agent --> AgentResponse : "返回" AgentResponse --> AgentStep : "包含"

    Tool 接口与工具开发¶

    • ToolInput：以 map[string]any 表达工具输入，通常由 LLM 生成 JSON Schema 并校验。
    • ToolResult：封装字符串结果、错误标记与元数据，统一工具输出格式。
    • Tool 接口：要求唯一名称、语义描述、输入模式与调用方法，便于与 LLM 协作。
    classDiagram class Tool { +Name() string +Description() string +InputSchema() map[string]any +Call(ctx, input) *ToolResult } class ToolResult { +string Result +bool IsError +map[string]any Metadata } Tool --> ToolResult : "返回"

    代理动作与推理决策¶

    • AgentActionType：定义可执行动作类型（检索、反思、结束），作为代理决策的原子指令。
    • AgentAction：承载动作类型与必要参数（如检索动作对应的查询文本）。
    • AgentReasoner：根据当前检索状态（查询、已检索块、已有答案）进行推理，输出指导性思考。
    • AgentActionSelector：基于推理结果选择下一步动作，形成闭环的 ReAct 决策。
    flowchart TD Start(["开始"]) --> Reason["AgentReasoner 推理"] Reason --> Select["AgentActionSelector 选择动作"] Select --> Decision{"动作类型"} Decision --> |检索| Retrieve["执行检索动作"] Decision --> |反思| Reflect["执行反思动作"] Decision --> |结束| Finish["生成最终答案"] Retrieve --> LoopBack["回到推理阶段"] Reflect --> LoopBack Finish --> End(["结束"])

    代理执行与检索管线集成¶

    • Agentic 检索器在执行管线中插入"代理执行步骤"，在每次检索前或后触发代理的 Chat，收集多步思考与工具调用结果。
    • 执行完成后，将代理步骤写入检索上下文的自定义字段，便于后续分析与溯源。
    sequenceDiagram participant Pipeline as "检索管线" participant Step as "代理执行步骤" participant Agent as "Agent 接口" participant Ctx as "检索上下文" Pipeline->>Step : "Execute(Ctx)" Step->>Agent : "Chat(Ctx.Query.Text, Ctx.Agentic.History)" Agent-->>Step : "AgentResponse{Response, Steps}" Step->>Ctx : "写入 Answer 与 Agent 步骤" Step-->>Pipeline : "完成"

    配置与调度机制¶

    • RAG 预设工厂：提供 DefaultNativeRAG、DefaultAdvancedRAG、DefaultAgenticRAG、DefaultGraphRAG 等便捷入口，内部按模式装配索引器与检索器。
    • DI 容器：提供全局默认容器，支持注册单例/瞬态服务、实例注册、类型安全解析与重置，便于在不同模式下注入具体实现。
    graph TB CFG["RAGConfig<br/>工作目录/向量库/维度/模型/TopK"] Factory["buildRAG(mode)<br/>装配索引器与检索器"] DI["Container<br/>Register/Resolve/Reset"] CFG --> Factory Factory --> DI DI --> Factory

    测试方法与集成策略¶

    • 单元测试：通过 testify mock 对 Agent 进行行为模拟，验证检索器在调用代理后的结果与元数据是否正确写入。
    • DI 容器测试：覆盖注册、解析、单例/瞬态、实例注册、类型安全解析与清理等场景，确保容器在集成测试中的稳定性。
    • 集成策略：在测试中使用 ResetForTesting 清理全局容器状态，避免跨用例污染；在生产环境禁止调用该函数。

    依赖关系分析¶

    • Agent 与 Tool：Agent 通过 AddTool 注入 Tool，Chat 中可多次调用 Tool，形成"计划-执行-观察"的 ReAct 循环。
    • 检索器与代理：Agentic 检索器在管线中调用 Agent，将代理步骤写入检索上下文，实现"代理增强型检索"。
    • 类型与接口：types.go 提供查询、结果、评估等基础类型；interfaces.go 提供检索、生成、意图、代理等接口，支撑上层组合。
    graph LR AgentIF["Agent 接口"] --> ToolIF["Tool 接口"] AgentIF --> Types["types.go 类型"] ToolIF --> Types AgenticRet["Agentic 检索器"] --> AgentIF AgenticRet --> Types AgenticRet --> IF["interfaces.go 接口"]

    性能考量¶

    • 代理循环的开销：每次 Chat 会触发一次或多轮工具调用，应控制工具数量与复杂度，避免过长的推理链。
    • 记忆与历史：历史消息会随请求传递，建议限制历史长度或进行上下文裁剪，降低 LLM 输入成本。
    • 向量化与检索：TopK 与维度直接影响检索性能，需结合业务场景权衡召回与延迟。
    • 并发与缓存：在工具实现中可引入缓存与并发控制，减少重复计算与外部依赖抖动。
