    自定义管道步骤¶

    本指南面向希望为 GoRAG 开发"自定义管道步骤"的工程师，系统讲解步骤接口规范、实现模式、数据在步骤间的传递机制、错误处理策略、步骤注册与配置方法，以及与现有步骤的集成方式。文档覆盖从简单的文件发现到复杂的向量搜索与检索增强生成的完整流程，并提供性能优化建议与最佳实践，确保新步骤与核心系统的兼容性。

    项目结构¶

    GoRAG 的管道步骤主要位于 pkg/steps 下，按功能域拆分（如 indexing、vector、rerank、fuse、prune、generate 等）。每个步骤包内通常包含若干具体步骤，均实现统一的 Step 接口，接收强类型上下文（IndexingContext 或 RetrievalContext），并在 Execute 中完成业务逻辑。

    graph TB subgraph "索引管线步骤" D["discover.go<br/>文件发现"] P["parse.go<br/>多解析器路由"] C["chunk.go<br/>语义分块"] end subgraph "检索/生成管线步骤" VS["vector/search.go<br/>向量搜索"] F["filter/filter.go<br/>过滤条件提取"] R["rerank/score.go<br/>交叉编码重排"] RR["fuse/rrf.go<br/>RRF 融合"] PR["prune/compress.go<br/>上下文压缩"] G["generate/generate.go<br/>答案生成"] SC["cache/cache.go<br/>语义缓存"] end D --> P --> C VS --> F --> R --> RR --> PR --> G SC -.-> G

    核心组件¶

    • 步骤接口与上下文
    • 步骤实现统一接口，接收强类型上下文并返回 error。上下文分为两类：
      • IndexingContext：用于索引管线，承载文件路径，元数据、文档/块通道、向量等。
      • RetrievalContext：用于检索与生成管线，承载查询、检索结果、并行结果、过滤条件、代理状态、最终答案等。

    • 关键类型与接口参见：pkg/core/state.go、pkg/core/types.go、pkg/core/interfaces.go

    • 步骤注册与配置

    • 步骤通过工厂函数创建（如 Discover、Multi、Chunk、Search、Score、RRF、Compress、Generate、Check、Store 等），并以 AddStep/AddSteps 挂载到管道。
    • 索引管线示例参考：pkg/indexer/README.md

    详细组件分析¶

    文件发现步骤（Discover）¶

    • 功能：校验文件路径有效性，填充基础元数据（来源、文件名、大小、修改时间）。
    • 关键点：
    • 名称固定，便于日志与可观测性。
    • 错误处理明确，非文件路径直接报错。
    • 数据传递：更新 IndexingContext.Metadata，供后续解析步骤使用。
    flowchart TD Start(["开始"]) --> Stat["检查文件是否存在且为文件"] Stat --> |否| Err["返回错误"] Stat --> |是| Fill["填充元数据到 IndexingContext"] Fill --> End(["结束"]) Err --> End

    多解析器路由步骤（Parse）¶

    • 功能：根据文件扩展名选择合适解析器，流式解析并输出文档通道。
    • 关键点：
    • 解析器集合通过工厂函数注入。
    • 通过 ParseStream 获取文档通道，供下游步骤消费。
    • 数据传递：设置 IndexingContext.Documents 通道。
    flowchart TD Start(["开始"]) --> Sel["根据扩展名选择解析器"] Sel --> Open["打开文件流"] Open --> Stream["调用 ParseStream 输出文档通道"] Stream --> Set["设置 IndexingContext.Documents"] Set --> End(["结束"])

    语义分块步骤（Chunk）¶

    • 功能：对文档通道中的所有文档进行语义分块，输出块通道。
    • 关键点：
    • 依赖 Chunker 接口，遍历文档并收集所有块。
    • 通过通道传递，避免一次性加载全部块。
    • 数据传递：设置 IndexingContext.Chunks 通道。
    flowchart TD Start(["开始"]) --> Read["读取文档通道"] Read --> Loop{"还有文档？"} Loop --> |是| Chunk["调用 Chunker 分块"] Chunk --> Acc["累积所有块"] Acc --> Loop Loop --> |否| Chan["创建块通道并关闭"] Chan --> Set["设置 IndexingContext.Chunks"] Set --> End(["结束"])

    向量搜索步骤（VectorSearch）¶

    • 功能：根据查询文本生成嵌入向量，在向量存储中搜索 TopK 结果，将结果转换为 Chunk 并追加到 RetrievedChunks。
    • 关键点：
    • 支持多种查询来源：HyDE 假设文档、融合子查询、Step-back 查询等。
    • 可配置 Filters 传入向量存储搜索。
    • 通过 Tracer 记录事件，便于可观测性。
    • 数据传递：更新 RetrievalContext.RetrievedChunks。
    flowchart TD Start(["开始"]) --> QSel["选择查询序列HyDE/融合/Step-back/默认"] QSel --> Emb["生成查询嵌入"] Emb --> Search["向量存储搜索"] Search --> Conv["转换为 Chunk 列表"] Conv --> Append["追加到 RetrievedChunks"] Append --> End(["结束"])

    过滤条件提取步骤（FilterFromQuery）¶

    • 功能：从自然语言查询中抽取结构化过滤条件，写入 Filters 与 AgenticContext。
    • 关键点：
    • 依赖 FilterExtractor 接口。
    • 若无 AgenticContext，自动创建并记录过滤信息。
    flowchart TD Start(["开始"]) --> Ext["调用 Extract 获取过滤条件"] Ext --> Set["写入 state.Filters 与 Agentic.Filters"] Set --> End(["结束"])

    重排步骤（Rerank）¶

    • 功能：使用交叉编码器对检索到的片段进行再打分与重排，保留 TopK。
    • 关键点：
    • 扁平化多路检索结果，调用 Reranker 接口。
    • 记录指标与日志，便于监控。
    flowchart TD Start(["开始"]) --> Flat["扁平化多路结果"] Flat --> Empty{"有片段？"} Empty --> |否| End(["结束"]) Empty --> |是| ReRank["调用 Reranker 重排"] ReRank --> Set["更新 RetrievedChunks"] Set --> End

    RRF 融合步骤（RRF-Fusion）¶

    • 功能：对多路召回结果执行 Reciprocal Rank Fusion，得到单一融合结果集。
    • 关键点：
    • 调用 FusionEngine.ReciprocalRankFusion。
    • 清空原有结果，替换为融合后的结果。
    flowchart TD Start(["开始"]) --> Fuse["调用 ReciprocalRankFusion"] Fuse --> Replace["替换 RetrievedChunks 为融合结果"] Replace --> End(["结束"])

    上下文压缩步骤（ContextCompression）¶

    • 功能：将检索到的片段压缩为更精炼的上下文，交由 LLM 生成最终答案。
    • 关键点：
    • 构造提示词，限制最大 token 数。
    • 更新 Answer 字段。
    flowchart TD Start(["开始"]) --> Flat["扁平化多路结果"] Flat --> Prompt["构造压缩提示词"] Prompt --> Call["调用 LLM 生成压缩内容"] Call --> Update["更新 Answer"] Update --> End(["结束"])

    答案生成步骤（Generator）¶

    • 功能：将查询与检索到的片段交给 Generator 接口生成最终答案。
    • 关键点：
    • 扁平化 RetrievedChunks，调用 Generate。
    • 更新 Answer 字段。
    flowchart TD Start(["开始"]) --> Flat["扁平化 RetrievedChunks"] Flat --> Gen["调用 Generator.Generate"] Gen --> Set["更新 Answer"] Set --> End(["结束"])

    语义缓存步骤（SemanticCacheCheck/Store）¶

    • 功能：在检索前检查缓存命中，命中则短路返回答案；在生成后写入缓存。
    • 关键点：
    • Check：若命中，设置 Agentic.CacheHit 并填充 Answer。
    • Store：仅当 Query 与 Answer 均存在时写入。
    flowchart TD Start(["开始"]) --> Check["CheckCache"] Check --> Hit{"命中？"} Hit --> |是| SetAns["设置 Answer 并标记 CacheHit"] Hit --> |否| Next["继续后续步骤"] SetAns --> End(["结束"]) Next --> End

    依赖分析¶

    • 步骤与核心接口
    • 各步骤通过依赖注入的方式持有核心接口（如 VectorStore、Parser、Reranker、Generator、FilterExtractor、Client 等），从而与具体实现解耦。

    • 示例接口定义参见：pkg/core/interfaces.go

    • 步骤与上下文

    • 索引管线：IndexingContext 通过通道传递文档与块，减少内存占用。

    • 检索管线：RetrievalContext 保存查询、检索结果、并行结果、过滤条件、代理状态与最终答案。

    • 容错与弹性

    • 提供通用包装器：限流（RateLimiterStepWrapper）、断路器（CircuitBreakerStepWrapper）。
    • 断路器支持回退步骤，提升系统韧性。
    classDiagram class IndexingContext { +Ctx +FilePath +Metadata +Documents +Chunks +Vectors +TotalChunks +Custom } class RetrievalContext { +Ctx +OriginalQuery +Query +RetrievedChunks +ParallelResults +RerankScores +Filters +Agentic +Answer +Custom +Metrics } class AgenticContext { +NextStep +Intent +History +Metadata +SubQueries +HypotheticalDocument +HydeApplied +StepBackQuery +CacheHit +Filters +Custom } IndexingContext --> AgenticContext : "组合" RetrievalContext --> AgenticContext : "组合"

    性能考虑¶

    • 使用通道传递中间产物（文档/块），避免一次性加载大量数据，降低内存峰值。
    • 在向量搜索前先提取过滤条件（FilterFromQuery），缩小搜索空间，提高检索效率。
    • 合理设置 TopK 与 Filters，减少后续重排与融合的成本。
    • 对生成步骤使用上下文压缩（ContextCompression），控制输入长度，提升响应速度与成本效益。
    • 使用断路器与限流包装器（CircuitBreakerStepWrapper、RateLimiterStepWrapper）保护下游服务，避免雪崩效应。
    • 记录指标（Metrics）与可观测性（Tracer/Span），持续监控各步骤耗时与成功率。

    故障排查指南¶

    • 常见错误来源
    • 输入缺失：如查询为空、无文档/块、未配置解析器或嵌入器等。
    • 外部依赖失败：向量存储搜索失败、解析器流式解析异常、生成器调用失败等。
    • 排查建议
    • 检查步骤名称与日志，定位具体失败步骤。
    • 验证上下文字段是否正确填充（Query、Documents、Chunks、Filters 等）。
    • 使用断路器包装关键步骤，启用回退逻辑，快速隔离问题。
    • 查看指标与追踪事件，确认瓶颈环节。

    结论¶

    通过遵循统一的步骤接口与强类型上下文约定，结合通道化数据流与可观测性设计，GoRAG 的管道步骤具备良好的扩展性与可维护性。开发者可以基于现有步骤模式快速实现自定义步骤，并通过断路器、限流等弹性设计保障系统稳定性。建议在实现新步骤时，严格遵循错误处理与日志记录规范，确保与现有生态无缝集成。
