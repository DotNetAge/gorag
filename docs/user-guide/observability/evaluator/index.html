    评估器实现¶

    本文件面向GoRAG评估器实现，聚焦于LLMJudge接口与RagasLLMJudge的具体实现，系统阐述以下内容： - LLMJudge接口设计与职责边界 - RagasLLMJudge的实现原理与提示词模板设计 - 三大评估指标的计算方法与评分标准： - 忠实度评估（EvaluateFaithfulness） - 答案相关性评估（EvaluateAnswerRelevance） - 上下文精准度评估（EvaluateContextPrecision） - 评估结果解析与处理细节 - 评估器配置选项、性能考虑与最佳实践 - 如何集成第三方评估模型与扩展自定义评估逻辑

    项目结构¶

    评估能力主要分布在以下模块： - 评估接口与运行框架：pkg/generation/evaluator - 评估步骤（流水线集成）：pkg/steps/generate/rag_eval.go - 核心类型与接口：pkg/core - 应用入口与预设：gorag.go

    核心组件¶

    • LLMJudge接口：定义了三个评估方法，分别用于评估答案的忠实度、答案与查询的相关性以及检索上下文的精准度。
    • RagasLLMJudge：基于标准RAGAS风格提示词模板，使用强模型作为评估器，输出分数与理由，并通过统一解析器提取结果。
    • 基准测试RunBenchmark：对给定测试集执行检索与生成，再调用LLMJudge进行三项评估，汇总平均分与耗时。
    • RAG评估器（RAGEvaluator）：在流水线中对生成答案进行整体质量评估，返回忠实度、相关性与总体分数。
    • CRAG评估器（CRAGEvaluator）：面向Corrective RAG的检索相关性评估，输出相关标签与分数。

    详细组件分析¶

    LLMJudge接口与RagasLLMJudge实现¶

    • 接口职责
    • EvaluateFaithfulness：判断答案是否严格基于检索到的上下文，避免幻觉。
    • EvaluateAnswerRelevance：判断答案是否直接回答用户问题。
    • EvaluateContextPrecision：判断检索到的上下文是否包含回答所需信息，且关键信息是否置顶。
    • 实现要点
    • 提示词模板：每个评估维度都有明确的评分标准与格式约束（Score: 0.0~1.0；Reason: 简要解释），便于统一解析。
    • 上下文拼接：buildContextText将多个块按顺序拼接，确保LLM能逐条核验。
    • 结果解析：parseEvalResponse按固定格式提取分数与理由，对非法分数默认为0.0，未提供理由时使用默认占位。
    classDiagram class LLMJudge { +EvaluateFaithfulness(ctx, query, chunks, answer) (float32, string, error) +EvaluateAnswerRelevance(ctx, query, answer) (float32, string, error) +EvaluateContextPrecision(ctx, query, chunks) (float32, string, error) } class RagasLLMJudge { -judgeLLM : chat.Client +EvaluateFaithfulness(...) +EvaluateAnswerRelevance(...) +EvaluateContextPrecision(...) -buildContextText(chunks) string -parseEvalResponse(prompt) (float32, string, error) } LLMJudge <|.. RagasLLMJudge

    评估指标计算方法与评分标准¶

    忠实度评估（EvaluateFaithfulness）¶

    • 计算思路：要求答案中的所有断言均可从上下文中直接推断；若包含外部知识或幻觉则为0.0；部分符合给中间值；完全符合为1.0。
    • 输入：query、chunks、answer
    • 输出：score ∈ [0.0, 1.0]，reason
    • 处理细节：上下文按块顺序拼接，提示词强制要求输出"Score:"和"Reason:"两行。
    flowchart TD Start(["开始"]) --> Build["拼接上下文文本"] Build --> Prompt["构造忠实度提示词"] Prompt --> CallLLM["调用Judge LLM"] CallLLM --> Parse["解析响应: 提取Score与Reason"] Parse --> End(["结束"])

    答案相关性评估（EvaluateAnswerRelevance）¶

    • 计算思路：答案是否直接、完整地回答查询；完全相关为1.0，完全无关为0.0。
    • 输入：query、answer
    • 输出：score ∈ [0.0, 1.0]，reason
    • 处理细节：提示词明确评分范围与输出格式。

    上下文精准度评估（EvaluateContextPrecision）¶

    • 计算思路：检索到的上下文是否包含回答所需的有用信息；最相关的信息是否置顶为1.0，完全无用为0.0。
    • 输入：query、chunks
    • 输出：score ∈ [0.0, 1.0]，reason
    • 处理细节：提示词强调"最相关信息是否在顶部"。

    评估结果解析与处理¶

    • 解析策略
    • 固定格式：Score: 浮点数；Reason: 文本
    • 容错处理：非法分数默认为0.0；未提供Reason时使用默认占位字符串
    • 调用链路
    • EvaluateXxx → 构造提示词 → 调用judgeLLM.Chat → parseEvalResponse → 返回score与reason
    flowchart TD A["收到LLM响应"] --> B["按行分割"] B --> C{"包含Score: ?"} C --> |是| D["尝试解析为float32"] D --> |成功| E["设置score"] D --> |失败| F["score=0.0"] C --> |否| G["继续扫描"] B --> H{"包含Reason: ?"} H --> |是| I["截取并设置reason"] H --> |否| J["保持默认reason"] E --> K["返回score与reason"] F --> K I --> K J --> K

    基准测试与结果聚合（RunBenchmark）¶

    • 流程概览：对每个测试用例执行检索与生成，随后依次调用三项评估，累计求和并计算平均值，记录每项用时与总耗时。
    • 关键数据结构：TestCase、CaseResult、BenchmarkResult
    • 性能关注：单次评估耗时与LLM调用次数成正比，建议批量与缓存结合优化。
    flowchart TD S(["开始RunBenchmark"]) --> Loop["遍历TestCase"] Loop --> Ret["Retriever.Retrieve(query, topK)"] Ret --> |成功| Eval1["EvaluateFaithfulness"] Ret --> |失败| Skip["跳过该用例"] Eval1 --> Eval2["EvaluateAnswerRelevance"] Eval2 --> Eval3["EvaluateContextPrecision"] Eval3 --> Acc["累加分数与耗时"] Acc --> Next["下一个用例"] Skip --> Next Next --> Done{"全部完成?"} Done --> |否| Loop Done --> |是| Avg["计算平均分与总耗时"] Avg --> R(["返回BenchmarkResult"])

    流水线集成（RAGEvaluation Step）¶

    • 作用：在检索后、生成前或生成后，对答案进行整体质量评估（忠实度与相关性），并将结果写入状态，便于可观测性与后续决策。
    • 关键点：构建上下文字符串、调用RAGEvaluator.Evaluate、记录日志与指标。
    sequenceDiagram participant P as "Pipeline" participant SE as "RAGEvaluation Step" participant EV as "RAGEvaluator" participant LOG as "Logger" participant MET as "Metrics" P->>SE : 执行步骤 SE->>SE : 构建上下文字符串 SE->>EV : Evaluate(query, answer, context) EV-->>SE : 返回RAGEvaluation SE->>LOG : 记录评估结果 SE->>MET : 记录评估指标 SE-->>P : 完成

    类型与接口支撑¶

    • 接口：Retriever、RAGEvaluator、CRAGEvaluator等，定义了评估器与系统其他组件的契约。
    • 类型：Query、RetrievalResult、RAGEvaluation、CRAGEvaluation等，承载评估输入与输出的数据结构。

    依赖分析¶

    • 评估器依赖gochat的chat.Client进行LLM调用，提示词由评估器内部构造。
    • 核心类型与接口位于pkg/core，被评估器与流水线步骤共同依赖。
    • 应用层通过gorag.go提供不同模式的RAG实例，评估可在其检索与生成流程中插入。
    graph LR J["RagasLLMJudge"] --> LC["chat.Client(LLM)"] RB["RunBenchmark"] --> J SE["RAGEvaluation Step"] --> RE["RAGEvaluator"] RE --> T["RAGEvaluation 类型"] RB --> RT["RetrievalResult 类型"] G["gorag.go 工厂"] --> RT

    性能考量¶

    • LLM调用成本
    • 三项评估每次调用一次LLM，建议控制测试集规模与topK，避免过度调用。
    • 可引入缓存机制，对相同输入（query、chunks、answer）的结果进行缓存复用。
    • 上下文长度
    • 上下文拼接会增加提示词长度，可能影响成本与延迟；可对chunks做裁剪或排序，保留最相关片段。
    • 并发与批处理
    • RunBenchmark当前逐个用例串行评估；如需加速，可在保证LLM速率限制前提下引入并发与批处理。
    • 日志与指标
    • 步骤中已记录评估结果与指标，建议结合观测系统进行长期监控与回归分析。

    故障排查指南¶

    • 提示词格式不符
    • 现象：解析不到Reason或Score为0.0
    • 排查：确认提示词严格输出"Score:"和"Reason:"两行；检查LLM输出是否被截断或过滤
    • LLM调用失败
    • 现象：评估报错
    • 排查：检查网络、鉴权与模型可用性；必要时增加重试与超时配置
    • 上下文为空
    • 现象：评估步骤跳过或返回空上下文
    • 排查：确认检索阶段chunks非空；检查拼接逻辑
    • 分数异常
    • 现象：非法分数导致score=0.0
    • 排查：检查LLM输出数值合法性；必要时在上游对输出做清洗

    结论¶

    • LLMJudge接口提供了清晰的评估抽象，RagasLLMJudge以标准提示词模板实现了三大核心指标的自动化评估。
    • RunBenchmark与RAGEvaluation Step分别覆盖离线基准测试与在线流水线评估场景，满足不同使用需求。
    • 通过统一的解析与容错策略，评估结果具备可复现性与可观测性；结合缓存与并发优化，可在生产环境中稳定落地。

    附录¶

    评估器配置选项与最佳实践¶

    • 配置选项
    • topK：影响上下文规模与评估成本，建议根据任务复杂度调整
    • 评测集规模：建议先小规模验证流程，再逐步扩大
    • 缓存策略：对重复输入启用缓存，降低LLM调用频次
    • 最佳实践
    • 明确评分阈值与告警策略，结合业务目标设定通过标准
    • 将评估结果纳入CI/CD，持续跟踪模型退化
    • 对提示词进行A/B测试，持续优化模板与理由生成

    集成第三方评估模型与自定义评估逻辑¶

    • 集成第三方模型
    • 通过替换chat.Client实现，即可接入任意兼容的LLM服务；无需修改评估器核心逻辑
    • 自定义评估逻辑
    • 可实现自定义的LLMJudge或RAGEvaluator，遵循现有接口与数据结构约定
    • 若需扩展指标，可在现有接口基础上新增方法，并在RunBenchmark与流水线步骤中集成
