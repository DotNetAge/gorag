    文本类解析器¶

    本文件聚焦 GoRAG 的文本类解析器，系统性阐述 Text、Markdown、Log 等纯文本格式解析器的实现机制与使用方法。内容涵盖： - 流式解析的工作原理与内存控制策略 - 字符编码与换行符处理 - 元数据提取与自动识别规则 - 实际使用示例（长文档、编码转换、性能优化） - 最佳实践与常见问题解决方案

    项目结构¶

    文本解析器位于 pkg/indexing/parser 下，按功能划分为多个子包，统一遵循流式解析接口与文档模型。

    graph TB subgraph "解析器基础" PBase["base/parser.go<br/>定义 Parser 接口"] PWrap["base/generic_wrapper.go<br/>通用流式包装器"] end subgraph "文本解析器" T["text/streaming_parser.go<br/>TextStreamParser"] M["markdown/streaming_parser.go<br/>MarkdownStreamParser"] L["log/streaming_parser.go<br/>Log Parser"] end D["core/document.go<br/>Document 结构"] PBase --> T PBase --> M PBase --> L PWrap --> T T --> D M --> D L --> D

    核心组件¶

    • 解析器接口：统一的 ParseStream、GetSupportedTypes 方法，确保所有解析器具备流式读取与类型识别能力。
    • 文档模型：Document 统一承载内容、元数据、来源与类型，便于后续分块与嵌入。
    • 流式包装器：用于将一次性读取的解析器桥接为流式接口，保持管道一致性。

    关键职责与契约： - 流式接口：ParseStream 返回文档通道，避免一次性加载大文件导致内存峰值。 - 类型识别：GetSupportedTypes 明确支持的扩展名或 MIME 类型。 - 元数据保留：解析器复制传入元数据，并注入解析器标识与分片索引等字段。

    架构总览¶

    文本解析器通过流式扫描器逐行读取输入，按策略累积内容并在阈值触发时产出 Document；元数据随文档传递，便于下游分块与检索。

    详细组件分析¶

    Text 文本解析器（TextStreamParser）¶

    • 设计目标：面向原始文本、Markdown、CSV、日志等纯文本格式的通用流式解析，确保 O(1) 内存占用。
    • 关键机制：
    • 使用 bufio.Scanner 逐行读取，避免一次性加载。
    • 以字符串构建器累积内容，达到阈值后产出单个 Document。
    • 支持通过上下文取消，保证资源及时释放。
    • 元数据：
    • 复制传入元数据，附加解析器名称与分片索引。
    • 从 metadata["source"] 获取来源，未提供时默认为 unknown。
    • 编码与换行：
    • 逐行读取天然兼容 UTF-8；换行符统一追加以保持结构完整性。
    • 使用建议：
    • 对超大日志文件，合理设置阈值以平衡吞吐与分片粒度。
    • 需要按主题切分时，可结合 Markdown 解析器使用。
    flowchart TD Start(["进入 ParseStream"]) --> Init["初始化通道/元数据/扫描器"] Init --> Loop{"扫描下一行"} Loop --> |成功| Append["追加行与换行符"] Append --> Check{"是否达到阈值?"} Check --> |是| Emit["构造 Document 并发送"] Emit --> Reset["清空缓冲/递增分片索引"] Reset --> Loop Check --> |否| Loop Loop --> |失败| Finalize{"缓冲是否为空?"} Finalize --> |否| EmitLast["发送最后部分"] Finalize --> |是| Close["关闭通道"] EmitLast --> Close

    Markdown 解析器（MarkdownStreamParser）¶

    • 设计目标：按标题层级切分 Markdown 文档，便于按章节检索。
    • 关键机制：
    • 通过 splitOnHeaderLevel 控制切分粒度（例如 1 表示 H1，2 表示 H2）。
    • 当遇到新标题前，若缓冲已有内容则立即产出上一部分。
    • 逐行扫描，保持 UTF-8 安全与换行一致性。
    • 元数据：
    • 复制传入元数据，附加解析器名称与分片索引。
    • 使用建议：
    • 合理设置标题层级以匹配文档结构。
    • 无标题文档将作为单一文档输出。
    flowchart TD Start(["进入 ParseStream"]) --> Init["计算标题前缀(#... )/初始化通道/元数据"] Init --> Scan{"扫描下一行"} Scan --> |成功| IsHeader{"是否命中新标题?"} IsHeader --> |是且有内容| Emit["发送当前缓冲为文档"] Emit --> Reset["清空缓冲/递增索引"] Reset --> Append["追加当前行与换行符"] IsHeader --> |否| Append Append --> Scan Scan --> |失败| FinalEmit{"缓冲是否为空?"} FinalEmit --> |否| EmitLast["发送最后文档"] FinalEmit --> |是| Close["关闭通道"]

    Log 日志解析器（Log Parser）¶

    • 设计目标：支持多种日志格式（Nginx、Apache、Syslog），并可自定义正则；提供分块与重叠策略。
    • 关键机制：
    • 逐行读取，跳过空白行。
    • 若指定格式，抽取结构化元信息并前置到内容中，便于检索。
    • 基于阈值与重叠窗口生成文档片段。
    • 支持自动格式识别（基于前几行特征）。
    • 元数据：
    • 固定包含 type、position、parser 字段；格式明确时附加 format。
    • 使用建议：
    • 对于未知格式，先调用 DetectFormat 进行识别。
    • 对于自定义格式，SetPattern 设置正则并切换为 Custom。
    • 合理设置 chunkSize 与 chunkOverlap 以平衡召回与噪声。

    文档模型与流式包装器¶

    • Document：统一承载内容、来源、类型与元数据，支持更新时间戳与内容变更。
    • 通用流式包装器：将一次性读取的解析器桥接为流式接口，便于在统一管道中使用。

    依赖分析¶

    • Text/Markdown/Log 解析器均实现统一的 Parser 接口，确保在索引管线中可互换使用。
    • Document 作为跨组件的数据载体，被所有解析器产出。
    • 通用包装器用于兼容一次性解析器，保持接口一致性。
    graph LR ParserIntf["Parser 接口"] --> TextP["TextStreamParser"] ParserIntf --> MarkdownP["MarkdownStreamParser"] ParserIntf --> LogP["Log Parser"] TextP --> Doc["Document"] MarkdownP --> Doc LogP --> Doc Wrap["GenericStreamWrapper"] --> ParserIntf

    性能考量¶

    • 内存复杂度：解析器采用流式扫描与固定缓冲策略，确保 O(1) 内存占用，适用于 GB 级日志文件。
    • 吞吐优化：
    • 合理设置阈值（如 Text 的最大读取字节数、Log 的分块大小与重叠）。
    • 使用上下文取消提前终止长时间任务。
    • 对超大文档优先使用分块器进行二次切分，避免单文档过大影响嵌入与检索。
    • 编码与换行：
    • 逐行读取天然适配 UTF-8；换行符统一追加，保证结构稳定。
    • I/O 优化：
    • 使用 bufio.Scanner 降低系统调用次数；Log 解析器显式增大扫描缓冲以提升大文件处理能力。

    故障排查指南¶

    • 上下文取消无效：
    • 确认在每次循环中检查 ctx.Done，避免阻塞。
    • 示例参考：Text/Markdown/Log 的 ParseStream 循环内均有 select <-ctx.Done 判断。
    • 输出通道未关闭：
    • 确保在 goroutine 退出前关闭通道；所有解析器均在 defer 中关闭输出通道。
    • 空文件或仅空白行：
    • Text/Markdown 在无内容时不会产生文档；Log 会跳过空白行，最终可能无输出。
    • 元数据丢失：
    • 确保传入的 metadata 被正确复制并附加；解析器会复制传入元数据并添加解析器标识与分片索引。
    • 格式识别失败：
    • 使用 Log 解析器的 DetectFormat 读取前几行进行判断；必要时手动 SetFormat 或 SetPattern。

    结论¶

    GoRAG 的文本类解析器以统一接口与流式设计为核心，兼顾性能与可维护性。Text/Markdown/Log 解析器分别针对不同场景提供灵活的切分策略与元数据提取能力。配合分块器与嵌入器，可在海量文本上实现高效索引与检索。

    最佳实践清单¶

    • 选择合适的解析器：
    • 纯文本/日志：TextStreamParser
    • Markdown 文档：MarkdownStreamParser
    • 结构化日志：Log Parser（自动识别或手动设置格式）
    • 合理配置阈值与重叠：
    • 根据文档长度与检索需求调整分块大小与重叠比例。
    • 元数据管理：
    • 通过 metadata 传递来源、作者、标签等信息，便于检索与溯源。
    • 编码与换行：
    • 默认 UTF-8 与换行符处理已满足大多数场景；如遇特殊格式，可在上游进行编码转换后再交给解析器。
