    数据类解析器¶

    本文件系统性梳理 GoRAG 中面向结构化数据（CSV、JSON、Excel、数据库 Schema）的“数据类解析器”实现，重点覆盖以下方面： - 流式解析与内存优化：通过通道驱动的流式处理，实现 O(1) 内存复杂度，适用于超大文件。 - 数据表结构提取与字段识别：对数据库 Schema 进行 CREATE TABLE/INDEX 的识别与分块输出。 - 关系抽取能力：提供基于提示词的关系三元组抽取工具，辅助构建知识图谱。 - 并发与上下文控制：解析过程支持 context 取消，避免资源泄漏。 - 格式转换与标准化：统一输出为文档对象，便于后续分块、嵌入与检索。 - 验证规则、错误恢复与性能调优：结合测试用例总结常见问题与优化建议。

    项目结构¶

    数据类解析器位于索引模块的解析子系统中，采用按格式分类的目录组织，并共享统一的解析接口与文档模型。

    核心组件¶

    • Parser 接口：定义 ParseStream、GetSupportedTypes 等方法，确保所有解析器遵循一致的流式协议。
    • Document 文档模型：统一承载内容、元数据、来源与类型信息，便于下游处理。
    • 通用包装器：将一次性读取的提取器桥接为流式 Parser，兼容历史解析器。
    • 三元组抽取器：基于 LLM 的关系抽取工具，用于从文本中提取实体与关系三元组。

    详细组件分析¶

    CSV 流解析器¶

    • 功能要点
    • 支持带/不带头的 CSV 输入，自动放宽字段数约束与懒加载引号。
    • 将多行打包为单个 Document，通过 rowsPerDocument 控制粒度。
    • 使用字符串构建器拼接结构化文本，保留列名与值的对应关系。
    • 支持上下文取消，遇到 EOF 或错误时优雅退出。
    • 元数据与输出
    • 输出文档类型为 text/csv，附加 parser、part_index 等元数据。
    • 错误与边界
    • 空文件或读取错误会提前结束；坏行被跳过以保证稳定性。
    • 性能与并发
    • 单协程流式读取，通道缓冲为 1，适合顺序消费。
    • 使用建议
    • 对超大 CSV，适当增大 rowsPerDocument 以减少文档数量。
    • 若需要字段级建模，可结合三元组抽取器进行关系挖掘。
    flowchart TD Start(["开始 ParseStream"]) --> Init["初始化 Reader/缓冲/计数"] Init --> Header{"是否含表头?"} Header --> |是| ReadHdr["读取表头"] Header --> |否| Loop["逐行读取记录"] ReadHdr --> Loop Loop --> Row["格式化为键值对文本"] Row --> Count{"达到 rowsPerDocument ?"} Count --> |是| Emit["构造 Document 并发送"] Count --> |否| Loop Emit --> Reset["重置缓冲/计数/索引"] Reset --> Loop Loop --> EOF{"EOF?"} EOF --> |是| Flush["剩余内容作为最后 Document 发送"] EOF --> |否| Loop Flush --> Done(["结束"])

    JSON 流解析器¶

    • 功能要点
    • 基于 Decoder 的 Token 流式遍历，边读边拼接，达到 chunkSize 后切分。
    • 支持 overlap，确保跨块语义连续性。
    • 可从上下文读取文件路径，写入文档元数据。
    • 支持上下文取消与 EOF 处理。
    • 元数据与输出
    • 输出文档类型为 application/json，附加 type=json、position、file_path 等。
    • 性能与并发
    • 单协程流式解码，通道缓冲为 10，兼顾吞吐与内存占用。
    • 使用建议
    • 对超大数组 JSON，合理设置 chunkSize 与 overlap。
    • 如需环境变量展开，可在上游预处理或启用扩展开关（当前默认关闭）。
    sequenceDiagram participant R as "Reader" participant D as "Decoder" participant P as "JSON 解析器" participant C as "输出通道" participant Doc as "Document" R->>D : "Token 流" loop "读取 Token" D-->>P : "下一个 Token" P->>P : "拼接至缓冲区" alt "缓冲长度 ≥ chunkSize" P->>Doc : "构造 Document(含位置/路径)" P-->>C : "发送 Document" P->>P : "保留 overlap 片段" end end D-->>P : "结束" P->>Doc : "构造最后 Document" P-->>C : "发送"

    Excel 流解析器¶

    • 功能要点
    • 使用第三方库打开 Excel，遍历所有工作表与行，按 chunkSize 切分。
    • 输出为纯文本，包含 Sheet 名与行号信息，便于检索定位。
    • 支持 overlap，保留跨块上下文。
    • 元数据与输出
    • 输出文档类型为 text/plain，附加 type=excel、sheet、position 等。
    • 性能与并发
    • 单协程顺序读取，通道缓冲为 1。
    • 使用建议
    • 对超大表格，适当增大 chunkSize 与 overlap。
    • 注意 Excel 打开失败时的容错处理（如非标准格式）。
    flowchart TD Open["打开 Excel 文件"] --> List["列举工作表"] List --> ForEachSheet{"遍历每个工作表"} ForEachSheet --> Rows["遍历行"] Rows --> Append["追加到缓冲区(含 Sheet/行号)"] Append --> Enough{"缓冲长度 ≥ chunkSize ?"} Enough --> |是| Emit["构造 Document 并发送"] Emit --> Keep["保留 overlap 片段"] Keep --> Rows Enough --> |否| Rows Rows --> ForEachSheet ForEachSheet --> Close["关闭文件"] Close --> Done["结束"]

    数据库 Schema 流解析器¶

    • 功能要点
    • 基于 Scanner 逐行扫描，识别 CREATE TABLE/INDEX 语句。
    • 对 CREATE TABLE 做括号匹配，完整捕获后作为一个 Document 输出；对 INDEX 单独输出。
    • 支持混合内容的通用分块，达到 chunkSize 后输出。
    • 可配置是否提取表、列、索引三类对象。
    • 元数据与输出
    • 输出文档类型为 text/plain，附加 type=dbschema、chunk_type(table/index/mixed)、table_name/object_name 等。
    • 性能与并发
    • 单协程扫描，通道缓冲为 1。
    • 使用建议
    • 对大型 DDL 文件，合理设置 chunkSize 与 overlap。
    • 提取索引时开启 extractIndexes，提取表结构时开启 extractTables。
    flowchart TD Scan["逐行扫描"] --> State{"是否处于 CREATE TABLE 体?"} State --> |否| Detect{"匹配 CREATE TABLE/INDEX ?"} Detect --> |TABLE| Enter["进入 TABLE 捕获状态<br/>记录表名/括号计数"] Detect --> |INDEX| EmitIdx["输出 INDEX 文档"] Detect --> |其他| Buffer["追加到通用缓冲"] Buffer --> Enough{"缓冲长度 ≥ chunkSize ?"} Enough --> |是| EmitMix["输出混合内容文档"] EmitMix --> Buffer Enter --> Collect["持续追加行/更新括号计数"] Collect --> EndTable{"括号归零?"} EndTable --> |是| EmitTbl["输出 TABLE 文档"] EmitTbl --> State Enough --> |否| Scan State --> |是| Collect

    配置文件流解析器¶

    • 功能要点
    • 通用配置文件解析（支持多种格式），按行读取并可选屏蔽敏感信息、展开环境变量。
    • 支持 chunkSize 与 overlap，逐行处理并按阈值切分。
    • 输出文档类型为 text/plain，附加 format、masked、env_expanded 等元数据。
    • 安全与合规
    • 默认屏蔽敏感信息（密码、密钥、令牌等），可通过配置开关控制。
    • 环境变量展开默认关闭，避免意外暴露。
    • 使用建议
    • 在生产环境谨慎开启环境变量展开，确保配置来源可信。
    • 对包含敏感信息的配置文件，务必启用 maskSecrets。

    通用包装器与三元组抽取器¶

    • 通用包装器
    • 将一次性读取的提取器桥接为流式 Parser，便于兼容旧解析器。
    • 输出为单个 Document，若内容过大可交由后续分块器处理。
    • 三元组抽取器
    • 基于 LLM 的提示词抽取实体关系三元组，输出 JSON 数组并清洗格式。
    • 可用于从结构化数据导出的关系中进一步挖掘隐含知识。

    依赖分析¶

    • 组件耦合
    • 所有解析器均实现统一的 Parser 接口，降低上层调用复杂度。
    • Document 作为跨模块的数据契约，解析器与下游处理链共享。
    • 外部依赖
    • Excel 解析依赖第三方库打开与遍历工作表。
    • JSON 解析依赖标准库 Decoder 实现流式解码。
    • 三元组抽取依赖外部 LLM 客户端（通过接口注入）。
    • 循环依赖
    • 解析器与核心模型之间为单向依赖，无循环风险。

    性能考虑¶

    • 流式内存复杂度
    • 所有解析器均采用流式/扫描策略，配合小容量通道，实现 O(1) 内存占用，适合处理超大文件。
    • 切分策略
    • CSV 通过 rowsPerDocument 聚合行；JSON/XLS/DDL 通过字节长度与 overlap 控制切分，兼顾语义连贯性。
    • 上下文与取消
    • 解析过程中频繁检查 context，支持快速取消，避免资源浪费。
    • 并发建议
    • 当前解析器为单协程实现；如需更高吞吐，可在上游对多个文件并行调度，下游再合并结果。
    • I/O 优化
    • JSON/DDL 使用较大的缓冲区，减少系统调用次数；Excel 预分配缓冲以降低扩容成本。

    故障排查指南¶

    • 常见问题与对策
    • CSV 读取异常：检查是否有不匹配字段数或特殊字符，解析器已放宽约束但仍可能跳过坏行。
    • JSON 解码失败：确认输入为合法 JSON；必要时先做预处理或启用更宽松的解码参数。
    • Excel 打不开：确认为标准 Excel 格式；非标准 XML 结构会导致解析失败。
    • DDL 不完整：CREATE TABLE 语句换行或注释可能影响括号匹配，建议保持简洁格式。
    • 配置文件敏感信息泄露：默认屏蔽敏感项，如仍出现请检查正则匹配与日志输出。
    • 验证与回归
    • 参考各解析器的单元测试，覆盖空文件、超大文件、上下文取消、元数据传递等场景。

    结论¶

    该数据类解析器体系以统一接口与文档模型为核心，围绕流式处理实现了对 CSV、JSON、Excel、数据库 Schema 以及通用配置文件的高效解析。通过合理的切分策略、上下文控制与元数据标注，既满足了大数据量场景下的内存与性能要求，也为后续的知识图谱与检索增强提供了坚实基础。建议在实际业务中结合数据特点选择合适的解析器与参数，并配合三元组抽取器进行关系挖掘与知识发现。
