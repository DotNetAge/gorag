    内置解析器实现¶

    本文件系统性梳理 GoRAG 内置解析器实现，覆盖 CSV、HTML、JSON、PDF、Text、Markdown 等格式的解析器设计与实现要点，重点说明其流式处理机制、元数据提取方式、错误处理策略、配置选项与性能优化技巧，并提供解析器选择策略与多格式文件处理的综合指南。

    项目结构¶

    内置解析器位于 pkg/indexing/parser 下，按格式分包组织；基础接口与通用桥接在 base 包中；配置型解析器与类型枚举在 config 与 config/types 中；核心文档模型在 pkg/core。

    核心组件¶

    • Parser 接口：统一的流式解析能力，要求 ParseStream 支持上下文取消与通道返回文档流，GetSupportedTypes 提供支持的扩展名或 MIME 类型。
    • 通用流式包装器：将“一次性读取全文”的内容提取器桥接为流式 Parser，便于兼容旧解析器。
    • 文档模型：Document 统一承载内容、元数据、来源与类型信息，作为解析结果的标准载体。

    架构总览¶

    解析器体系采用“接口统一 + 多格式实现 + 注册表管理”的架构。注册表默认初始化内置解析器集合，按类型动态选择解析器；流式解析器通过 goroutine + 通道 + 上下文取消实现高吞吐低内存占用。

    详细组件分析¶

    CSV 解析器（CSVStreamParser）¶

    • 功能特性
    • 基于标准库 CSV 读取器，支持宽松字段数与懒引号。
    • 可选是否包含表头；有表头时将键值对格式化为结构化文本。
    • 将 N 行打包为一个 Document，平衡粒度与内存占用。
    • 流式处理机制
    • 使用 goroutine 异步读取，按行累积到缓冲区，达到阈值后发送一个 Document。
    • 支持上下文取消，异常时安全退出。
    • 元数据提取
    • 复制传入 metadata，追加 parser 名称与 part_index。
    • 若存在 source，用于设置文档 Source 字段。
    • 错误处理策略
    • EOF 正常结束；读取错误跳过当前记录，继续处理后续行。
    • 读取空文件直接关闭通道。
    • 性能优化
    • RowsPerDocument 控制批量大小；HasHeader 影响首行处理逻辑。
    • 使用 strings.Builder 减少字符串拼接开销。
    • 配置选项
    • rowsPerDocument：每文档行数，默认 100。
    • hasHeader：是否包含表头，默认 true。
    flowchart TD Start(["开始 ParseStream"]) --> Init["初始化 reader 与缓冲区"] Init --> HeaderCheck{"是否包含表头?"} HeaderCheck --> |是| ReadHeader["读取首行作为 headers"] HeaderCheck --> |否| Loop["进入行循环"] ReadHeader --> Loop Loop --> ReadRow["读取一行记录"] ReadRow --> EOF{"EOF?"} EOF --> |是| Flush["刷新剩余内容为最后一个文档"] EOF --> |否| Format["格式化为结构化文本"] Format --> Count["计数 +1"] Count --> Threshold{"达到 rowsPerDocument ?"} Threshold --> |是| Emit["复制元数据并发送文档"] Emit --> Reset["重置缓冲区与计数"] Reset --> Loop Threshold --> |否| Loop Flush --> End(["结束"])

    HTML 解析器（两套实现）¶

    • 传统解析器（Parser）
    • 使用标准库 HTML tokenizer，支持移除 script/style 标签、提取链接、分块与重叠。
    • 通过回调逐块输出 Chunk，适合非流式场景。
    • 流式包装器（HtmlStreamParser）
    • 将传统解析器封装为流式 Parser，先完整解析为多个 Chunk，再合并为单个 Document 输出。
    • 保留文件路径等元数据，便于溯源。
    classDiagram class HtmlParser { +SetChunkSize(size) +SetChunkOverlap(overlap) +SetCleanScripts(clean) +SetCleanStyles(clean) +SetExtractLinks(extract) +Parse(ctx, r) []Chunk +ParseWithCallback(ctx, r, cb) error +SupportedFormats() []string } class HtmlStreamParser { -legacyParser : HtmlParser +GetSupportedTypes() []string +ParseStream(ctx, r, meta) <-chan *Document +SetChunkSize(size) +SetChunkOverlap(overlap) +SetCleanScripts(clean) +SetCleanStyles(clean) +SetExtractLinks(extract) +Parse(ctx, content, meta) *Document } HtmlStreamParser --> HtmlParser : "委托"

    JSON 解析器（JsonStreamParser）¶

    • 功能特性
    • 使用标准库 json.Decoder 进行流式令牌遍历，按字符长度阈值切分为多个 Document。
    • 支持文件路径注入到元数据，便于溯源。
    • 流式处理机制
    • 读取下一个 token 并序列化为字符串累加，达到阈值后发送文档并保留重叠部分。
    • 元数据提取
    • 追加 type、position、file_path 等字段。
    • 错误处理策略
    • decoder.More()/Token() 错误直接终止；EOF 正常结束。
    • 性能优化
    • 通过 chunkSize 与 chunkOverlap 调整吞吐与重叠成本。
    • 配置选项
    • SetChunkSize/SetChunkOverlap：控制切分粒度与重叠。
    flowchart TD Start(["开始 ParseStream"]) --> Init["创建 Decoder 与缓冲区"] Init --> ReadToken["读取下一个 token"] ReadToken --> More{"decoder.More() ?"} More --> |否| Finalize["处理最后 token 并发送"] More --> |是| Append["序列化 token 并追加到缓冲区"] Append --> Threshold{"缓冲区长度 >= chunkSize ?"} Threshold --> |是| Emit["构造 Document 并发送"] Emit --> Overlap["保留重叠部分并重置缓冲区"] Overlap --> ReadToken Threshold --> |否| ReadToken Finalize --> End(["结束"])

    PDF 解析器（pdf.Parser）¶

    • 功能特性
    • 基于第三方库读取 PDF，逐页提取纯文本，按页生成 Document。
    • 自动合并元数据（如页码、总页数）。
    • 流式处理机制
    • 读取全部输入以满足 ReaderAt 要求，逐页异步发送文档。
    • 元数据提取
    • 追加 page_number、total_pages、parser 标识。
    • 错误处理策略
    • 读取失败或创建 Reader 失败返回错误；空页或解析失败跳过。
    • 性能优化
    • 合理设置 chunkSize（页面级），避免大对象阻塞通道。
    • 配置选项
    • 默认 chunkSize=2000；可通过扩展增加参数。
    sequenceDiagram participant Client as "调用方" participant PDF as "pdf.Parser" participant Lib as "PDF 库" participant Out as "Document 通道" Client->>PDF : ParseStream(ctx, reader, metadata) PDF->>PDF : 读取全部输入 PDF->>Lib : 创建 PDF Reader loop 每一页 PDF->>Lib : Page(i).GetPlainText() Lib-->>PDF : 文本内容 PDF->>Out : NewDocument(..., page_meta) end PDF-->>Client : 通道关闭

    Text 解析器（TextStreamParser）¶

    • 功能特性
    • 真正的流式解析：按行扫描，累计到阈值后发送文档。
    • 支持多类型扩展名与 MIME 类型。
    • 流式处理机制
    • 使用 bufio.Scanner，按最大字节阈值触发发送。
    • 元数据提取
    • 追加 parser 名称与 part_index。
    • 错误处理策略
    • 扫描器 EOF 正常结束；上下文取消提前终止。
    • 性能优化
    • 默认 10MB 分块，适合日志等超大文件。
    flowchart TD Start(["开始 ParseStream"]) --> Scan["scanner.Scan() 循环"] Scan --> Line["读取一行并追加到缓冲区"] Line --> Threshold{"缓冲区长度 >= maxReadBytes ?"} Threshold --> |是| Emit["发送文档并重置缓冲区"] Emit --> Scan Threshold --> |否| Scan Scan --> EOF{"扫描器 EOF ?"} EOF --> |是| Flush["发送剩余内容作为最后文档"] Flush --> End(["结束"])

    Markdown 解析器（MarkdownStreamParser）¶

    • 功能特性
    • 按指定标题层级拆分文档，例如 H1/H2 等。
    • 以缓冲区累积内容，遇到新标题即发送上一段。
    • 流式处理机制
    • 使用 bufio.Scanner，检测前缀匹配触发发送。
    • 元数据提取
    • 追加 parser 名称与 part_index。
    • 错误处理策略
    • 上下文取消提前终止；空文件不产生文档。
    • 性能优化
    • splitOnHeaderLevel 控制拆分粒度，建议优先 H1。

    配置文件解析器（ConfigStreamParser）¶

    • 功能特性
    • 支持多种配置格式（.toml/.ini/.yaml/.env 等），按行流式处理。
    • 可选隐藏敏感信息与环境变量展开。
    • 流式处理机制
    • 逐行读取，按 chunkSize 切分，保留 chunkOverlap。
    • 元数据提取
    • 记录 format、masked、env_expanded、position/part_index。
    • 错误处理策略
    • 读取 EOF 正常结束；错误直接返回。
    • 性能优化
    • maskSecrets 与 expandEnv 可控安全与性能权衡。

    通用流式包装器（GenericStreamWrapper）¶

    • 功能特性
    • 将一次性读取全文的 ContentExtractor 包装为 Parser 接口，适配旧解析器。
    • 在 goroutine 中执行提取，完成后发送单个 Document。
    • 错误处理策略
    • 提取失败时关闭通道（不抛错），保持流式语义一致。
    • 元数据提取
    • 复制传入元数据并追加 parser 名称。

    三元组抽取器（TriplesExtractor）¶

    • 功能特性
    • 基于 LLM 的知识抽取，从文本中提取实体与关系三元组。
    • 清洗 LLM 输出中的 JSON 包裹标记，解析为结构化数组。
    • 错误处理策略
    • LLM 调用失败或 JSON 解析失败返回错误。

    依赖分析¶

    • 接口与实现解耦：Parser 接口统一了所有解析器的行为契约，具体实现可独立演进。
    • 注册表集中管理：DefaultRegistry 默认注册 20+ 内置解析器，按类型检索与复用。
    • 元数据传播：解析器统一将 parser 名称、类型、位置等元数据写入 Document，便于溯源与后处理。

    性能考虑¶

    • 流式优先：所有解析器均采用流式处理，避免一次性加载大文件导致内存峰值过高。
    • 分块与重叠：JSON、HTML、Text、Markdown、配置文件均支持 chunkSize 与 chunkOverlap，平衡吞吐与上下文连续性。
    • 上下文取消：在关键节点检查 ctx.Done()，确保长耗时操作可被及时中断。
    • 缓冲策略：使用 strings.Builder 或 bufio.Scanner，减少频繁分配与拷贝。
    • 元数据最小化：仅注入必要字段，避免额外序列化开销。

    故障排查指南¶

    • 通道未关闭或无输出
    • 检查输入是否为空或读取错误；确认上下文未提前取消。
    • 参考各解析器的 EOF/错误分支处理。
    • 文档内容为空
    • PDF 页面无文本、HTML 移除了 script/style 导致空白、Markdown 无标题等场景需注意。
    • 元数据缺失
    • 确认传入 metadata 是否包含 source 等关键字段；检查解析器是否正确复制与追加元数据。
    • 性能问题
    • 调整 chunkSize 与 chunkOverlap；对 Text/JSON/HTML/Markdown 适当增大阈值以降低通道发送频率。

    结论¶

    GoRAG 内置解析器以统一接口为核心，结合流式处理与元数据标准化，实现了对多格式文件的高效、可控与可扩展解析。通过注册表与配置型解析器，系统在保证性能的同时提供了灵活的配置与扩展能力。建议在实际应用中根据文件规模与业务需求选择合适的解析器与分块策略，并充分利用元数据进行溯源与后处理。

    附录：使用示例与最佳实践¶

    • 选择策略
    • 文本类（日志/纯文本/Markdown）：优先 Text/Markdown 解析器，按内容体量调整分块大小。
    • 结构化数据（CSV/JSON）：CSV/JSON 解析器，注意字段对齐与嵌套结构处理。
    • 文档类（PDF/HTML）：PDF 按页拆分，HTML 可选择保留正文或移除脚本样式。
    • 配置类（多格式）：ConfigStreamParser，启用敏感信息掩码与环境变量展开。
    • 最佳实践
    • 明确 source 与自定义元数据，便于检索与溯源。
    • 对超大文件设置合理的 chunkSize，避免通道拥塞。
    • 使用上下文取消控制长时间任务，提升可观测性。
    • 在生产环境开启错误日志与指标采集，配合单元测试覆盖边界场景。
