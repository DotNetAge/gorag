    文档类解析器¶

    本文件聚焦于GoRAG中“文档类解析器”的实现与使用，覆盖PDF、DOCX、PPTX、图片（视觉RAG）以及配置类文档（含敏感信息掩码、环境变量展开、自动格式识别）等复杂文档格式的解析流程。重点阐述以下方面： - 文档结构解析：如何从不同容器格式中抽取文本或结构化内容 - 页面/幻灯片级提取：按页或按幻灯片切分输出 - 流式处理机制：基于通道的O(1)内存模型，适配超大文档 - 内存管理与性能优化：缓冲区复用、重叠拼接、上下文取消 - 配置项与参数：页面范围、图像质量、掩码策略、环境变量展开等 - 使用示例与最佳实践：如何选择解析器、如何组合多解析器、如何在大规模索引中落地

    项目结构¶

    解析器位于pkg/indexing/parser目录下，按文件类型划分子包；通用接口与包装器位于base目录；配置解析器位于config目录；核心数据结构定义在pkg/core。

    核心组件¶

    • 接口定义：Parser接口统一了Parse与ParseStream两种能力，后者通过通道实现流式输出，便于大文件处理。
    • 文档模型：Document承载内容、元数据、来源与类型，是解析器输出的标准载体。
    • 基础包装器：GenericStreamWrapper将一次性读取的解析器桥接到流式接口，保证兼容性。
    • 三元组抽取：TriplesExtractor可结合LLM从文本中抽取实体与关系，用于知识图谱构建。

    架构总览¶

    解析器体系采用“注册表 + 类型枚举 + 各格式解析器”的模式，统一对外暴露Parse/ParseStream接口，并通过Metadata传递上下文信息（如source、页号、幻灯片号等）。流式解析器在内部goroutine中逐段产出Document，配合context实现取消控制。

    sequenceDiagram participant U as "调用方" participant R as "解析器注册表" participant P as "具体解析器" participant D as "Document通道" U->>R : 请求解析器(类型) R-->>U : 返回解析器实例 U->>P : 调用 ParseStream(ctx, reader, metadata) P->>D : 在goroutine中产出多个Document loop 流式消费 U->>D : 读取下一个Document D-->>U : 返回Document(内容+元数据) end

    详细组件分析¶

    PDF解析器（流式）¶

    • 支持类型：.pdf
    • 流式策略：将Reader读入内存后构造ReaderAt，逐页读取纯文本，每页产出一个Document；通过ctx.Done()实现取消。
    • 元数据：包含页号、总页数、解析器标识等。
    • 性能要点：默认块大小为字符级阈值，避免单页过大导致内存峰值；对不可寻址Reader自动全量读取以满足依赖库要求。
    flowchart TD Start(["开始 ParseStream"]) --> Read["读取输入到内存<br/>生成ReaderAt"] Read --> Open["创建PDF读取器"] Open --> Loop{"遍历页码"} Loop --> |逐页| Page["获取页对象"] Page --> Plain["提取纯文本"] Plain --> Trim{"是否为空？"} Trim --> |是| Loop Trim --> |否| Meta["合并元数据<br/>page_number,total_pages,parser"] Meta --> Emit["创建Document并发送"] Emit --> Loop Loop --> |完成| End(["结束"])

    DOCX解析器（流式）¶

    • 支持类型：.docx
    • 流式策略：将输入读入内存，作为zip容器解析，读取可编辑内容，整体产出一个Document；通过ctx.Done()实现取消。
    • 元数据：包含解析器标识、文件数量等。
    • 注意：当前实现仅抽取可编辑文本，未做表格/图片等结构化提取。
    sequenceDiagram participant S as "ParseStream" participant Z as "ZIP容器" participant D as "DOCX引擎" participant C as "通道" S->>S : 读取全部输入 S->>Z : 以ReaderAt打开ZIP S->>D : 从内存读取DOCX D-->>S : 可编辑内容 S->>C : 产出单个Document(内容+元数据)

    PPT解析器（流式）¶

    • 支持类型：.pptx
    • 流式策略：将输入读入内存，打开PPTX，遍历幻灯片，每张幻灯片产出一个Document；通过ctx.Done()实现取消。
    • 元数据：包含幻灯片号、总张数、解析器标识等。
    • 注意：当前实现为占位逻辑，实际文本抽取仍需完善。
    flowchart TD Start(["开始 ParseStream"]) --> Read["读取输入到内存"] Read --> Open["打开PPTX文档"] Open --> Loop{"遍历幻灯片"} Loop --> |逐张| Slide["生成占位内容"] Slide --> Meta["合并元数据<br/>slide_number,total_slides,parser"] Meta --> Emit["创建Document并发送"] Emit --> Loop Loop --> |完成| End(["结束"])

    图像解析器（视觉RAG）¶

    • 支持类型：.jpg/.jpeg/.png/.webp
    • 功能：提取EXIF元数据（经纬度、拍摄时间），并通过LLM生成图像描述；最终产出一个Document。
    • 元数据：包含解析器标识、字节数、EXIF字段等。
    • 注意：当前LLM描述为占位逻辑，实际多模态调用需根据gochat客户端能力扩展。
    sequenceDiagram participant S as "ParseStream" participant E as "EXIF解析" participant L as "LLM(可选)" participant C as "通道" S->>S : 读取全部输入 S->>E : 解码EXIF alt 提供LLM S->>L : 发送视觉提示 L-->>S : 返回描述 else 未提供LLM S->>S : 使用占位描述 end S->>C : 产出Document(描述+元数据)

    配置类文档解析器（流式）¶

    • 支持类型：.toml/.ini/.cfg/.conf/.properties/.env/.yaml/.yml
    • 特性：
    • 流式分块：按行读取，达到阈值即产出一块，支持重叠拼接
    • 安全：可启用敏感信息掩码（密码、密钥、令牌等）
    • 环境变量：可选择性展开（默认关闭）
    • 自动格式检测：可自动识别格式类型
    • 元数据：包含解析器标识、是否流式、掩码状态、位置索引、格式名等
    flowchart TD Start(["开始 ParseStream"]) --> Read["按行读取"] Read --> Proc["处理行<br/>去注释/掩码/展开"] Proc --> Len{"达到块大小？"} Len --> |否| Read Len --> |是| Overlap["叠加上一块重叠部分"] Overlap --> Emit["创建Document并发送"] Emit --> Update["更新重叠缓冲区"] Update --> Read Read --> EOF{"到达EOF？"} EOF --> |是| Flush["发送剩余内容"] EOF --> |否| Read Flush --> End(["结束"])

    基础接口与包装器¶

    • Parser接口：统一Parse与ParseStream，后者返回<-chan *Document，实现O(1)内存复杂度
    • GenericStreamWrapper：将一次性读取的解析器桥接为流式接口，便于统一接入
    • TriplesExtractor：基于LLM抽取三元组，用于知识图谱
    classDiagram class Parser { +Parse(ctx, content, metadata) *Document +Supports(contentType) bool +ParseStream(ctx, reader, metadata) <-chan *Document +GetSupportedTypes() []string } class GenericStreamWrapper { -extractor ContentExtractor -supportedTypes []string -parserName string +ParseStream(ctx, reader, metadata) <-chan *Document +GetSupportedTypes() []string } class TriplesExtractor { -llm Client +Extract(ctx, text) []Triple } Parser <|..     依赖分析¶

    • 解析器注册表：集中注册所有内置解析器，按类型映射到具体实现
    • 类型枚举：ParserType涵盖文本、Markdown、代码、PDF、DOCX、Excel、CSV、JSON、XML、YAML、日志、HTML、图片、邮件、PPT、数据库Schema等
    • 依赖关系：各解析器仅依赖自身所需第三方库与core.Document；注册表负责装配
    graph LR T["types.ParserType"] --> R["types.Registry"] R --> PDF["pdf.Parser"] R --> DOCX["docx.Parser"] R --> PPT["ppt.Parser"] R --> IMG["image.Parser"] R --> CFG["config.ConfigStreamParser"]

    性能考虑¶

    • 流式内存模型：ParseStream通过通道与goroutine实现O(1)内存占用，适合超大文件
    • 分块与重叠：配置解析器支持块大小与重叠，平衡吞吐与语义连续性
    • 上下文取消：在关键路径检查ctx.Done()，及时终止耗时操作
    • 读取策略：PDF/DOCX/PPTX等需要ReaderAt或全量读取，解析器内部自动处理
    • 元数据最小化：仅注入必要元数据，减少下游处理负担

    故障排查指南¶

    • 输入为空或不可读：检查上游Reader是否正确传入，确认ReadAll阶段无错误
    • PDF/DOCX/PPTX解析失败：确认依赖库可用，检查ReaderAt构造与文档格式合法性
    • 图像解析无描述：若未注入LLM客户端，将使用占位描述；请注入支持多模态的客户端
    • 配置解析掩码/展开异常：检查掩码正则与环境变量展开开关
    • 流式中断：确认ctx是否提前取消，或通道阻塞导致的资源泄漏

    结论¶

    该解析器体系以统一接口与流式通道为核心，覆盖PDF、DOCX、PPTX、图片与配置类文档等多种格式。通过注册表与类型枚举实现灵活扩展，结合上下文取消与分块策略，有效支撑大规模文档索引场景。未来可在PPTX文本抽取、图片多模态描述、表格与图像嵌入等方面进一步增强。

    附录：使用示例与最佳实践¶

    • 选择解析器
    • 使用注册表按类型获取解析器实例，或直接调用对应包的DefaultParser函数
    • 对于需要一次性读取的格式（如DOCX/PDF/PPTX），可使用Parse；对于大文件或需要流式的场景，优先使用ParseStream
    • 流式处理最佳实践
    • 明确ctx生命周期，避免长时间阻塞；在循环内定期检查ctx.Done()
    • 合理设置块大小与重叠，平衡吞吐与语义连贯
    • 将大文本交由后续Chunker进行细粒度切分
    • 配置解析器参数
    • 块大小与重叠：根据内容密度调整
    • 掩码敏感信息：默认开启，确保安全
    • 展开环境变量：默认关闭，按需开启
    • 自动格式识别：默认开启，可显式指定格式
    • 复杂文档处理
    • PDF：按页拆分，保留页号与总页数元数据
    • DOCX：抽取可编辑文本，注意空内容过滤
    • PPTX：按幻灯片拆分，逐步完善文本抽取
    • 图片：结合EXIF与LLM描述，形成检索友好的文本摘要
