    存储管理¶

    本文件面向 GoRAG 存储管理系统，系统性阐述文档存储、向量存储与图谱存储的设计理念与实现方式；详解 SQLite、Bolt、GoVector 等后端的特点与配置方法；说明多存储后端的协调机制与一致性保障；提供性能优化策略（索引设计、批量操作、缓存机制）；给出存储迁移与备份恢复方案，并给出容量规划与成本优化建议。

    项目结构¶

    GoRAG 将“核心实体”与“具体存储实现”分层组织： - 核心接口与实体位于 pkg/core：定义文档、向量、图节点/边、以及各存储接口（文档、图谱、索引、聊天记忆）。 - 具体存储实现位于 pkg/indexing：按类型划分目录，如 sqlite、bolt、vectorstore（milvus、pinecone、memory、govector）等。

    核心组件¶

    • 文档存储接口与实现
    • 接口职责：负责持久化完整文档与分块（含父子关系），支持按文档或分块检索。
    • 实现：SQLite、Bolt 提供关系型/键值存储方案；可扩展至其他数据库。
    • 图谱存储接口与实现
    • 接口职责：维护节点（实体）、边（关系），支持邻居查询、语义查询、社区摘要等。
    • 实现：SQLite 支持递归邻居查询与社区表；Bolt 通过邻接列表实现邻居遍历。
    • 向量存储接口与实现
    • 接口职责：支持向量 Upsert、Search、Delete、Close；支持元数据过滤。
    • 实现：GoVector（本地嵌入式存储）、Milvus（分布式向量数据库）、Pinecone（云向量服务）、内存向量存储（开发/测试）。
    • 索引与聊天记忆
    • 索引存储：记录已构建索引拓扑，支撑增量更新与路由。
    • 聊天记忆：会话级消息缓冲，适配多轮/代理式 RAG。

    组件详解¶

    文档存储（DocStore）¶

    设计理念 - 区别于仅存向量与元数据的向量库，文档存储保存完整文本与结构关系（父子分块），支撑父文档检索、多跳检索等高级 RAG 场景。 - 提供文档级写入/删除与分块级读取能力，便于检索阶段回取原文。

    实现要点 - SQLite 文档存储 - 表结构：documents（主键 id）、chunks（主键 id，外键 document_id），并建立分块表的 document_id 索引。 - 批量写入：使用事务包裹插入，提升吞吐。 - 删除：事务中先删文档再删分块，确保一致性。 - Bolt 文档存储 - 桶：documents、chunks，分别存放序列化后的文档与分块。 - 批量写入：逐条 Put，适合小批量或流式写入。 - 分块按文档检索：游标遍历桶，筛选 document_id 匹配项。

    图谱存储（GraphStore）¶

    设计理念 - 支持节点（实体）与边（关系）的增改查，提供邻居查询、语义查询与社区摘要等能力，满足 GraphRAG 的检索与总结需求。 - SQLite 通过递归 CTE 实现邻居遍历；Bolt 通过邻接列表（出边/入边桶）实现高效遍历。

    实现要点 - SQLite 图谱存储 - 表结构：nodes（id, type, properties）、edges（id, type, source, target, properties，含外键约束）、communities（可选，用于社区摘要）。 - 邻居查询：使用递归 CTE 获取指定深度内的节点集合，再回查节点与边。 - 查询：原生 SQL 查询，参数映射受限；复杂查询需自定义解析器。 - Bolt 图谱存储 - 桶：nodes、edges、out_edges（source->edgeID[]）、in_edges（target->edgeID[]）、communities。 - 邻居查询：广度优先遍历，基于邻接列表桶维护边集合，避免全表扫描。 - 复杂查询：当前未实现，保留扩展空间。

    向量存储（VectorStore）¶

    设计理念 - 以向量为核心，支持高维相似度检索；同时保留 chunk_id 与元数据，便于回取原文与二次过滤。 - 支持多种后端：本地嵌入式（GoVector）、分布式向量数据库（Milvus）、云向量服务（Pinecone）、内存向量存储（开发/测试）。

    实现要点 - GoVector - 本地嵌入式存储，支持 HNSW 索引与 Cosine 距离；默认 collection 名称、维度与路径可配置。 - Upsert：批量写入，注入 chunk_id 到 payload；Search：支持过滤条件与 TopK。 - Milvus - 自动检测/创建集合，插入后 Flush 并创建索引，最后 LoadCollection。 - Upsert：批量插入向量、chunk_id、metadata JSON；Search：基于表达式过滤（metadata 中包含键值对）。 - Pinecone - 基于 API Key 连接，描述索引后建立连接；Upsert：构建向量元数据（包含 chunk_id）；Search：支持结构化过滤。 - 内存向量存储 - 基于内存 map 与预计算范数，采用余弦相似度与快速选择/排序返回 TopK。

    索引存储与聊天记忆¶

    • 索引存储（IndexStore）
    • 记录索引拓扑（IndexID、类型、节点集合、摘要、配置），支撑增量更新与路由。
    • 聊天记忆（ChatMemory）
    • 会话级消息缓冲，支持添加用户/助手消息、获取历史、清理。

    依赖关系分析¶

    • 接口与实体
    • 核心接口（VectorStore、DocStore、GraphStore）定义了统一的 CRUD 与检索能力，便于替换不同后端。
    • 实体（Document、Chunk、Vector、Node、Edge）承载数据模型，被各存储实现直接使用。
    • 存储实现
    • SQLite/Bolt 文档/图存储：直接对接数据库驱动，提供关系/键值语义。
    • 向量存储：GoVector 本地嵌入式；Milvus/Pinecone 为外部服务，需网络与配置。
    • 耦合与内聚
    • 存储实现与核心接口耦合低，内聚高；通过选项模式（WithXxx）灵活配置。
    • 向量搜索与元数据解耦，便于二次过滤与结果增强。

    性能考量¶

    • 索引设计
    • SQLite：为 chunks(document_id)、edges(source)/edges(target) 建立索引，加速分块检索与邻居查询。
    • Bolt：通过邻接列表桶（out_edges/in_edges）避免全表扫描，邻居查询时间复杂度显著降低。
    • 向量：GoVector 支持 HNSW；Milvus/Pinecone 默认索引类型可按维度与数据规模调优。
    • 批量操作
    • 文档/图存储：使用事务（BeginTx/Commit）包裹批量写入，减少磁盘刷写次数。
    • 向量存储：批量 Upsert，减少网络往返（Milvus/Pinecone）或 IO 次数（GoVector/内存）。
    • 缓存机制
    • 内存向量存储：预计算向量范数，加速余弦相似度计算；TopK 使用快速选择+排序。
    • 可在应用层对热点文档/图节点进行缓存（依据业务场景）。
    • 查询优化
    • SQLite 递归 CTE 需限制 depth 与 limit；Bolt 遍历队列与访问集控制上限。
    • 向量过滤：尽量使用精确键值过滤，避免昂贵的全文/模糊匹配。

    故障排查指南¶

    • 文档/图存储常见问题
    • SQLite：打开数据库失败、建表失败、事务回滚；检查路径权限与磁盘空间。
    • Bolt：打开失败、桶初始化失败；确认文件权限与目录存在。
    • 向量存储常见问题
    • GoVector：存储初始化失败、集合创建失败；检查 dbPath 权限与磁盘空间。
    • Milvus：客户端创建/连接超时、HasCollection/创建集合失败；检查地址可达性与认证。
    • Pinecone：API Key 错误、索引描述失败；核对密钥与索引名。
    • 查询与过滤
    • SQLite/图查询：参数绑定顺序与占位符不匹配导致错误；确保查询字符串与参数一致。
    • 向量过滤：Milvus 元数据过滤表达式格式；Pinecone 结构化过滤字段类型。

    结论¶

    GoRAG 通过清晰的接口分层与多样化的存储实现，实现了文档、向量与图谱的协同工作。SQLite/Bolt 提供轻量、易部署的本地方案；GoVector/Milvus/Pinecone 提供可扩展的向量检索能力；配合索引与批量操作策略，可在不同规模与成本目标下取得平衡。多存储后端通过统一接口与选项模式实现灵活切换与组合，满足从开发到生产的多样化需求。

    附录¶

    多存储后端协调与一致性¶

    • 协调机制
    • 文档/向量：通过 chunk_id 关联，检索到向量后再回取文档/分块原文。
    • 图谱/向量：节点属性与边关系可用于过滤与重排，结合向量相似度融合。
    • 一致性保证
    • 文档/图存储：事务写入（SQLite/Bolt）确保批量操作原子性。
    • 向量存储：Milvus/Pinecone 的 Flush/Index/Load 流程保证可见性；GoVector 本地存储在进程内一致。

    配置方法速览¶

    • SQLite 文档/图存储
    • 默认路径：gorag_docs.db / gorag_graph.db；可通过构造函数传入自定义路径。
    • Bolt 文档/图存储
    • 默认路径：gorag_docs.bolt / gorag_graph.bolt；确保目录存在且有写权限。
    • GoVector
    • 选项：WithDBPath、WithCollection、WithDimension、WithHNSW；默认路径 gorag_vectors.db。
    • Milvus
    • 选项：WithCollection、WithDimension；默认 localhost:19530，自动创建集合。
    • Pinecone
    • 选项：WithIndex、WithEnvironment、WithDimension、WithNamespace；默认环境 gcp-starter。
    • 内存向量存储
    • 适用于测试/开发，无需额外配置。

    存储迁移与备份恢复¶

    • SQLite
    • 备份：直接复制 .db 文件；恢复：停止服务后替换文件。
    • 迁移：导出/导入表结构与数据，或使用 .backup 原生命令。
    • Bolt
    • 备份：复制 .bolt 文件；恢复：同上。
    • 迁移：导出桶内容，转换为新格式后导入。
    • GoVector
    • 备份：复制存储目录；恢复：替换目录后重启。
    • Milvus/Pinecone
    • 备份：Milvus 侧通过备份/快照；Pinecone 侧通过 SDK 导出/导入。
    • 迁移：重建集合/索引，重新导入数据。

    容量规划与成本优化¶

    • 文档存储
    • SQLite/Bolt：根据文档数量与分块数量估算表大小；为分块表建立索引，减少查询开销。
    • 图谱存储
    • SQLite：edges 外键约束带来一致性，但索引与约束增加写入成本；Bolt 通过邻接列表降低遍历成本。
    • 向量存储
    • GoVector：本地磁盘成本低，适合中小规模；Milvus/Pinecone：按向量数量、维度与索引类型计费，建议按峰值 QPS 与数据增长预测扩容。
    • 成本优化
    • 写入批量化、索引按需创建、冷热数据分离、定期清理无效数据；在开发/测试环境优先使用内存向量存储降低资源占用。
