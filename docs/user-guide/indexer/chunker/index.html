    分块策略¶

    本文件系统性阐述 GoRAG 的分块策略，覆盖字符级分块、标记级分块与语义分块的工作原理、参数配置、对检索效果的影响、策略选择指南、性能优化建议、质量评估与调试技巧，以及自定义分块器的开发与集成方式。文档面向不同技术背景的读者，既提供高层概览，也给出可操作的落地建议。

    graph TB subgraph "索引流水线" DISC["Discover<br/>发现文件"] MULTI["Multi Parser<br/>多解析器流式解析"] CHUNK["Chunk Step<br/>分块步骤"] EMBED["Embed Step<br/>批量嵌入"] UPSERT["Upsert Step<br/>向量库入库"] end subgraph "分块器" CHAR["CharacterChunker<br/>字符级分块"] TOK["TokenChunker<br/>标记级分块"] SEM["SemanticChunker<br/>语义分块(装饰器)"] end DISC --> MULTI --> CHUNK --> EMBED --> UPSERT CHUNK --> CHAR CHUNK --> TOK CHUNK --> SEM

    核心组件¶

    • 分块接口与实体
    • Chunk：文档分块实体，包含内容、元数据、起止索引、向量ID等。
    • Chunker：统一的分块接口，定义 Chunk 方法。
    • SemanticChunker：扩展接口，支持父子层级分块与上下文注入。
    • 分块器实现
    • CharacterChunker：基于字符（Rune）的硬切分，速度快、无需外部词表。
    • TokenChunker：基于 Token 的切分，严格遵守上下文窗口，适合生产环境。
    • SemanticChunker：装饰器模式，包装基础分块器，提供 HierarchicalChunk 与 ContextualChunk 能力。

    详细组件分析¶

    字符级分块（CharacterChunker）¶

    • 特点
    • 基于字符（Rune）计数进行硬切分，速度快、零外部依赖。
    • 默认参数：分块大小 1000，重叠 150；分隔符优先级用于尽量保持段落/句子完整。
    • 参数
    • ChunkSize：每块最大字符数
    • ChunkOverlap：相邻块重叠字符数
    • Separators：尝试的分隔符序列（段落、行、单词、字符）
    • 适用场景
    • 快速原型、日志/纯文本、对语义精度要求不高
    • 注意事项
    • 重叠有助于避免关键上下文被截断
    • 若需更严格的语义边界，可结合分隔符策略或升级到 TokenChunker

    标记级分块（TokenChunker）¶

    • 特点
    • 基于 Token 编码（tiktoken）进行切分，严格遵守 LLM/Embedding 的上下文窗口。
    • 默认参数：500 Token 分块、50 Token 重叠、cl100k_base 词表。
    • 参数
    • ChunkSize：每块最大 Token 数
    • ChunkOverlap：相邻块重叠 Token 数
    • Encoding：tiktoken 编码模型
    • 适用场景
    • 生产环境、中英混合、含代码/特殊符号、严格控制 API 成本
    • 注意事项
    • 重叠步长确保不会出现无限循环
    • 精确的字符索引映射较复杂，实现中采用索引标识简化

    语义分块（SemanticChunker）¶

    • 角色与模式
    • 装饰器模式：包装基础 Chunker（如 Token/Character），扩展高级能力。
    • 两个核心能力：HierarchicalChunk（父子层级）、ContextualChunk（上下文注入）。
    • 参数
    • BaseChunker：基础分块器实例
    • ParentChunkSize/ChildChunkSize：父块与子块大小
    • Overlap：父子块通用重叠
    • 工作原理
    • HierarchicalChunk：先按父块大小切分，再对每个父块按子块大小切分，形成父子层级结构，便于“精查+全看”。
    • ContextualChunk：在每个 Chunk 内容前注入文档摘要，降低 LLM 断章取义风险。
    • 适用场景
    • 事实问答、深度报告分析、法律合同问答、需要高精度与上下文一致性的复杂问答
    flowchart TD START(["开始"]) --> PARENT["父块切分<br/>ParentChunkSize, Overlap"] PARENT --> CHILD["子块切分<br/>ChildChunkSize, Overlap/2"] CHILD --> HIER["输出: 父块集合 + 子块集合"] START --> CTX["上下文注入: 将摘要前置到每个 Chunk"] CTX --> OUT["输出: 注入后的 Chunk 列表"]

    分块步骤（Pipeline Step）¶

    • 职责
    • 从状态通道读取文档，调用分块器生成 Chunk，写入下一个阶段的通道。
    • 关键点
    • 对空分块器与空文档通道进行显式校验
    • 将所有 Chunk 汇总后统一放入通道，便于下游批处理
    sequenceDiagram participant PIPE as "索引流水线" participant STEP as "Chunk 步骤" participant CHUNKER as "分块器" PIPE->>STEP : 传入 IndexingContext(含 Documents 通道) STEP->>STEP : 遍历文档 STEP->>CHUNKER : 调用 Chunk(doc) CHUNKER-->>STEP : 返回 Chunk 列表 STEP-->>PIPE : 写入 Chunks 通道

    嵌入步骤（Embed Step）¶

    • 职责
    • 从 Chunks 通道批量生成向量，记录指标，更新 Chunk 的向量ID。
    • 关键点
    • 逐 Chunk 调用嵌入服务，构造向量实体并写回状态
    • 统计总分块数，上报嵌入指标

    依赖分析¶

    • 接口与实现
    • core.Chunker 定义分块接口；CharacterChunker、TokenChunker 实现该接口。
    • core.SemanticChunker 扩展 Chunker，提供 HierarchicalChunk 与 ContextualChunk。
    • 分块步骤依赖 core.SemanticChunker 接口，可注入任意实现。
    • 外部依赖
    • TokenChunker 依赖 tiktoken-go 进行 Token 编解码。
    • SemanticChunker 在未指定父块分块器时，内部回退到 CharacterChunker。

    性能考虑¶

    • 分块大小与重叠率
    • ChunkSize 越大，单块语义越完整但召回粒度粗；越小则召回更精细但易丢失上下文。
    • Overlap 越大，跨块连续性越好，但向量数量与存储/计算成本上升。
    • 字符 vs Token
    • CharacterChunker 速度最快，适合快速迭代与纯文本；TokenChunker 更安全，避免越界与成本波动。
    • 层级分块的权衡
    • 父子层级增加向量化与存储成本，但检索时可“先精后全”，提升最终回答质量。
    • 重叠缓冲与流式解析
    • 解析器在流式切分中使用重叠缓冲，确保跨块边界完整性，可借鉴到自定义分块器设计中。

    故障排查指南¶

    • 常见问题与定位
    • 分块器未配置：分块步骤会返回错误，检查是否正确注入 Chunker。
    • 文档通道为空：确认上游解析器已产出文档通道。
    • Token 编码失败：检查模型名与词表可用性。
    • 层级分块异常：确认父/子块尺寸与重叠设置是否合理。
    • 调试技巧
    • 打印分块数量与平均长度，评估是否符合预期。
    • 对比父子层级的召回分布，观察是否出现“过细/过粗”现象。
    • 使用上下文修剪工具评估最终上下文质量，结合相关性分数进行优化。
    • 质量评估
    • 可使用上下文修剪模块的评分逻辑，对召回的 Chunk 进行相关性打分与裁剪，间接评估分块质量。
    • 结合检索后评估（如 Self-RAG）反馈，持续迭代分块参数。

    结论¶

    • 选择建议
    • 初期原型：CharacterChunker（默认参数即可）。
    • 生产稳定：TokenChunker（严格控制上下文窗口与成本）。
    • 高质量问答：SemanticChunker（父子层级 + 上下文注入）。
    • 运维建议
    • 以实验驱动调整 ChunkSize 与 Overlap，结合业务指标持续优化。
    • 在复杂场景下优先采用 TokenChunker 作为基础分块器，再叠加语义能力。
    • 借助上下文修剪与评估工具闭环优化分块策略。

    附录¶

    分块策略选择指南¶

    • 场景判断
    • 本地测试/英文为主/结构简单 → CharacterChunker
    • 生产部署/中英混合/严格成本控制 → TokenChunker
    • 复杂问答/长文总结/避免断章取义 → SemanticChunker
    • 初始化示例
    • 字符分块器：使用默认参数快速起步。
    • Token 分块器：使用默认 Token 参数与常用词表。
    • 语义分块器：内置 Token 分块器，按需微调父子块大小与重叠。

    自定义分块器开发与集成¶

    • 开发步骤
    • 实现 core.Chunker 接口（Chunk 方法）。
    • 设计参数：ChunkSize、ChunkOverlap、必要时加入分隔符策略。
    • 在索引流水线中通过分块步骤注入自定义实现。
    • 集成要点
    • 保持与现有 Chunk 结构兼容，继承文档元数据。
    • 如需层级/上下文能力，可参考 SemanticChunker 的装饰器模式进行组合。
