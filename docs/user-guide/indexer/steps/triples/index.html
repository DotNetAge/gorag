    三元组提取步骤¶

    本章节聚焦于 GoRAG 索引器中的“三元组提取”步骤，系统性阐述如何从文档分块内容中抽取“实体-关系-实体（Subject-Predicate-Object）”三元组，并将其写入知识图谱存储后端。文档覆盖以下要点： - 三元组抽取算法与提示词设计 - 三元组数据结构与存储格式 - 与图谱存储后端（SQLite、BoltDB、Neo4j）的集成方式 - 配置选项与参数调优建议 - 质量评估与去重策略 - 知识图谱构建最佳实践与场景示例

    项目结构¶

    围绕三元组提取与图谱构建的相关模块分布如下： - 抽取层：基于 LLM 的三元组抽取器 - 步骤层：将抽取结果写入图谱存储的流水线步骤 - 存储层：多种图存储实现（SQLite、BoltDB、Neo4j） - 辅助工具：去重与检索增强

    graph TB subgraph "抽取层" TE["TriplesExtractor<br/>抽取三元组"] end subgraph "步骤层" STEP["ExtractTriples 步骤<br/>写入图谱"] end subgraph "存储层" GS["GraphStore 接口"] SQ["SQLiteGraphStore"] BT["BoltGraphStore"] NJ["Neo4jGraphStore"] end TE --> STEP STEP --> GS GS --> SQ GS --> BT GS --> NJ

    核心组件¶

    • 三元组数据结构
    • 字段：主体（Subject）、谓语（Predicate）、客体（Object）、主体类型（SubjectType）、客体类型（ObjectType）
    • 结构体定义与 JSON 序列化标签见抽取器源码
    • 三元组抽取器
    • 基于 LLM 的提示词工程，约束输出为 JSON 数组
    • 清洗 LLM 输出（去除代码块包裹），并进行 JSON 反序列化
    • 图谱节点与边
    • 节点（Node）：ID、类型（Type）、属性（Properties）
    • 边（Edge）：ID、类型（Type）、源节点（Source）、目标节点（Target）、属性（Properties）
    • 图谱存储接口
    • 统一的 UpsertNodes/UpsertEdges、GetNode、GetNeighbors、Query、GetCommunitySummaries、Close 等方法

    架构总览¶

    三元组提取在索引流水线中的位置与数据流向如下：

    sequenceDiagram participant P as "索引上下文" participant S as "ExtractTriples 步骤" participant E as "TriplesExtractor" participant G as "GraphStore" P->>S : 提供分块流 Chunks S->>S : 缓冲分块并恢复下游通道 loop 遍历每个分块 S->>E : Extract(text) E-->>S : 返回三元组数组 alt 成功 loop 遍历每个三元组 S->>G : UpsertNodes(主体节点) S->>G : UpsertNodes(客体节点) S->>G : UpsertEdges(边) end else 失败 S->>S : 记录警告并跳过 end end S-->>P : 完成并返回

    详细组件分析¶

    三元组抽取器（TriplesExtractor）¶

    • 设计要点
    • 使用固定提示词模板，约束抽取仅限明确陈述的事实，谓语长度限制为 1–3 个词，要求标注实体类型
    • 将待处理文本注入提示词，调用 LLM 的 Chat 接口获取 JSON 输出
    • 清洗 LLM 输出（Markdown 包裹的 JSON），再进行反序列化
    • 错误处理
    • LLM 调用失败或 JSON 解析失败时返回错误
    • 输出
    • 三元组数组，每个元素包含主体、谓语、客体及其类型
    classDiagram class TriplesExtractor { +llm : chat.Client +NewTriplesExtractor(llm) +Extract(ctx, text) []Triple } class Triple { +Subject : string +Predicate : string +Object : string +SubjectType : string +ObjectType : string } TriplesExtractor --> Triple : "生成"

    索引步骤：ExtractTriples¶

    • 功能职责
    • 从索引上下文的分块流中读取所有分块，恢复下游通道以保持流式处理
    • 对每个分块调用抽取器生成三元组
    • 将三元组对应的主体与客体作为节点写入图谱，谓语作为边写入图谱
    • 写入策略
    • 节点属性包含来源分块 ID 与来源文件路径，便于溯源
    • 边 ID 由“主体-谓语-客体”拼接而成，属性包含来源分块 ID
    • 错误处理
    • 抽取失败时记录警告并跳过该分块；节点/边写入失败时记录错误并继续

    图谱存储后端集成¶

    • GraphStore 接口
    • 统一的节点/边写入、节点查询、邻居查询、通用查询、社区摘要查询与关闭接口
    • SQLite 实现
    • 使用关系表存储节点与边，索引优化邻接查询；支持递归 CTE 进行邻居遍历
    • Upsert 使用 INSERT OR REPLACE，属性以 JSON 文本存储
    • BoltDB 实现
    • 使用键值桶存储节点与边，额外维护出边/入边邻接列表；自动去重邻接边 ID
    • Neo4j 实现
    • 使用 MERGE 保证节点与边的幂等写入；通过 Cypher 查询邻居与执行自定义查询
    • 支持社区摘要查询（Community 节点）

    与检索增强的关系抽取对比¶

    • 本仓库同时提供两类抽取能力：
    • 三元组抽取（面向结构化事实抽取）
    • 关系抽取（GraphExtractor，返回节点与边集合）
    • 两者均可用于知识图谱构建，选择取决于任务需求与数据形态

    依赖关系分析¶

    • 抽取器依赖 LLM 客户端接口（chat.Client）
    • 步骤依赖 GraphStore 接口，不绑定具体实现
    • 各存储实现均实现 GraphStore 接口，可互换替换
    • 去重步骤可用于检索阶段，避免重复内容影响检索质量
    graph LR TE["TriplesExtractor"] --> E["Extract(ctx, text)"] E --> TR["[]Triple"] STEP["ExtractTriples 步骤"] --> GS["GraphStore 接口"] GS --> SQ["SQLite 实现"] GS --> BT["BoltDB 实现"] GS --> NJ["Neo4j 实现"] DEDUP["去重步骤"] --> RET["检索结果去重"]

    性能考量¶

    • 流式处理
    • 步骤会先缓冲分块，再恢复下游通道，确保后续步骤可继续消费
    • 写入批处理
    • 当前实现逐条 Upsert 节点与边；在高吞吐场景下可考虑批量写入以减少事务开销
    • 邻居查询
    • SQLite 使用递归 CTE，Neo4j 使用 Cypher，BoltDB 使用邻接列表；合理设置 depth 与 limit 控制查询规模
    • 去重
    • 在检索阶段可采用内容去重，降低重复上下文对 LLM 的干扰

    故障排查指南¶

    • LLM 输出格式异常
    • 症状：JSON 解析失败
    • 处理：确认提示词约束与 LLM 输出格式；必要时调整提示词或模型参数
    • 图谱写入失败
    • 症状：UpsertNodes/UpsertEdges 报错
    • 处理：检查存储连接、权限与表/桶初始化状态；查看日志定位具体错误
    • 邻居查询为空
    • 症状：GetNeighbors 返回空
    • 处理：确认节点 ID 是否正确、是否存在边；适当增大 limit 或 depth
    • 去重无效
    • 症状：重复内容仍被检索返回
    • 处理：确认去重阈值与内容比较逻辑；确保在检索前执行去重步骤

    结论¶

    三元组提取步骤通过统一的提示词工程与 LLM 推理，将非结构化文本转化为结构化的知识图谱三元组，并以幂等方式写入多种图存储后端。结合检索增强与去重策略，可在不同规模与场景下稳定构建高质量知识图谱。

    附录¶

    三元组数据结构与存储格式¶

    • 三元组字段
    • 主体（Subject）、谓语（Predicate）、客体（Object）、主体类型（SubjectType）、客体类型（ObjectType）
    • 节点与边属性
    • 节点属性包含来源分块 ID 与来源文件路径
    • 边属性包含来源分块 ID
    • 存储格式
    • SQLite：节点/边表，属性以 JSON 文本存储
    • BoltDB：节点/边桶，邻接列表桶维护出/入边
    • Neo4j：节点与关系，属性映射为属性字段

    配置选项与参数调优建议¶

    • 提示词约束
    • 严格限定输出为 JSON 数组，限制谓语长度，强制标注实体类型
    • LLM 参数
    • 适当提高温度以提升创造性，但需平衡稳定性；必要时启用输出校验
    • 存储参数
    • SQLite：合理设置递归深度与查询 limit；建立索引以优化邻接查询
    • BoltDB：利用邻接列表自动去重；控制写入批次大小
    • Neo4j：使用 MERGE 保证幂等；针对高频查询建立索引或复合索引
    • 流水线参数
    • 分块大小与重叠度：兼顾抽取准确性与性能
    • 去重阈值：根据内容相似度设定，避免过度去重导致信息丢失

    三元组质量评估与去重机制¶

    • 质量评估
    • 基于提示词约束与 JSON 输出校验，减少噪声
    • 在检索阶段可引入去重步骤，避免重复内容影响答案质量
    • 去重机制
    • 检索阶段按内容去重，保留首次出现的分块
    • 三元组层面的去重可结合边 ID（主体-谓语-客体）进行判重

    知识图谱构建最佳实践与场景示例¶

    • 最佳实践
    • 明确实体类型体系与谓语规范，确保抽取一致性
    • 分块粒度适中，避免跨句事实被切分
    • 在写入前进行内容去重，减少冗余
    • 根据场景选择合适存储：本地开发用 SQLite/BoltDB，生产用 Neo4j
    • 场景示例
    • 法律合同：抽取“公司-签署日期-合同编号”、“个人-职位-公司”等三元组
    • 技术文档：抽取“模块-依赖-模块”、“类-方法-异常”等三元组
    • 新闻资讯：抽取“人物-事件-地点”、“组织-关系-组织”等三元组
