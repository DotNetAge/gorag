    分块处理步骤¶

    本节面向 GoRAG 索引器的“分块处理”阶段，系统性阐述分块策略与实现细节，重点覆盖以下内容： - 分块函数与管道步骤的职责边界与调用链 - 三种分块策略：字符分块、令牌分块、语义分块的原理、特点与适用场景 - 分块参数（分块大小、重叠长度）对检索效果与成本的影响 - 元数据保留与上下文维护机制 - 策略选择指南与最佳实践

    项目结构¶

    分块处理位于索引流水线的“语义分块”阶段，由独立的分块器包提供实现，并通过索引器构建器注入到流水线中。核心文件分布如下： - 分块器实现：字符分块、令牌分块、语义分块 - 核心类型与接口：Chunk、Chunker、SemanticChunker - 流水线步骤：语义分块步骤负责消费文档通道并产出分块通道 - 索引器构建：将分块器注入到流水线中

    graph TB subgraph "索引器流水线" DISC["Discover 步骤"] MULTI["Multi 解析步骤"] CHUNK["Chunk 语义分块步骤"] EMB["Batch/MultimodalEmbed 嵌入步骤"] UPS["MultiStore 写入步骤"] end subgraph "分块器实现" CHAR["CharacterChunker<br/>字符分块"] TOK["TokenChunker<br/>令牌分块"] SEM["SemanticChunker<br/>语义分块(装饰器)"] end DISC --> MULTI --> CHUNK --> EMB --> UPS CHUNK --> CHAR CHUNK --> TOK SEM --> TOK

    核心组件¶

    • Chunk：分块实体，包含唯一标识、所属文档 ID、层级与父子关系、内容、元数据、时间戳以及可选的向量 ID 等字段。
    • Chunker 接口：统一的分块入口，接收单个文档并返回分块列表。
    • SemanticChunker 接口：在 Chunker 基础上扩展了层级分块与上下文注入能力。

    架构总览¶

    分块处理在索引流水线中的位置与职责： - 输入：IndexingContext 中的文档通道（Documents） - 处理：语义分块步骤遍历文档，调用分块器生成分块 - 输出：将所有分块写入新的分块通道（Chunks），供后续嵌入与写入步骤使用

    sequenceDiagram participant IDX as "索引器" participant PIPE as "流水线" participant STEP as "Chunk 步骤" participant CKR as "分块器(任意)" participant OUT as "IndexingContext.Chunks" IDX->>PIPE : 初始化流水线 PIPE->>STEP : 执行语义分块步骤 STEP->>STEP : 从 state.Documents 读取文档 STEP->>CKR : 调用 Chunk(ctx, doc) CKR-->>STEP : 返回 []*Chunk STEP->>OUT : 将所有分块写入 state.Chunks STEP-->>PIPE : 完成

    详细组件分析¶

    字符分块器（CharacterChunker）¶

    • 设计要点
    • 基于字符（Rune）数量进行硬切分，支持分隔符序列以尽量保持段落/句子完整性
    • 提供默认参数（分块大小 1000，重叠 150），适合快速原型与简单文本
    • chunkText 实现逻辑
    • 将输入文本转为 rune 列表，按步长（分块大小 - 重叠）迭代切分
    • 末尾不足时截断至文本末
    • Chunk 方法
    • 继承文档元数据，为每个分块生成唯一 ID
    • 记录起止索引（按 rune 计算），便于后续定位
    • 适用场景
    • 本地快速验证、日志/纯文本等对语义精度要求不高的数据
    flowchart TD START(["进入 chunkText"]) --> EMPTY{"文本为空?"} EMPTY --> |是| RETNIL["返回空切片"] --> END EMPTY --> |否| INIT["初始化 runes 列表"] INIT --> LOOP["按步长(ChunkSize-Overlap)循环"] LOOP --> ENDCHK{"到达末尾?"} ENDCHK --> |是| APPEND["追加最后一个片段"] --> DONE["结束"] ENDCHK --> |否| SLICE["切片 [i, i+ChunkSize)"] --> APPEND2["追加片段"] --> LOOP

    令牌分块器（TokenChunker）¶

    • 设计要点
    • 基于词表将文本编码为 Token 序列，按 Token 数量切分，再解码回文本
    • 严格遵守下游模型的上下文窗口限制，避免越界与截断
    • 提供默认参数（分块大小 500，重叠 50，词表 cl100k_base），适配主流模型
    • chunkTokens 实现逻辑
    • 编码为 Token 数组，按步长（分块大小 - 重叠）切分
    • 解码回字符串形成最终片段
    • Chunk 方法
    • 继承文档元数据，并记录分块序号（chunk_index），便于后续定位
    • 起止索引（按 rune 计）留空，以保持高性能
    • 适用场景
    • 生产环境、中英混杂、含代码/特殊符号的复杂文档
    flowchart TD S(["进入 chunkTokens"]) --> TEMPTY{"文本为空?"} TEMPTY --> |是| TRET["返回空切片"] --> TEND TEMPTY --> |否| ENC["tiktoken.Encode 文本为 tokens"] ENC --> STEP["计算步长 = ChunkSize - Overlap"] STEP --> LOOP["按步长遍历 tokens"] LOOP --> DECODE["Decode tokens[i..i+ChunkSize] 为字符串"] DECODE --> PUSH["追加到结果切片"] PUSH --> DONE{"是否到达末尾?"} DONE --> |否| LOOP DONE --> |是| TRET

    语义分块器（SemanticChunker）¶

    • 设计要点
    • 采用装饰器模式，包装基础分块器（通常是 TokenChunker 或 CharacterChunker）
    • 提供两种高级能力：层级分块（父子块）、上下文注入（文档摘要）
    • 层级分块（HierarchicalChunk）
    • 先以较大分块（父块）切分，再对每个父块进行较小分块（子块）
    • 父子块均向量化存储，检索时优先命中子块，组装提示词时以父块提供完整上下文
    • 上下文注入（ContextualChunk）
    • 将文档摘要注入到每个子块的内容与元数据中，缓解 LLM 断章取义
    • 默认配置
    • 默认使用 TokenChunker 作为基础分块器
    • 默认父子块大小与重叠：父 1000、子 250、重叠 50
    classDiagram class Chunker { +Chunk(ctx, doc) []*Chunk } class SemanticChunker { +Chunk(ctx, doc) []*Chunk +HierarchicalChunk(ctx, doc) (parents, children) +ContextualChunk(ctx, doc, summary) []*Chunk } class CharacterChunker { +Chunk(ctx, doc) []*Chunk +chunkText(text) []string } class TokenChunker { +Chunk(ctx, doc) []*Chunk +chunkTokens(text) []string } SemanticChunker ..|> Chunker CharacterChunker ..|> Chunker TokenChunker ..|> Chunker SemanticChunker --> Chunker : "装饰基础分块器"

    语义分块步骤（Chunk 步骤）¶

    • 职责
    • 校验分块器是否配置
    • 从 IndexingContext 的文档通道读取文档
    • 调用分块器对每个文档执行分块
    • 将所有分块写入新的分块通道，供后续步骤使用
    • 错误处理
    • 分块器未配置、无文档、分块失败等情况均返回错误
    sequenceDiagram participant ST as "Chunk 步骤" participant DC as "Documents 通道" participant CK as "分块器" participant CC as "Chunks 通道" ST->>ST : 校验分块器与文档通道 loop 遍历文档 ST->>DC : 读取下一个文档 ST->>CK : Chunk(ctx, doc) CK-->>ST : 返回 []*Chunk ST->>CC : 写入所有分块 end ST-->>ST : 关闭 Chunks 通道

    依赖关系分析¶

    • 分块器与接口
    • CharacterChunker、TokenChunker 实现 Chunker 接口
    • SemanticChunker 实现 SemanticChunker 接口，并装饰基础 Chunker
    • 流水线集成
    • 索引器构建器在初始化流水线时，将语义分块步骤加入管线
    • 若传入的分块器为 SemanticChunker，则可在自定义流程中调用其高级能力

    性能考量¶

    • 字符分块
    • 仅做 rune 级切分，无需词表或编码，开销最小，适合快速原型与纯文本
    • 无法保证句内不被截断，可能影响语义连贯性
    • 令牌分块
    • 严格遵守模型上下文窗口，避免越界与二次切分，稳定性高
    • 编码/解码带来额外 CPU 开销，但可接受
    • 语义分块
    • 层级分块增加一次父块切分与二次切分，计算量上升
    • 上下文注入为常量时间操作，对整体性能影响有限
    • 参数对性能与效果的影响
    • 分块大小越大，召回更完整但检索精度下降，且向量存储与查询成本上升
    • 重叠越大，上下文连续性更好，但分块数量与存储/查询成本上升
    • 建议从默认参数起步，结合业务效果微调

    故障排查指南¶

    • 常见问题与处理
    • 分块器未配置：语义分块步骤会返回错误，请确保通过构建器注入分块器
    • 无文档可分块：检查上游解析步骤是否正确输出文档通道
    • 分块失败：检查分块器实现（如 TokenChunker 的词表初始化）与输入文本编码
    • 建议排查路径
    • 确认索引器构建时已添加语义分块步骤
    • 校验分块器参数（大小、重叠、词表）是否合理
    • 观察下游嵌入与向量存储是否正常接收分块通道

    结论¶

    • 字符分块适合快速验证与简单文本；令牌分块适合生产稳定与跨语言/代码场景；语义分块提供层级检索与上下文注入，适用于高质量问答与复杂文档分析
    • 合理设置分块大小与重叠，能在检索精度、上下文完整性与成本之间取得平衡
    • 建议从默认分块器起步，逐步引入语义分块能力以获得更好的检索与生成效果

    附录¶

    分块策略选择指南¶

    • 快速原型或纯英文文本：优先 CharacterChunker
    • 生产环境或中英混杂/代码：优先 TokenChunker
    • 高质量问答/长文分析/减少幻觉：优先 SemanticChunker

    分块参数配置与影响¶

    • 分块大小（ChunkSize）
    • 影响检索精度与存储/查询成本：越大越完整、越小越聚焦
    • 重叠长度（Overlap）
    • 影响上下文连续性与分块数量：越大越连续、越小越节省资源
    • 默认建议
    • 字符分块：1000 字符，150 重叠
    • 令牌分块：500 Token，50 重叠
    • 语义分块：父 1000，子 250，重叠 50

    元数据保留与上下文维护¶

    • 元数据继承
    • 分块器在生成分块时复制文档元数据，确保后续步骤可访问原始信息
    • 上下文维护
    • 语义分块的层级结构通过 Level、ParentID 维护父子关系
    • 上下文注入将文档摘要写入分块内容与元数据，辅助生成阶段减少断章取义

    使用示例与最佳实践¶

    • 基础字符分块（快速起步）
    • 使用默认字符分块器，直接注入索引器
    • 令牌分块（生产推荐）
    • 使用默认令牌分块器，确保 Token 不越界
    • 语义分块（高级 RAG）
    • 使用默认语义分块器，开启层级分块与上下文注入
    • 最佳实践
    • 从默认参数起步，结合检索效果与成本进行微调
    • 对复杂文档优先考虑语义分块
    • 注意重叠与分块大小的权衡，避免过度切分或上下文破碎
