    如何更换其它的 Embedding 模型？¶

    1. 问题分析：为什么要换模型？¶

    Embedding 模型（向量化模型）是 RAG 系统的“翻译官”，它负责把人类的文字翻译成机器能理解的“多维坐标”。 GoRAG 默认内置了基于 ONNX 的本地 BGE 模型，这在测试和中小型项目中非常方便。但在以下场景，你必须更换它： - 专有领域词汇多：标准模型无法理解你们公司的专业黑话（比如特定的医药名词或代码缩写），你需要使用微调过的私有模型。 - 多语言混合：你的文档包含中、英、日、阿拉伯语，你需要更换为支持多语种的 text-embedding-3-large (OpenAI)。 - 硬件限制：你的服务器没有 GPU 跑不动本地模型，必须调用外部的 API（如 DashScope 或百川）。

    开发者自然想知道：换个模型是不是意味着我要把原来存的向量库全删了重来？代码上该怎么无缝切换？

    2. 机制解析：Provider 接口与维度对齐¶

    GoRAG 对大模型的调用全部抽象在 gochat/pkg/embedding 的 Provider 接口中：

    type Provider interface {
    // 批量将文本转为向量
    Embed(ctx context.Context, texts []string) ([][]float32, error)
    // 声明该模型输出的向量维度（如 768 或 1536）
    Dimension() int
}

    维度对齐机制： 向量数据库（VectorStore）在初始化时，必须知道它要存多少维的向量（比如 Milvus 建表时必填 Dimension）。 当你通过 gorag.WithEmbedder(myProvider) 注入自定义模型时，GoRAG 的核心框架会自动调用 Dimension() 方法，并使用这个返回值去初始化底层的 VectorStore。这就是 GoRAG 能做到“换模型即插即用”的底层魔法。

    3. 实战指南：接入 OpenAI Embedding¶

    假设我们不想用本地模型了，想换成 OpenAI 的 API 服务。

    package main

import (
    "context"

    "github.com/DotNetAge/gochat/pkg/embedding"
    "github.com/DotNetAge/gorag"
)

// 1. 实现 Provider 接口
type OpenAIEmbedder struct {
    APIKey string
}

func (o *OpenAIEmbedder) Embed(ctx context.Context, texts []string) ([][]float32, error) {
    // 在这里编写调用 https://api.openai.com/v1/embeddings 的 HTTP 逻辑
    // 返回对应的 float32 数组
    return [][]float32{}, nil
}

// OpenAI 的 text-embedding-ada-002 是 1536 维
func (o *OpenAIEmbedder) Dimension() int {
    return 1536
}

func main() {
    ctx := context.Background()
    // 注意：如果你完全不用本地模型，其实可以跳过 PrepareEnvironment

    // 2. 实例化你的模型
    myOpenAI := &OpenAIEmbedder{APIKey: "sk-proj-xxxxxx"}

    // 3. 注入 RAG 实例
    rag, err := gorag.DefaultNativeRAG(
        gorag.WithWorkDir("./data_openai"), // 强烈建议：换模型一定要换新目录！
        gorag.WithEmbedder(myOpenAI),       // 👈 框架会拉取你的 Dimension()
    )
    if err != nil {
        panic(err)
    }
    defer rag.Close()

    // 现在的索引和检索都将走你的 OpenAI API
    rag.IndexFile(ctx, "company_intro.pdf")
}

    4. 延伸思考：致命的“维度不匹配”灾难¶

    这是一个血泪教训：如果你原本用 BGE (768维) 存了 10 万篇文档在 ./data 目录下。今天你改了代码，注入了 OpenAI (1536维)，但你忘记修改 WithWorkDir。

    会发生什么？ 当你执行 Search 时，OpenAI 返回了 1536 维的查询向量，但底层的 GoVector 库里存的全是 768 维的向量。在计算余弦相似度（Cosine Similarity）时，数组越界，程序将直接 Panic 崩溃！

    最佳实践： 永远不要在同一个物理目录里混用不同维度的 Embedding 模型！如果你更换了模型，必须： 1. 指定一个新的 WithWorkDir。 2. 重新执行 IndexDirectory 重新全量向量化你的文档。
